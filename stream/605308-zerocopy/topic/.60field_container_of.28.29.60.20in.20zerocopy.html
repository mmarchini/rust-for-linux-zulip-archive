Boqun Feng (May 29 2026 at 18:05):

@Joshua Liebow-Feeser Hi, I'm taking a quick look at the Ptr type you mentioned in last meeting. And I find out we could do projection from type to field with project(). I'm curious about whether there is any plan to support "coproject" from field to type?

For context, we have a few intrusive data structure design in kernel. And currently we use the approach like impl_has_work!, which is a bit verbose IMO. I've proposed something here:

pub unsafe trait HasField<T, F: Field<T>> {
    /// Returns a pointer to the [`Field<T>`] field.
    ///
    /// # Safety
    ///
    /// The provided pointer must point at a valid struct of type `Self`.
    unsafe fn raw_get_field(ptr: *mut Self) -> *mut F;

    /// Returns a pointer to the struct containing [`Field<T>`] field.
    ///
    /// # Safety
    ///
    /// The pointer must point at a [`Field<T>`] field in a struct of type `Self`.
    unsafe fn field_container_of(ptr: *mut F) -> *mut Self;
}

to cope this.

Since zerocopy already has something similar, I thought it's worth while to bring it up.

Joshua Liebow-Feeser (May 29 2026 at 18:39):

HasField::field_container_of should be relatively easy to add in theory. You'd just need to make sure that ptr has provenance for all of Self, and I'm not sure how y'all handle that sort of thing right now (if at all – I'd understand just hand-waving it).

Currently, HasField::project takes a PtrInner, which has the invariant that it has provenance for its referent. However, I don't see why we couldn't say, in a particular instance, that it has provenance for some other referent as well (in this case, all of Self). We'd just need HasField::field_container_of to be unsafe (unlike HasField::project).
