Philipp Stanner (Apr 22 2026 at 13:45):

I think I have an idea.

The thing with the code presented by and Daniel is that we would basically store the same information that the C backend already has another time, in another allocation, in Rust.

But this information is already stored deep down in the C dma_fence: 

struct dma_fence {
    union {
        spinlock_t *extern_lock;
        spinlock_t inline_lock;
    };
    const struct dma_fence_ops __rcu *ops;

    union {
        struct list_head cb_list;

There's a list of all callbacks that are waiting for this fence to be signalled. The fence knows which callbacks are registered and which are signalled already.

Now, if we were able to port the C backend so that the backend uses an XArray instead of a list, we might benefit twice. Once because the backend gets a bit faster since the xarray is more cache-friendly when iterating the callbacks.

Twice because it might be possible to replace Daniel's FenceCallbackRegistration with some sort of ID wrapper

struct FenceRegistration {
  fence: ARef<Fence>, // no new memory, does not need allocation
  id: usize,
}

whose id would be used to obtain the correct callback through the XArray in the C fence. And then we could do container_of, if we can derive what generic data a callback data contains

So we might end up with:

let cb_object = FenceCallback::new(data)?; // allocates

...

let registration = fence.register_callback(cb_object); // does not need allocation

If you'd drop your registration, you'd look into the C backend whether it's still there, if yes, you get it back and can drop it – if not, it's signaled already (ownership over data being transferred back to the driver).

Just had the idea a few mins ago. Not sure if I'm missing something, but it could be very clean because we'd rely on the C backend, who already has all the relevant information, doing the work? @Boris Brezillon

Philipp Stanner (Jun 16 2026 at 08:04):

@Boris Brezillon

It would seem that the "drop in atomic context" problem goes even a few centimeters deeper than expected

If we now indeed drop DriverFenceData with call_rcu(), then the last reference to FenceCtx can also drop in atomic context.
