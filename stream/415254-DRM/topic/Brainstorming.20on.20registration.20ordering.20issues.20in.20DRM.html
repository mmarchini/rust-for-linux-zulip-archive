Topic: Brainstorming on registration ordering issues in DRM

Lyude (Oct 29 2025 at 19:13):

So - now that I'm finally at a point where all of the dependencies for rvkms are waiting for review on the mailing list, I've started rebasing rvkms back onto the more recent DRM work and I ran into an issue that had slipped my mind.
A while back we tried having users only create Registration objects for new DRM drivers, and then acquire the actual DRM Device from that. We ended up changing this for Tyr but, unfortunately the current solution that we have doesn't work for rvkms (or any KMS driver) without changing it.
The main one being the ordering of when registration happens. Currently, the whole set of KMS bindings that I have for rust rely on the assumption that we will never get a fully fledged Device until the KMS initialization has completed, which we can't guarantee at all if there's a split between device creation and device registration - since the user would have access to a Device well before KMS initialization finished. Checking if KMS init at runtime would be a really ugly fix I'm not really willing to do, because "calling a KMS function too early" doesn't really have a reasonable recovery path in a driver. And then there's the issue of: if Device is our default device type, and we already have plenty of object types that can return references to their owning devices (like GEM objects), any of those objects can also be used to retrieve a Device and call KMS methods on it before KMS has been setup.
So - I'd really like to again propose the idea of not exposing Device before registration finishes, and instead come up with a pre-registration type that could be used for various pre-registration init purposes (like creating the gem objects that Tyr needs early on, though I do really wish we could create these outside of gem). And I'd definitely like to some feedback here.

My main goals:

  • Keep having a Device type, but make it so that this type is only ever exposed in contexts where we're fully registered, and then design pre-reg De vice types
  • Not breaking Tyr's usecase for early gem object creation
  • Not handling this in reverse. E.g. I would strongly like to avoid a RegisteredDevice type, because it is a lot easier to define the surface for a restricted Device variant then it is to assume our default Device type might not even be registered yet. Remember: there's a lot of types in DRM that can (and need to) return backpointers to their owning device, and we want to avoid making all of those backpointers return what might not be a registered device since that means the caller can never use those backpointers without having to check the registration state at runtime. And we should probably be designing bindings around registered drivers in the first place.

The one idea I've had so far is adding a sort of "UnregisteredDevice" type, and then adding traits for the various capabilities that we need before registration. For instance: GemDevice could represent any type that can return a drm_device pointer and is known to be in a state where it can create gem objects. We could also add a type market to gem objects that indicates what type of device was used for creating the device and have it default to Device so that users don't need to check registration status outside of gem objects they created before registration.

The idea is pretty much the same as DeviceCtx: we introduce a type parameter representing the registration state of the device and then generalize any DRM objects that can exist before registration over said parameter. So like:

drm::Device<T> -> drm::Device<T, R: RegistrationCtx = Registered>
drm::gem::Object<T> -> drm::gem::Object<T, R: RegistrationCtx = Registered>

etc.

You can even take references to a device before it's registered! Though, all such references will be drm::Device<T, Any>

FWIW: I still think it might not be a bad idea to allow some limited ARef in the way I described before, e.g. DRM bindings can get an early ARef in the EarlyInternal context but actual users can't. If not mostly just because in the device.rs file I showed you: we still need to handle resource cleanup if say, we successfully create the Device before registration but fail on the KMS portion. We can currently clean that up, but that implies needing to manually call the destructor functions for the device (which we could just skip if we had an ARef to drop).
