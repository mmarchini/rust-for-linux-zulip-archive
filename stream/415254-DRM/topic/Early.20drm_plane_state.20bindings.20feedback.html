Alice Ryhl (Feb 09 2024 at 21:00):

The reason why you don't necessarily have to use pinning or Opaque is that those only really matter for structs that are accessed with a refernce or Box. If you never create those to the struct while it is in use, then it doesn't make a difference.

Alice Ryhl (Feb 09 2024 at 21:01):

But anything where the user of your library can have direct ownership of it cannot use that strategy.

Lyude (Feb 09 2024 at 21:04):

Alice Ryhl said:

But anything where the user of your library can have direct ownership of it cannot use that strategy.

So - if we're allocating structs like the example I mentioned, where the struct itself is in a Box<> but the ffi type in question is embedded in that struct, I assume that -does- require pinning?

Alice Ryhl (Feb 09 2024 at 21:19):

Basically, whenever the Box exists, you assert uniqueness over the allocation. That's incompatible with self-referential structs.

Alice Ryhl (Feb 09 2024 at 21:36):

But Opaque opts-out of that uniqueness assertion.

@Maíra Canal

For DRM it's true that most of the infrastructure is in the Asahi tree. However, I'd like to avoid that we just wildy cherry-pick stuff from one driver tree to another and, instead, have a common baseline where we improve things collaboratively until we hit upstream.

This way we avoid things diverging and getting scattered. Some drivers might extend improve things cherry-picked from somewhere else. Without a common baseline this will be very hard to handle. In the worst case we end up with every driver carrying it's own version of some piece of infrastructure.

Plus it has the advantage that we only have to continuously rebase the common baseline against upstream instead of every driver rebasing the cherry-picked mess themselfes.

rust-dev is an experimental branch for integration purposes. It is a queue for patches that "look good enough".

- Finding merge/apply conflicts as early as possible.
- Providing a common base for development that requires features that are not yet in mainline or rust-next, i.e. giving early access to features. This may include Rust-related changes from other subsystems.
- Providing extra testing to patches by making them easily available to more developers.

Note that this branch may be updated/rebased frequently and it might be gone in the future.

Miguel Ojeda (Feb 14 2024 at 12:08):

We discussed whether to make this integration branch of the topic branches part of rust-dev or as a new branch: rust-dev will continue to be meant only for things already submitted to the LKML or present in upstream kernel subsystem trees (e.g. Boqun may merge trees from other maintainers sometimes when needed, but he is not expected to be merging our topic branches).

So we can create a different integration branch for the topic branches -- essentially like a linux-next for the topic branches (a bit more relaxed in requirements) and like the old rust branch way. I would suggest we do that when we have at least 2 or 3 branches to integrate. Ideally one or more of the maintainers of the most active topic branches would maintain it, since those topic branches will be triggering most of the changes.

By the way, the topic branches do not need to necessarily be in our repository: if an upstream maintainer is willing to have the development/topic branches in their own subsystem, that is ideal (and it is still possible to integrate those into the integration branch). Of course, this is meant for things that are not in linux-next, i.e. the point is not to recreate linux-next, so the usual branches that a subsystem already has there would not really be the target for this integration branch.
