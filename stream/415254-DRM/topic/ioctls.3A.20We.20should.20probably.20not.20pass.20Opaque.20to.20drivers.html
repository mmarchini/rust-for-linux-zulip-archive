Topic: ioctls: We should probably not pass Opaque to drivers

Daniel Almeida (May 12 2025 at 14:01):

@Danilo Krummrich

Hi Danilo, I see that you have changed from &mut T to &Opaque<T> in the ioctl macro.

We should probably dereference that instead of handing the Opaque to the drivers to avoid the proliferation of unsafe and the handling of raw pointers at the driver level. This comes at a cost of two extra memcpy's per ioctl.

My proposal is:

                            let data = unsafe {
                                &*(raw_data as *const $crate::types::Opaque<$crate::uapi::$struct>)
                            };

                            // Copy into our stack and pass that to the drivers
                            // so they get a &mut T, instead of &Opaque<T>
                            let args  = unsafe {*data.get()};

                            // SAFETY: This is just the DRM file structure
                            let file = unsafe { $crate::drm::File::as_ref(raw_file) };

                            match $func(dev, &mut args, file) {
                                Err(e) => e.to_errno(),
                                Ok(i) => i.try_into()
                                            .unwrap_or($crate::error::code::ERANGE.to_errno()),
                            }

                            // Writeback the updates, if any.
                            core::ptr::write(data.get(), args);

WDYT?

Danilo Krummrich (May 12 2025 at 14:07):

How do you guarantee that the data behind data.get() is valid? I think we can only ever guarantee that if we only allow uAPI structures that are impossible for userspace to create invalid instances from for Rust.

But even if we do that, my reasoning was that it's not worth at all. uAPIs typically pass pointers through u64 fields to the kernel, which must be handled by drivers anyways.

I think the correct answer is to provide generic helper infrastructure, to make it easy for drivers to generate the corresponding uAPI abstractions. See what I do in nova-drm [1], I think we should generalize that.

[1] https://gitlab.freedesktop.org/drm/nova/-/blob/staging/nova-drm/drivers/gpu/drm/nova/uapi.rs?ref_type=heads

IIUC:

The current solution builds an Opaque<T> from a raw pointer, i.e.:

                            // SAFETY: This is just the ioctl argument, which hopefully has the
                            // right type (we've done our best checking the size).
                            let data = unsafe { &mut *(raw_data as *mut $crate::uapi::$struct) };

So the opaque is only as valid as is the raw pointer it wraps.

The current solution then passes &Opaque<T> to the drivers, which now have to call .get() to be able to actually access the uapi data, so the whole thing only works if we assume that .get() returns valid data. It's just that this responsibility now lays with the driver.
