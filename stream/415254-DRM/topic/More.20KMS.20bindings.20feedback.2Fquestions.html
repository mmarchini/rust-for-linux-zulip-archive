Lyude (Feb 14 2024 at 21:08):

I guess just to give some more context too since I assume not everyone is familiar with the atomic modesetting interface: generally, a "modeset" ("modeset" means setting a display mode on a monitor, like a DisplayPort display or something like that) happens like this:

  • allocate a drm_atomic_state to represent the new display state, this is mutable by the user
  • Add the modesetting objects (e.g. drm_plane/drm_crtc/drm_connectors) that will be affected by this state change to the drm_atomic_state. This happens by grabbing each modesetting object's ww lock, and then duplicating the current atomic state set on each modeset object to give the user a state they can mutate. Nothing actually gets committed to hardware yet
  • Perform an atomic check, which is basically handing the new display state to the kernel and having it verify that this state is actually physically possible, and this state is also checked against the currently programmed display state to figure out what changes are needed. This prevents us from getting in a situation where we commit an impossible display state, and then get stuck because the hardware crashes and we don't have a good way of recovering.
  • If the atomic check passes, then an atomic commit can happen. This is the final step and causes us to swap the current display states on each modesetting object out with the new states we created, and then the changes from the new display state are actually applied to hardware. This step is infallible, which was guaranteed by the atomic check we just performed (both this and the previous step happen under lock. if they happened as separate steps, another atomic check is performed beforehand to prevent racing).

It's mainly ^ the last step that I would assume would access Plane<T> (or our future Crtc<T>, etc.) directly, as a driver might have information it needs that's specific to a plane - especially static info that is populated at driver load time. As well, information there usually refers to the actual hardware state that's currently programmed. That can be important sometimes since non-blocking modesets are a thing, and we swap the old and new display states in that case before the actual modeset completes in hardware. In which case the only actual source of truth we would have for the state that's currently programmed in the hardware would be in the Plane<T>

Lyude (Feb 14 2024 at 21:09):

Atomic checks on the other hand shouldn't ever really look at Plane<T> for anything beyond static information that's populated at driver load, since if non-blocking modesets are a thing you aren't guaranteed there isn't already a modeset happening while you're in the atomic_check phase.
