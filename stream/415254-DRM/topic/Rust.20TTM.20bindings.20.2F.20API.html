Eliot Courtney (Apr 28 2026 at 12:49):

I’ve had a little bit of time to look at this so I wanted to post what I have so far to make sure I’m not going down a bad path. I was looking at the GEM bindings and have mostly proceeded according to patterns established there - in particular, I looked a lot at the GEM shmem “backend” for GEM. What I have so far is some basic set of types for ttm_device, ttm_buffer_object, and ttm_tt. I’ve made a few simplifying assumptions, like:

  • Assuming a 1:1 correspondence between TTM driver and DRM driver (implemented by same object)
  • Must select one kind of GEM+TTM object as the user facing GEM object (must specify to DRM driver trait)
  • Use one type of DriverBufferObject per driver, since we need to be able to recover the type in callbacks (either BufferObject<T::BufferObject> or a transfer buffer object - drivers would have to use an internal enum / inner sort of thing to have multiple BO types)

I’m trying to keep the following things:

  • Keep object lifecycle handled within framework code.
  • Impossible to misuse from drivers
  • Encode state of objects using the type system, e.g. try to represent reserved and pinned for buffer objects using the type system. There’s a bunch of methods that can only be called for reserved BOs and we should make it structurally impossible to call them for non-reserved BOs.

You can see the (very rough) WIP code here: https://github.com/Edgeworth/linux/commits/ttm-prototype/ . I talk about this later, but I think there are some directional problems here that need drm::Device to have a lifetime to solve in a tractable way.

Essentially it works similarly to the gem shmem bindings in that it defines a struct which embeds driver specific data and the gem+ttm object. There’s a “DriverTTM” trait which groups together the set of types a driver wants to use for ttm (e.g. the buffer object type, the tt type). There is some difficulty with the lifetimes and initialization states, as some C methods need cleanup if the initialization fail, and some don’t. So the bindings need to be very careful with e.g. putting deinit in Drop. Objects can also be used in a partially initialized state - I’ve tried to formalise this with “InitRef”.

One other thing I noticed is there doesn’t appear to be anything preventing a gem::Object from outliving the drm::Device, and it has a public safe method to create a reference to its drm::Device, which seems unsound to me (see gem_object_mints_device_ref_after_device_free test demonstrating this). So I suppose for this kind of issue we either need to ref drm::Device via ARef, or hold a lifetime. ttm::BufferObject has to deal with this same issue.

As an experiment, I added a DeferredARef of drm::Device in ttm::BufferObject for this reason, but it means we need to be careful about cycles (i.e. we can’t store an ARef to drm::Device inside ttm::Device, since the natural location that drivers could store ttm::Device in inside drm::Device::Data). It’s unsound with a regular ARef because ttm’s delayed BO release could cause drm::Device+ttm::Device to be freed while cleanup for them is already running. Instead, DeferredARef defers dropping an ARef until later using workqueues.

If we have rust lifetime scoped drm::Devices using the ForLt stuff that Danilo and Gary worked on then we can avoid this mess and also fix gem::Object by making it not outlive the drm::Device lifetime. So we would have ttm::BufferObject carry a &‘drm drm::Device, where that lifetime represents the actual lifetime of the drm Device (not hardware device), then we don’t need to worry about it outliving drm::Device or it causing drm::Device to be destroyed - we just need to ensure that the C side doesn’t hold onto those objects for longer than the ‘drm lifetime. I feel it’s going to be very complex without having a lifetime’d drm::Device. @Danilo Krummrich are you looking at this already (w.r.t. extending ForLt stuff to drm::Device)?
