While working on the hrtimer bindings, one of the things that has come up is timer types where we know that we have unique access to the given type - which would basically allow us to be able to call certain functions outside of the normal timer callback-context so long as we know the timer isn't running and the type has unique access. For Arc<>, the natural assumption I had was that we could try to use UniqueArc<> for this - which lead me to Arc::into_unique_or_drop(). Ideally, I'd like to be able to fallibly convert into a UniqueArc but the existence of this function makes me wonder if there's some reason this isn't possible to do that I'm missing.
Here's what's confusing me: into_unique_or_drop appears to rely on the fact that if we're able to successfully atomically decrement the refcount of the Arc to 0, we're guaranteed that no one else can take a reference and that this Arc<> is unique. That makes sense, however - it kind of seems unecessary unless I misunderstand how this works.
If we were to simply read the refcount atomically and it was 1, and we've been given ownership of the Arc<>, doesn't that already mean it's impossible for anyone else to grab a reference and thus is safe to convert to UniqueArc without needing to decrement the refcount? Since we'd be the only ones with access to the Arc<> anyway
