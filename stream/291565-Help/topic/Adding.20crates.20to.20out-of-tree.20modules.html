Andreas Hindborg (Apr 29 2023 at 07:15):

Miguel Ojeda said:

Please be aware that if you add third-party crates, your code is likely going to be hard to upstream. (Previous discussion at https://rust-for-linux.zulipchat.com/#narrow/stream/384413-Filesystems/topic/FS/near/348298262).

I think it is important to recognize that not all code are intended to be included in mainline. Some code might be impossible to upstream because of licence issues, or perhaps the code is not intended to be open sourced ever. While others do not agree, I support the idea of having a build system for out-of-tree modules that include cargo support. I think this is a means of enabling even more kernel code to be written in Rust, even if this code will never make it into mainline Linux.

The problem with this approach is that some projects might might start relying on the features provided by this out-of-tree build system (cargo and huge dependency graphs) just for convenience, even though they could maybe use other means and might have been candidates for upstream. Such a build system should come with HUGE warning banners about the consequences of not following upstream guidelines.

Either way, someone is going to build this system eventually, and we might as well support it to get said warnings into the system.

Miguel Ojeda (Apr 29 2023 at 09:35):

I think it is important to recognize that not all code are intended to be included in mainline.

Nobody has said otherwise! :) In fact, that is fine and supported.

The question here is not to support out-of-tree code or not, but whether to support a feature which may never land upstream and may never be available for in-tree modules.

Ideally such a feature (i.e. easily reusing existing crates) would land for both sides at the same time (even if, for in-tree, individual third-party crates would need to be explicitly allowed one-by-one), potentially as part of the new build system.

Some code might be impossible to upstream because of licence issues, or perhaps the code is not intended to be open sourced ever.

Side-note: to be clear, if there are licenses issues, then they really need to be careful about what they are doing, even if they are out-of-tree, because distributing those kernel modules may not be legal.

On top of that, core Rust symbols are exported with EXPORT_SYMBOL_GPL, and thus the kernel module loader actively refuses to link proprietary modules against them (i.e. bypassing that may be easily seen as a deliberate action).

The problem with this approach is that some projects might might start relying on the features provided by this out-of-tree build system (cargo and huge dependency graphs) just for convenience, even though they could maybe use other means and might have been candidates for upstream.

This is definitely an issue, but as long as the feature is supported in-tree and it is just a matter of allowing/disallowing crates in a list and warning the user otherwise, it should be fine.

Either way, someone is going to build this system eventually, and we might as well support it to get said warnings into the system.

If upstream does not want a feature, then, of course, it would not be a surprise if somebody supports it outside anyway; but we are focused on getting things upstreamed, i.e. available for everybody.

Miguel Ojeda (May 05 2023 at 16:08):

I meant that I could prepare a branch with serde.

The syn one is here, but it is fairly old, and based on rust: https://github.com/Rust-for-Linux/linux/pull/910

So I rebased that one on top of Linus', then added serde and serde_derive on top, and a sample that uses a data format defined in kernel which then gets used in the sample kernel module for a custom type via derive. So this works:

#[derive(Serialize, Deserialize, Debug)]
struct S {
    a: (),
    b: bool,
    c: bool,
    d: (),
}

...

let original = S {
    a: (),
    b: false,
    c: true,
    d: (),
};
crate::pr_info!("    original = {:?}", original);

let serialized = kernel::test_serde::to_vec(&original).unwrap();
crate::pr_info!("  serialized = {:?}", serialized);

let deserialized: S = kernel::test_serde::from_vec(&serialized).unwrap();
crate::pr_info!("deserialized = {:?}", deserialized);
