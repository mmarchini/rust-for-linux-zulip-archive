Topic: try_format! macro

Mateusz Jabłoński (Jul 31 2021 at 15:20):

Hi! I was thinking about implementing try_format! macro, the panic-free equivalent of format!. But the more I am looking into the code, the more questions I have and more problems I can see:

  • It will require implementing a few more try_* functions - Are we OK with implementing new try_* functions inside alloc? Or should we generally avoid any other than 'critically required' changes there?

  • There is a fmt::Write trait implementation used inside, which uses push_str, which may panic. It would be consistent to create and use something like TryWrite trait. But I see even more problems there:
    - The write_* methods of the Write trait return fmt::Result. What should be returned from try_write_* methods? I do not like the idea of Result<Result<..., ...>, ...>, I do not like creating a new container error type either,
    - For the same reason, the TryWrite name is confusing in my opinion. All methods of Write already return Result, so they already 'try' to do something,
    - The Write trait is defined inside core crate. It may be inconsistent if we define TryWrite somewhere else.

  • The format function calls .expect() on the Result returned from Write trait implementation. I haven't investigated if it can ever happen for String, but I don't very like it there.

What do you think about it? Should I try to implement the try_format!, or should we avoid such ideas?
