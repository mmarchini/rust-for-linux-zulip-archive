Benno Lossin (Apr 04 2025 at 11:50):

Miguel Ojeda said:

Is that the orphan rule? Then, yeah, one way would be putting them there. Alternatively, could we use a transparent wrapping type?

Yeah it's the orphan rule. A transparent wrapper type is probably very annoying and won't scale. (If the bindings type is only used in Opaque, then there isn't a problem, as Opaque already should implement Zeroable. if it doesn't, it's an easy fix)

If we are going with the bindings approach, then I think ideally we would have bindgen annotate things like this automatically. Could we perhaps use --with-derive-custom to annotate those?

Yeah, I think such an option would be nice (after all, we have the derive(Zeroable) macro). Maybe we need a derive(MaybeZeroable) that doesn't error when it can't implement Zeroable due to not all fields being Zeroable (IIRC the normal one does error).

Benno Lossin (Apr 04 2025 at 12:20):

Rob Herring said:

Is there a way to make Zeroable not foreign (as it wasn't in 6.14 I guess since this worked then)?

I moved all of the items that belong to pin-init into their own crate for better maintainability, see https://lore.kernel.org/all/20250308110339.2997091-1-benno.lossin@proton.me

I don't think there is a way to make it not foreign, as the pin-init crate also depends on it.

We asked the rust folks about relaxing the orphan rule, but there hasn't been a lot of progress there.
