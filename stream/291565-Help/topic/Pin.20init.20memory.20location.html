I'm trying to understand where the memory is allocated for this module:

/// Kernel module that exposes a single file system implemented by `T`.
#[pin_data]
pub struct Module<T: FileSystem + ?Sized> {
    #[pin]
    fs_reg: Registration,
    _p: PhantomData<T>,
}

impl<T: FileSystem + ?Sized + Sync + Send> crate::InPlaceModule for Module<T> {
    fn init(module: &'static ThisModule) -> impl PinInit<Self, Error> {
        try_pin_init!(Self {
            fs_reg <- Registration::new::<T>(module),
            _p: PhantomData,
        })
    }
}

As far as I've read, try_pin_init! creates an in-place constructor. What I don't understand is where this module will end up: on the heap, on the stack, in the data segment? How can I figure this out?

Benno Lossin (Apr 03 2024 at 18:34):

As you said correctly, try_pin_init! creates an in-place constructor. So a value has not yet been created. This initializer can be used to create a value at any location you want. Some examples of ways to use an initializer:

  • Box::pin_init/Arc::pin_init allocate on the heap
  • stack_pin_init! on the stack

But since you are implementing the InPlaceModule trait, you are probably also using the module! macro. That macro automatically creates a static value of the type given to the module! macro. It is automatically initialized on module load via the initializer returned by InPlaceModule::init.

impl<T: Module> InPlaceModule for T {
    fn init(module: &'static ThisModule) -> impl init::PinInit<Self, error::Error> {
        let initer = move |slot: *mut Self| {
            let m = <Self as Module>::init(module)?;

            // SAFETY: `slot` is valid for write per the contract with `pin_init_from_closure`.
            unsafe { slot.write(m) };
            Ok(())
        };

        // SAFETY: On success, `initer` always fully initialises an instance of `Self`.
        unsafe { init::pin_init_from_closure(initer) }
    }
}
