There are some good userspace Rust bindings to io_uring that you may be able to build off of https://github.com/tokio-rs/io-uring (cc @Alice Ryhl who's basically the leader of all things Rust async)

If io_uring specifically is something you are interested in, I don't think it hurts to reach out to the maintainers and see what their thoughts are https://github.com/torvalds/linux/blob/4f82870119a46b0d04d91ef4697ac4977a255a9d/MAINTAINERS#L11008C3-L11008C3 (cc the RFL mailing list if you decide to). Just make sure you say that you are looking for an experiment and want their opinions before getting too into it. A few things you may want to note:

  1. How well can io_uring exist without other subsystems that we don't yet have abstractions for?
  2. I suspect that a lot of the io_uring bugs come from the core implementation rather than the interface, so I'm not sure how much benefit Rust will add (I may be wrong here). Maybe that's a long-term argument for replacing some core components with Rust? I wouldn't keep my fingers crossed on that but who knows...
  3. Be ready for a very long term commitment if you're serious about this :smile:

Finally, by way of a general question, are there any good tutorials online on how to construct bindings/abstractions? That’s something I struggled with in the io_uring hacking.

I only know of this https://michael-f-bryan.github.io/rust-ffi-guide/overview.html but I don't think it's especially well suited for everything here. Really just by looking at good examples, and trying to answer the question "how do I make mistakes in C impossible to do from Rust"

When I added the io_uring types header files to the Rust bindgen inputs, only some of the structs were translated fully into Rust, while others mapped to stub implementations.

That isn't uncommon for types that bindgen can't figure out. Sometimes you need to handwrite them instead (as you mentioned), sometimes you can get away with opaque items, sometimes tweaking the bindgen options for that type can give you better results. If you figure out why they failed we may be able to help you

Additionally, the io_uring subsystem uses many static functions from headers, which I was unable to access in Rust files.

bindgen can't do static inlines, so the current solution is to provide a rust_helper_x method in C to make it not inline https://github.com/torvalds/linux/blob/master/rust/helpers.c. bindgen has an unstable feature to do these automatically, but it's not finalized.

I worked around these difficulties by rewriting structs/functions manually, which I think is probably not the intended solution. (I saw in recent mailing list discussion that some of the other subsystems requested new abstractions to only be sent alongside upstream projects that utilize them, so I don’t plan to send abstractions as an standalone product, but I imagine it will be necessary to construct them along the way.)

You can always make a PR to the RFL repo and send a message here if you would like something reviewed! Some maintainers may be OK with RFC patches if you have something that may be upstream-ready, probably good to check with them first and make it very clear in the cover letter "this is not for upstreaming, I do not yet have a use case, I just want to make maintainers aware of my work, can you think of any use cases, etc"

Trevor Gross (Nov 16 2023 at 23:52):

Hm, good question. I know about this from the hardware level but not so much from a firmware level

The hardware network is usually processor <- PCIe -> MAC <- MII/RMII/GMII/RGMII/SGMII/XGMII -> Phy and then you also have the MDIO interface to talk to the phy and access its registers directly. A lot of modern NICs usually combine the MAC and the Phy into one (I'm not really sure what happens with MDIO here)

If you're using SFP then it's basically the same for the high speed signals, SGMII for SFP and XGMII (sometimes aka XFI) for SFP+. The difference is that you don't have a direct MDIO connection - you have I2C which usually goes to a bouncer chip that sets SFP-specific registers, and forwards some to the phy's config registers
