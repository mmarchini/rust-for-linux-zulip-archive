Topic: Exposing stack-allocated Rust APIs to C

David Rheinsberg (Sep 10 2024 at 13:06):

Hey, do we have a consensus on how to expose Rust kernel APIs to other C code, if said Rust code uses stack allocated objects? I want to allow C-callers to allocate the API objects on the stack, without necessarily exposing the struct details. I would have to somehow ensure the hand-crafted C header uses struct-sizes that are in-sync with the Rust objects, but that kinda requires a generator (as struct-layout is very much architecture dependent, even with repr(C)).

Any hints or even previous art on what to do? My workaround for now is to always use heap-allocations, but that is a bit annoying for cheap operations.

Alice Ryhl (Sep 10 2024 at 13:07):

You can make a struct on the C side with sufficient space and assert inside Rust that the size and alignment of the C struct is sufficient.

David Rheinsberg (Sep 10 2024 at 13:14):

Yeah, but size requirements can differ greatly between architectures, especially when using repr(c) on the object.

David Rheinsberg (Sep 10 2024 at 13:15):

And then object-size might depend heavily on CONFIG_* knobs. So imagine putting struct mutex into the object. It changes size heavily if you enable lockdep, for instance.

Alice Ryhl (Sep 10 2024 at 13:18):

It's okay if the size is too large.

Alice Ryhl (Sep 10 2024 at 13:18):

Other than that, we don't really have any way to do it right now.

Alice Ryhl (Sep 10 2024 at 13:19):

Maybe you could set up some sort of makefile thing to generate a file containing the size at build time ... but that sounds complicated.

Benno Lossin (Sep 10 2024 at 14:11):

David Rheinsberg (Sep 11 2024 at 06:47):

I could define the struct in C and then use it in Rust, but that means Rust would have to work on a raw struct, rather than a struct using the abstractions provided by kernel. E.g., if I put a mutex in it, the Rust side would have to work on the raw mutex, rather than the lock abstraction. Or I would have to transmute on each call, if the abstraction uses repr(transparent).
