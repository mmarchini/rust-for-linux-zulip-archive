Topic: Why Arc<T> uses KBox<T> tinstead of Box<T, A>

HangShu (Nov 06 2025 at 06:25):

My question is regarding the design choice of Arc<T>. Why does Arc<T> internally use KBox<T> to allocate memory for ArcInner<T>, instead of adopting the more flexible Box<T, A>, which would allow users to specify their own allocators?

Is the use of KBox<T> strictly necessary, or would it be more advantageous to switch to Box<T, A>? If the latter is preferable, would it be acceptable for me to submit a PR to replace KBox with Box<T, A>?

Alice Ryhl (Nov 06 2025 at 07:49):

I don't think that would be useful with kvmalloc or vmalloc, but if we also added kmem cache, then I think that might be a useful feature I could use in Binder.

HangShu (Nov 06 2025 at 08:02):

Currently, it seems that using KVMalloc and VMalloc for Arc doesn't offer clear advantages. However, introducing a default generic parameter like Arc<T, A = Kmalloc> doesn't appear to introduce any additional overhead. Moreover, as you said, if kmem_cache support becomes available in the future, I believe Arc<T, A> could be a more flexible and preferable choice.

By the way, if I were to submit a PR for this, would the community be open to considering it?

Miguel Ojeda (Nov 06 2025 at 14:27):

If you think you have an improvement, then definitely please feel free to send it to the mailing list! Having the actual change in front usually helps to evaluate it, and sometimes uncovers unexpected changes needed, etc.

Of course, it will depend on the reviews and the maintainers of the relevant code etc.

Please try to justify the change in the commit message as best as you can, like the potential use cases. That usually helps a lot, especially in cases where there is no concrete user just yet. Having an actual user is even better, of course.
