Asahi Lina (Sep 01 2022 at 08:15):

Hi! Yesterday I tried writing an abstraction for the DRM GEM object layer (and the shmem helpers) and got a bit stuck on how to model it.

The basic design I'm trying to wrap is:

  • There is a base drm_gem_object struct
  • Drivers subclass (embed) it into their own object struct
  • The DRM subsystem uses refcounting to keep track of the lifetime of the GEM object
  • Drivers provide a creation callback that allocates a new (driver-specific) object struct and returns a pointer to the drm_gem_object within
  • Drivers provide a freeing callback that takes a drm_gem_object, container_of's into the driver struct, and frees any driver-specific state
  • There are optional helper layers like drm_gem_shmem, so you end up with a driver object embedding a drm_gem_shmem_object embedding a drm_gem_object.

Yesterday it was late and I didn't really get anywhere, but thinking about it again, I guess it would be something along these lines?

  • Wrap drm_gem_object in some opaque GEM::Object type
  • The driver provides a type (implementing some trait) that represents its GEM objects (Foo::Object)
  • That trait provides a function that converts a reference to the object to a reference to the GEM::Object within (wrapped in some opaque type) (must enforce via lifetimes that the GEM::Object is contained?)
  • The driver provides a callback that creates and initializes a Foo::Object (should this be in a Box<> already? also, the contained GEM::Object would be left uninitialized/zero-inited by default - how do I prevent it being used until it is initialized?)
  • Either on return from the GEM object creation struct, or as some explicit create object function, the DRM abstraction installs the object funcs (in particular the free callback, equivalent to Drop) and calls drm_gem_object_initwhich initializes the refcounting, then leaks the memory (after putting the object in a Pin<Box<>> if not already?) and returns some kind of reference type that wraps the reference instead
  • The reference type implements Clone which increases the refcount, and Drop which decreases it
  • In the abstraction free handler, the pointer to the drm_gem_object passed is upgraded to a Pin<Box<Foo::Object>> again (how, and how do I expose this safely?) and dropped, which invokes the user's Drop handler (if any).

And then shmem makes this more complicated:

  • The shmem free function does kfree itself, so it assumes the user subclass struct has the shmem member at the top (but you still have to kzalloc yourself in the driver object create callback). I guess that means that for the Rust shmem abstraction, instead of this being a user object that contains the SHMem::Object, it should be a SHMem::Object<T> that takes the user fields as a generic parameter? Does it make sense to use this design for the main GEM layer too?
  • The shmem free function also isn't installed directly as a free handler, but rather the driver's free handler is supposed to call it last. So we somehow need to go from a (now owned/boxed, since this is the free handler) reference to a GEM::Object from the GEM side, to whatever type the user driver provided, to its Drop implementation, and then finally the SHMem::Object Drop implementation which needs to be guaranteed to run last (and to run) and actually calls the shmem code to call kfree.

I think I can figure out something to make it work, but there are a lot of subtleties here and I would appreciate some advice from people with more Rust experience! ^^

Wedson Almeida Filho (Sep 01 2022 at 11:08):

You may want to check out rust/kernel/sync/arc.rs for some inspiration. You probably don't need most of what's there, but the main commonality is that we have some struct T in which we need to embed a C struct: instead of doing it the C way, we define an outer struct that contains both T and the C struct (RefInner<T> in the arc.rs).

So my suggestion is for you to define something like gem::Object<T>, which just holds a pointer to a private type gem::ObjectInner<T> (which itself contains a T and a drm_gem_object). You also require that T implement some trait, say gem::ObjectOperations; then you implement unsafe callbacks that use container_of to go from drm_gem_object to gem::ObjectInner<T> and call the interface functions. It's all part of the abstraction implementation and doesn't require users to use unsafe code.
