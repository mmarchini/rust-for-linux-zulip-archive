Gary Guo (Nov 12 2023 at 15:24):

I highly doubt kernel people would like SVD. If I understand correctly:

  • It's format by ARM, largely intended just for ARM embedded systems (although I do see some RISC-V projects adopting it)
  • CMSIS is licensed in Apache and is not compatible with GPL-2
  • It's XML, so maintainability is questionable
  • Adding svd2rust or C equivalent will add another build dependency to the kernel.

I think we'd be better off with just declarative/procedural macros.

I don't know if SVD is usable or not, but the whole concept is one we should investigate for use in the kernel. We should have some kind of intermediate format that describes layout of a block of registers, the fields of those registers, and the allowed values of those fields along with names for these values. It would probably make sense to allow encoding of some kind of documentation as well.

Regarding build time dependencies, I would imagine we would check in the generated code. You would only need the dependency if you wanted to add new types.

I think depending in macros for this is a bad idea, because they are not easily maintained. They can especially be difficult to understand when written without access to crates like syn. An external tool would be far simpler to maintain.

I know that some work is going on to do a similar thing for NVMe, and from what I hear, block layer subsystem is very interested in that work. This work would also rely on third party tools. In fact we should probably sync up with the people who are doing that work, since it is sort of similar.
