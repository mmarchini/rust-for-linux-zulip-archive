Topic: DoubleEndedIterator support for RBTree

HangShu (Dec 02 2025 at 03:33):

Linux RBTree is naturally a data structure that can be traversed from both ends, but Rust's support for RBTree only provides a one-way iterator. When I was writing a module using Rust for Linux, I had a need for a DoubleEndedIterator, so I extended the IterRaw of RBTree to the following code:

/// A raw iterator over the nodes of a [`RBTree`].
///
/// # Invariants
/// - `self.next` is a valid pointer.
/// - `self.next` points to a node stored inside of a valid `RBTree`.
/// - `self.prev` is a valid pointer.
/// - `self.prev` points to a node stored inside of a valid `RBTree`.
struct IterRaw<K, V> {
    prev: *mut bindings::rb_node,
    next: *mut bindings::rb_node,
    _phantom: PhantomData<fn() -> (K, V)>,
}

impl<K, V> Iterator for IterRaw<K, V> {
    type Item = (*mut K, *mut V);

    fn next(&mut self) -> Option<Self::Item> {
        if self.next.is_null() {
            return None;
        }

        // SAFETY: By the type invariant of `IterRaw`, `self.next` is a valid node in an `RBTree`,
        // and by the type invariant of `RBTree`, all nodes point to the links field of `Node<K, V>` objects.
        let cur = unsafe { container_of!(self.next, Node<K, V>, links) }.cast_mut();

        if self.next == self.prev {
            self.next = core::ptr::null_mut();
            self.prev = core::ptr::null_mut();
        } else {
            // SAFETY: `self.next` is a valid tree node by the type invariants.
            self.next = unsafe { bindings::rb_next(self.next) };
        }

        // SAFETY: By the same reasoning above, it is safe to dereference the node.
        Some(unsafe { (addr_of_mut!((*cur).key), addr_of_mut!((*cur).value)) })
    }
}

impl<K, V> DoubleEndedIterator for IterRaw<K, V> {
    fn next_back(&mut self) -> Option<Self::Item> {
        if self.prev.is_null() {
            return None;
        }

        // SAFETY: By the type invariant of `IterRaw`, `self.prev` is a valid node in an `RBTree`,
        // and by the type invariant of `RBTree`, all nodes point to the links field of `Node<K, V>` objects.
        let cur = unsafe { container_of!(self.prev, Node<K, V>, links) }.cast_mut();

        if self.next == self.prev {
            self.next = core::ptr::null_mut();
            self.prev = core::ptr::null_mut();
        } else {
            // SAFETY: `self.prev` is a valid tree node by the type invariants.
            self.prev = unsafe { bindings::rb_prev(self.prev) };
        }

        // SAFETY: By the same reasoning above, it is safe to dereference the node.
        Some(unsafe { (addr_of_mut!((*cur).key), addr_of_mut!((*cur).value)) })
    }
}

I would like to inquire whether RBTree truly requires this double-ended iterator functionality. If it's needed, I can attempt to submit a patch to the community.
