Miguel Ojeda (Aug 22 2024 at 14:24):

what's the best way to determine if equivalent Rust APIs are available?

If you mean in-tree, then you can check the online docs (e.g. for linux-next: https://rust.docs.kernel.org/next/kernel/), or grep, or generally take a look at rust/kernel/.

If you mean things that are getting upstreamed, that would be the mailing list -- the archives are at Lore: https://lore.kernel.org/rust-for-linux/

If you mean things people may have worked on, that gets harder, but you can check the PRs people sent to GitHub, ask here for particular abstractions, and also check @Trevor Gross's table at https://github.com/tgross35/RFL-patch-registry

I'm not sure what I'm doing wrong.

We should probably improve that error message, but the reason is that Rust is not enabled (CONFIG_RUST, even if the toolchain is available).

Miguel Ojeda (Aug 22 2024 at 15:07):

Is that incorrect, should I ignore the branch rust-dev?

rust-dev has some that have been sent to the mailing list, so generally it would have more things than rust-next, yeah, but it is not a 100% guarantee (i.e. @Boqun Feng perhaps did not have a chance to update it, since he usually bases it on top of rust-next I think).

In general, you would want to follow the mailing list and see what is upcoming, since then you can pick the things you need and help reviewing them / testing them etc. for your use case (i.e. Boqun does not pick everything from the mailing list).

The Quick Start # Configuration documentation says it should show up when make LLVM=1 rustavailable reports success.

Note the ", as long as the other requirements are met." part, i.e. there are indeed other requirements -- rustavailable is about the toolchain only:

Rust targets:
  rustavailable   - Checks whether the Rust toolchain is
            available and, if not, explains why.

You can inspect the usual Kconfig requirements and whether they are met for instance using / in menuconfig.

Perhaps we could clarify that sentence somehow ", as long as the other Kconfig requirements are met (i.e. not toolchain ones)").
