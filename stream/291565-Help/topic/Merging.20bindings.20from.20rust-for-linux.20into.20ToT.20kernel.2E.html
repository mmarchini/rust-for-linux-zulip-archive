Miguel Ojeda (Feb 15 2023 at 09:59):

Yeah, it is fairly involved -- I can relate.

If the branch is meant to be used only for upstreaming (i.e. rather than something more persistent), then I would suggest you create a branch where you put your changes on top of the unrelated-to-Binder abstractions/patches you may need. For those, you don't need to keep the exact history, so it should be much simpler -- you can e.g. pick the final files you need, and see who contributed to each, which should be much simpler (or even just saying it is a draft, without signing off the commit).

After that, you can submit an RFC to the mailing list using as base commit that branch (and indicating so in the cover letter), so that the patches you submit are Binder-related, and not other core abstractions/prerequisites. That way reviewers can focus on the Binder changes and not the prerequisite bits. Then, you can rebase that branch (and the patches on top) as needed as the abstractions get upstreamed in the interim.

It is also great to have such a branch, because it tells us exactly what abstractions should be prioritized, if you can share it with us. The Asahi Linux project is doing something similar here: https://github.com/Rust-for-Linux/linux/pull/952.

Miguel Ojeda (Feb 15 2023 at 10:39):

So your suggestion is to base this off the rust-for-linux instead of the upstream kernel?

If you mean the rust branch, no, I meant rust-next (i.e. upstream kernel). In other words, you would have 6.2 in your tree.

In other words, the stack would look something like this:

  • Binder patches N...1, i.e. the ones sent as an RFC to the mailing lists to start the conversation around it.
  • Prerequisite Rust patches to make your Binder bits compile, in a public branch somewhere, i.e. anything unrelated to Binder but that you need, e.g. file_operations. These will eventually land through the Rust tree, and it is useful to know which parts you need to prioritize those.
  • rust-next (currently with some patches on top of Linus' v6.2-rc4 that will land in 6.3).
  • Linus tag (e.g. currently v6.2-rc4).

Then, in a few weeks, Linus creates v6.3-rc1 and rust-next will rebase to that one (for example), landing more Rust patches there, ideally including ideally some abstractions you need. Then, if you wanted to send RFC v2 because there was some feedback on the RFC, you can rebase your "Prerequisite Rust patches" branch into the new rust-next, which should then be smaller if you something you needed is now in mainline, and you rebase the Binder RFC patches on top of that branch, and submit them again. Eventually, there will be no more prerequisite patches you need, and you can finally send the actual patches to be picked by the Binder maintainers, ideally basing it on a Linus' tag directly.

Miguel Ojeda (Feb 15 2023 at 10:49):

Of course, you can also send all the patches (i.e. "Binder patches N...1" + "Prerequisite Rust patches") -- I would recommend that if they are not too many, so that context is clear (marking those clearly as prerequisites).

Miguel Ojeda (Feb 15 2023 at 10:54):

In any case, to be clear, we are not preserving full history of the rust branch either way (though we do our best to keep all contributors via Co-developed-by tags, which is why the original series had so many). So you need to worry about that for Binder patches that you are upstreaming, but for most of the prerequisites, it will be done by whoever upstreams them (which should be the original author(s) ideally).

Miguel Ojeda (Feb 15 2023 at 11:49):

That is of course an option, but I understood Jeff wants to start the conversation with upstream (and that using 6.2 was needed). If sending an RFC/patches is not intended, then I would suggest waiting a bit until we make the rust branch a bit closer to upstream, which would help on that.

On the rebasing: it is definitely time consuming to rebase, but most of the work is making the patches suitable for upstreaming (i.e. splitting the changes properly, writing proper commit messages, linking things, keeping track of contributors and signatures, and so on).
