Danilo Krummrich (Apr 14 2026 at 16:54):

(Cc: @Boris Brezillon)

@Alistair Popple, @Link Mauve The problem is that DRM ioctls are not synchronized with the device / driver lifecycle, i.e. there is no guarantee that the parent device is still bound to the driver.

As @Janne Grunau mentioned the long term plan is to actually provide this guarantee, but it is not just a DRM problem. It is a kernel wide topic around cdev synchronization, DRM is just one affected subsystem.

I definitely plan to (keep) working on this, but in the meantime we can do the following workaround [1], which leverages drm_dev_enter(), drm_dev_exit() and drm_dev_unplug(), which is also used on the C side to protect device resources.

Devres::try_access() basically does the same thing more generically, despite using RCU instead of SRCU, which the DRM helper is based on.

However, for this case I do not recommend to use Devres::try_access(), as we do not really want to access an individual device resource from a scope where we do not have a &Device<Bound>, we simply want a &Device<Bound> for an entire scope.

Besides that, the fundamental problem here is really that we want DRM ioctls to provide this guarantee, whereas Devres::try_access() is really just a last resort. Actually, I think all users have been converted to access(), so we might even remove it, until we find a valid use-case.

That said, the patch in [1] should solve the issue for now. I based it on top of the DeviceContext series, as this would be unsound to implement for a drm::Device that is not guaranteed to be registered.

But for local testing, you can also just cherry-pick the HEAD commit and drop the drm::Device type state.

I hope this helps!

[1] https://git.kernel.org/pub/scm/linux/kernel/git/dakr/linux.git/log/?h=drm/unbind_guard

However for our auxdev use case it doesn't remove the need for the unsafe statement to turn the Device<Normal> into Device<Bound> to get at the PCI parent device data:

    pub(crate) fn get_param(
        dev: &NovaDevice,
        getparam: &mut uapi::drm_nova_getparam,
        _file: &drm::File<File>,
    ) -> Result<u32> {
        let guard = dev.unbind_guard().ok_or(ENODEV)?;
        let nova_core = unsafe {
            guard
                .parent()
                .ok_or(ENODEV)?
                .as_bound()
                .drvdata::<NovaCore>()?
                .get_ref()
        };

Whilst in this case it is actually safe with the protection of unbind guard it would be nicer if we could avoid the use of unsafe, otherwise it's very easy to forget the guard. I'm not really sure how to go about that though so am interested in any thoughts you might have.

Danilo Krummrich (Apr 16 2026 at 10:12):

That's because parent() can not guarantee a bound device for any generic device, but only for an auxiliary device, see [1].

The DRM UnbindGuard (and the DRM device) should leverage the AsBusDevice trait to provide you with the actual auxiliary device -- let me hack this up later today.

Besides, in nova-drm you shouldn't mess with the parent device yourself. Instead you should pass the &auxiliary::Device<Bound> to nova-core, so it can derive the parent itself, gather its private data and do things on behalf of nova-drm.

[1] https://rust.docs.kernel.org/kernel/auxiliary/struct.Device.html#method.parent
