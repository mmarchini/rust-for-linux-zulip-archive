How can I create a data type that contains both a SpinLock and a data type whose creation can fail?

struct Inner {
    a: u32,
    b: u32,
}

struct Other {

}

impl Other {
    fn new() -> Result<Self> {
        Ok(Self{})
    }
}

#[pin_data]
struct Example {
    c: u32,
    #[pin]
    d: SpinLock<Inner>,
    o: Other,
}

impl Example {
    fn new() -> impl PinInit<Self> {
        try_pin_init!(Self {
            c: 10,
            d <- new_spinlock!(Inner { a: 20, b: 30 }),
            o <- Other::new(),
        })
    }
}

new_spinlock! seems to be only usable with pin_init, but Other needs try_pin_init, so…

Philipp Stanner (Nov 14 2025 at 08:44):

Or is it that the return type needs to be a PinInit error version?

Alice Ryhl (Nov 14 2025 at 09:11):

When you write impl PinInit<T> then you are saying that the initializer is infallible. Change the return type to impl PinInit<T, Error>.

And how would I canonically properly create something like Arc<Revocable<SpinLock<Inner>>>?

Ultimately I have

struct Inner;

struct Outer {
  inner: Arc<Revocable<SpinLock<Inner>>>,
}

impl Outer {
  fn new() -> Result<Self>
}

Do I have to describe every stage with pin, and then only the creator of Outer actually allocates everything?

Alice Ryhl (Nov 14 2025 at 09:21):

I think you would write:

inner: Arc::pin_init(Revocable::new(new_spinlock!(Inner::new())))

with Inner::new() being a method that returns impl PinInit<Inner, _>

Philipp Stanner (Nov 14 2025 at 09:34):

But could I then construct Outer::new() -> Result<KBox<Self>> or would I end up with having to return a PinInit<Self, Error>, too?

Alice Ryhl (Nov 14 2025 at 09:37):

Outer::new() can return a Result<Self> or Result<KBox<Self>> or whatever you want.

Alice Ryhl (Nov 14 2025 at 09:37):

The Arc::pin_init call runs the initializer, so you are not forced to return an initializer to the caller.
