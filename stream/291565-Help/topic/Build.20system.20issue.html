It is caused by a git config, intex.skipHash = true, being incompatible with cargo (or libgit2) and it is tracked here. It is usually no big deal to work around it. But in our case the issue triggering another bug in our build system.

The problem is manifested in that changing the skipHash feature for the linux repository does not solve the issue. This is because cargo is searching for a git repository upwards in the file system when building std. But if the build directory is parallel to the source directory, cargo will never find the linux git repository. In my case, it found another repo that I use to store some workflow stuff in. My linux checkout and my build directory just happen to live inside this git tree.

A workaround for me is to change the skipHash feature for my workflow repository. It cost me a few hours to figure this out, so I post it here if anyone is hit by this.

Also, this only occurs when we use cargo to build std for testing, when skipHash is enabled on the first git repo that is encountered on upwards traversal from the build directory. So probably a bit rare. But annoying none the less.

If anyone has some input on how to actually fix this, I am all ears.
