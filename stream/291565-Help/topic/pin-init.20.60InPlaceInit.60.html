Benno Lossin (Mar 08 2023 at 16:11):

At the moment pin-init provides the following trait:

pub trait InPlaceInit<T>: Sized {
    /// Use the given initializer to in-place initialize a `T`.
    ///
    /// If `T: !Unpin` it will not be able to move afterwards.
    fn pin_init<E>(init: impl PinInit<T, E>) -> error::Result<Pin<Self>>
    where
        Error: From<E>;

    /// Use the given initializer to in-place initialize a `T`.
    fn init<E>(init: impl Init<T, E>) -> error::Result<Self>
    where
        Error: From<E>;
}

it is implemented by Box and UniqueArc.

My current problem is that this references the error::Result, i.e. the kernel Error type.
As I also want to publish this as a standalone crate - and the more rusty pattern is to allow arbitrary error types, I have to find a different error type. However, that is not as easy as using Result<_, E>, since inside of pin_init and init, I want to e.g. allocate a Box, so I first tried to do this (only showing init for simplicity):

pub trait InPlaceInit<T>: Sized {
    fn init<E>(init: impl Init<T, E>) -> Result<Self, E>
    where
        E: From<AllocError>;
}

This works, but places a type requirement on the initializer. Now one cannot do

Box::init(init!(Foo {
    ...
}));

since init! returns impl Init<Foo, Infallible>.

My next try was this:

pub trait InPlaceInit<T>: Sized {
    fn init<E, Ei>(init: impl Init<T, Ei>) -> Result<Self, E>
    where
        E: From<AllocError>,
        E: From<Ei>;
}

This is better, since now one could theoretically do the above example with Foo, but AllocError: From<Infallible> is not satisfied (I think this could be added easily, I asked a question about this on the rust-zulip, but have not gotten an answer).

Does anyone have any other ideas?
