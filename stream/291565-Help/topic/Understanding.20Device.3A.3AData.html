One common issue is userspace having an open file descriptor to a device that was removed or made inaccessible in some way. When that happens, the driver has to keep the userspace client state around, and just fail all I/O operations, while considering the device unbound and cleaned up. A lot of subsystems don't get this quite right today, e.g. I've seen it go wrong with FireWire. Hot-unplug a host controller connected via Thunderbolt, and the driver/core gets stuck in unbind until all userspace processes that had devices open are manually killed. In this case, that probably means the driver isn't designed to be able to safely unbind and remove itself while still having dangling userspace references open, so it just waits until they go away... (and of course, the kernel isn't allowed to kill userspace processes for no reason, so it has to wait, potentially forever, and that has to be decoupled from the existence of a device bound to the driver in a properly designed driver!). In this case, you might need to actually keep some kind of producer state around even after the device is long gone, potentially whole Registrations, even if they are zombies that fail all operations...

I also still don't quite get all the wrappers, like how Ref/RefBorrow are used with device data, and the PointerWrapper trait which seems to be used for context data for consumer APIs.

I think it might be useful to have some prose-style documentation about this, to make it easier for driver authors to understand what the Rust-for-Linux model is here (especially for people not that familiar with Rust, which includes most Linux kernel developers!). I could help write that, but first I'd have to understand it myself ^_^;

Wedson Almeida Filho (Old Google account) (Aug 19 2022 at 13:08):

Answering some of the questions:

Rust mostly mimics how things are done in C wrt device lifetime, on removal (for whatever reason) a remove function on the device is called, and that is responsible for properly cleaning things up, including ensuring that IO resources aren't used after removal, either because the device is not present anymore (e.g., it was unplugged) or because another driver may now be using it (e.g., this driver was unbound and a different one was bound) -- in any case, those resources _must_ not be used anymore after removal.

Now, experience with C has shown us that drivers don't always get this right. So we introduced a new concept in Rust, revocable structs: the fields are accessible until revoke is called, after which the fields are destructed (drop is called on them) and they can't be accessed anymore (the implementation also ensures that ongoing uses complete before dropping everything). So we get help from to compiler to ensure that parts of driver data (e.g., IO resources) are no longer accessible after remove. However, we still need drivers to call revoke on the relevant fields and they may forget to do so.

So we require device data to implement the driver::DeviceRemoval trait, which only has one method called device_remove, which is called automatically by subsystems after remove is called on a device. It's a safety mechanism to force developers to think about what they want to revoke when a device is removed.

I also implemented a device::Data<T, U, V> struct that categorises things as described in its documentation. And revokes things on device_remove per the order described in the documentation. But this is just a convenience struct, if the order doesn't suit you, you can just implement your own struct and implement the driver::DeviceRemoval trait.

The pl061 driver uses this: every time we touch io resources, we first need to check if they're still accessible (we can't access them without this check). This is because gpio drivers also expose a char device to userspace, and unregistering just prevents new files from being opened, but existing ones remain open, so they still have a ref-counted pointer to device data. But in Rust we ensure that this ref-counted pointer cannot be used to touch IO resources (io mem in this case) after device removal because that part of the struct is revoked.

Does it make sense? Happy to try to clarify anything else that doesn't seem to make sense.
