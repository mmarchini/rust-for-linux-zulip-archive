So I need to have custom data which is only meaningful for abstraction to test if probe is finish or not. So basically AbstractionPrivateData. But question is where to store this so we can access this data during callbacks. Do we store this along with driver drvdata or along with serdev_device_driver.

We would do register like this in callback we could cast *serdev_device_driver to SerdevAbstraction.

Or with Drvdata we would do two step initializing. This way we set_drvdata beginning of probe_callback but initialize driver field end of probe_callback

#[pin_data]
struct Drvdata<T: Driver + 'static> {
    probe_done: Atomic<u32>,
    #[pin]

}

Kari Argillander (Dec 28 2025 at 12:08):

So idea is that in probe we do

            dev.set_drvdata(Drvdata::<T>::new_uninit())?;

            // SAFETY: Safe as we just set drvdata.
            let data = unsafe { dev.drvdata_borrow::<Drvdata<T>>() };

            let driver_init = T::probe(opener, info);
            // SAFETY: Only called once during of `Drvdata` lifetime.
            unsafe { data.init_driver(driver_init)? };

That way we multistate initialize drvdata an that atomic makes sure that we only access actual driver data after probe is called successfully.
