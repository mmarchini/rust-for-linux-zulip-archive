Topic: Rust ownership of C allocated objects

Ariel Miculas (May 16 2023 at 13:40):

I found myself in a situation where I would like the reverse functionality of the ForeignOwnable trait: that is, get a C allocated object, transfer ownership to Rust, then release it at the end (or implement Drop and free it that way).
Specifically what I want to achieve is to call clone_private_mount when my filesystem is initialized, store the result in s_fs_info and make it available in fs::file::Operations::open.
I can do that with a rust allocated object via ForeignOwnable:

  • call into_foreign at init and from_foreign at kill
  • call borrow on the raw s_fs_info pointer and pass that pointer to open => I get a reference to my object of type ForeignOwnable::Borrowed

Ultimately I want to define a rust structure which includes a C allocated object in some way (e.g. reference or Box), pass that to the C side (in s_fs_info) and get it back in rust (in the open callback).
So:

  • is there a way I could get a ForeignOwnable object from a C allocated object? (the safety requirements suggest I can't)?
  • is there a way to transfer ownership to rust from a C allocation (maybe it wouldn't be sound, since rust wasn't responsible for its allocation)?
  • should I store a static reference to the C allocated object in my structure? (seems unsafe at first thought, but if I only deallocate the memory when the filesystem is unmounted, maybe it's the right approach)

Andreas Hindborg (May 16 2023 at 13:48):

You can wrap a pointer to your C struct and implement Drop for the wrapper, call kfree() or whatever cleanup you want in drop.

Andreas Hindborg (May 16 2023 at 13:51):

I'm not familiar with the API you are using, but perhaps you could call the allocator from the Rust side? Pages does this: https://github.com/Rust-for-Linux/linux/blob/rust/rust/kernel/pages.rs
