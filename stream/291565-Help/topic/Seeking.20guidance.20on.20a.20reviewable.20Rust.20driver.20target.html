wenzhao (Apr 04 2026 at 20:19):

Hi Miguel, thanks for reaching out! You've captured the core of it, but I want to clarify a few important nuances to address your questions.

1. Regarding "AI-generated code" and the tooling: Yes, the workflow does utilize LLM components under the hood for the C-to-Rust semantic translation. However, it is not a raw code generator. It is a strictly structured pipeline designed to enforce safety boundaries (e.g., ensuring #![forbid(unsafe_code)] in the driver logic).

I want to emphasize that our goal is never to dump raw AI output onto the mailing list or treat maintainers as our debuggers. We heavily manually audit, locally test, and verify the generated code before submitting anything. What we are seeking from the community is not "finding AI bugs," but rather architectural review: Are the Rust abstractions used idiomatically? Is this the right way to model the kernel APIs?

2. The target (In-tree vs Out-of-tree): Exactly as you noted, the nlmon feedback taught us that unsolicited rewrites of working in-tree C drivers are generally not welcome. Therefore, we are looking for:

  • An in-tree module where the maintainer actively wants a Rust reference to exercise their new abstractions.

  • OR, an out-of-tree/missing driver that we could implement in Rust and help upstream.

3. Is the tooling open source? The tooling is part of an ongoing academic research project focused on sound systems migration. It is currently in the active prototyping phase and not yet open source. However, we fully intend to open-source the toolchain alongside our research publication. Right now, we want to contribute useful, well-tested driver code to RFL to ensure our methodology aligns with real upstream standards.

That is definitely better, but it all depends on the degree of review that has been done etc. It will likely heavily depend on the maintainer you need to talk to, as well as the quality of the patches submitted. In other words, it is all whether the time investment is worth it -- hard to say "in the abstract" how it will go. For instance, very bad LLM patches (even if audited by a human), may not be worth it; but good patches that are close to what a kernel developer would have submitted would be more successful.

In addition, please note that, submitters of code generated by tools are expected to understand the code as if they had written it themselves. That is, even if a tool generates perfect code, the submitter should still be able to answer questions about it etc.

Finally, maintainers have discretion in how to handle contributions like this, so you will definitely want to find someone that is open to this (and I thank you for being open about your system too!).

For the official kernel guidelines on how to handle generated content (including LLM-assisted content etc.), please see:

  • https://docs.kernel.org/process/generated-content.html
  • https://docs.kernel.org/process/coding-assistants.html
