Miguel Ojeda (Jun 24 2021 at 00:17):

There are several issues:

  • hv_fcopy_cancel_work is a static function, i.e. it is not exported, so nobody outside hv_fcopy.c can use it, not even other C files (translation units).
  • The function is not designed to be used by other files: it manipulates static global variables -- if you make the function non-static, then you will be manipulating the internals of the other kernel module!
  • Rust drivers should only use abstractions from the kernel crate, not access arbitrary C functions. The abstraction in rust/kernel/ is the one that has access to the bindings, which are generated from the headers in rust/kernel/bindings_helper.h.

Miguel Ojeda (Jun 24 2021 at 00:32):

So it all depends on what you are trying to accomplish. If you are attempting a rewrite of hv_fcopy.c in Rust, then you should first add the abstractions that you need (e.g. a work queue) in rust/kernel/. These abstractions call the usual kernel APIs. Then, write hv_fcopy_rust.rs to use those abstractions.

Miguel Ojeda (Jun 24 2021 at 00:32):

However, if what you are attempting is to write a kernel module partially in C and partially in Rust and have C functions call Rust ones and viceversa within the same kernel module, then we do not really support that. It is technically possible, e.g. you can "hack it" by declaring the C functions in the Rust file and so on (like you are attempting to do in hv_fcopy_rust.rs), but please do not do so. The reason is that we want to develop Rust abstractions over time which are as safe as possible, and have drivers only depend on that API.
