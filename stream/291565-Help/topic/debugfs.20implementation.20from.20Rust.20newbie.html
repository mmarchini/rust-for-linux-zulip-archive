Andrew Ballance (Apr 08 2025 at 23:17):

OK, so i took a quick look at your code
it looks like benno already fixed the compile error.
i have a couple of notes:
1) it looks like some of your abstractions are unsound
debugfs_create_file_full takes a &'a mut T as an argument but the returned dentry stores &mut T so if anyone tries to read the data after T is dropped bad things will happen.
you could either add a associated lifetime to struct DebugfsEntry or (what i would recommend) require that T impl ForeignOwnable and get the data through into_foreign / from_foreign

2) debugfs_create_file_full will always fail to set the file operations
because you pass a reference to a stack allocated fops: bindings::file_operations, as soon as it returns, the fops is dropped. so the pointer stored in the dentry points to invalid memory

also if you are planning to upstream this, public public api's should wherever possible avoid using bindings. I would recommend taking a look at the #[vtable] macro and the miscdevice api https://rust.docs.kernel.org/src/kernel/miscdevice.rs.html because they already build a vtable based on file_operations.

Benno Lossin (Apr 10 2025 at 22:52):

Timur Tabi said:

What does #[vtable] actually do?

for every function foo that you have in the body of the trait, it gives you access to a bool constant HAS_FOO that is true when the implementer of the trait has specified the method (for this to work, you need to specify a default implementation in the trait definition). it allows you to set it to NULL when it isn't defined, as that AFAIK makes the C side select the default implementation

Benno Lossin (Apr 10 2025 at 22:52):

that way we don't have to reimplement the default behavior on the rust side
