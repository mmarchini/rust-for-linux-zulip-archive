Topic: Using device::Data with blocking operations

Maciej Falkowski (Feb 04 2022 at 16:56):

Hello,

The new platform API uses device::Data<T, U, V> for holding device-specific data with resources field being wrapped in revocable type Revocable<T> which uses rcu_read_lock() underneath. This makes all usages of device resources happen in RCU read-side critical sections
that prevent usage of blocking operations. In the driver for trng generator that I try to convert, the i/o address is polled using the readl_poll_timeout() function that is a blocking operation (calls might_sleep_if()) as shown:

fn read(trng: RefBorrow<'_, DeviceData>, data: &mut [i8], _wait: bool) -> Result<u32> {
    let trng = trng.resources().ok_or(Error::ENXIO)?;
    /// ...
    let _ =
            trng.mem
                .readl_poll_timeout(EXYNOS_TRNG_FIFO_CTRL, |val| *val == 0, 200, 1000000)?; // called in RCU read-side critical section
       /// ...
}

How should access to device resources be done in a blocking context?

Wedson Almeida Filho (Old Google account) (Feb 05 2022 at 19:47):

There are two options: the device state only really needs to implement the DeviceRemoval trait, that is, it needs to implement a device_remove callback that gets called when the device is removed (to release resources and ensure that they're not used anymore). So one option is for you to implement a struct of your choosing and then implement the trait.

Alternatively, (and preferably IMO) I'm working on a revocable rwsem (not ready yet, but you can see it here) which is meant to be used as device data. A not yet ready driver that uses it is available here. The idea is that you'd call try_access or try_lock to access the data (shared or exclusively, respectively), and it only succeeds if access hasn't been revoked. Access is revoked when device_remove is called -- and it acquires the rwsem in exclusive mode while revoking access, so the data is guaranteed to remain accessible to anyone holding a guard returned by try_access/try_lock very much like Revocable<T>, but they can also sleep because they're not in an RCU read-side critical section. I intend to finish and this out for review some time next week.
