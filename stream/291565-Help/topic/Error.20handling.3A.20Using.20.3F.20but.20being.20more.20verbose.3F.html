int err = try_push(numbers, 72);

This makes sense, since unwrap_or_else usually would be used in a case where you still are in the happy path, but the easy/fast path did not work.

You essentially have three more options (aside from manually matching it):

  1. add additional context to the error type, in userspace rust this is what the anyhow crate does with its Context trait.
  2. create a macro that does the match for you
  3. create a function that does the match for you

Trevor Gross (Jan 23 2024 at 18:54):

The errors themselves probably don't have any info about where they came from, so I don't think there is a perfect solution. But you can do something like this pretty easily for each line:

 let foo = a.b().inspect_err(|e| dbg!(e))?;
 let bar = foo.baz().inspect_err(|e| dbg!(e))?;

Result::inspect_err lets you easily do something to the error without touching the value. It is unstable so you need #![feature(result_option_inspect)], but it will be stable in the next version (1.76 in 16 days :smile:)

Then the dbg! macro (taken from std::dbg!) will print a value with debug formatting, including the file and line number. It also returns the value, so you can put it around anything without affecting the types/values, let foo = bar.baz()? gets the same as let foo = dbg!(bar).baz()?. Pretty handy for debugging

Trevor Gross (Jan 23 2024 at 18:56):

Also file! and line! should be in the prelude so I don't think you need to prefix them with ::core::

Trevor Gross (Jan 26 2024 at 05:45):

Part of the reasoning is that most errors in RFL are just thin wrappers around errno, so there is not really any way you could attach context to that. So the next best thing is to print error context, which ScopeGuard and things like inspect_err help you do.
