Ewan Chorynski (Jun 16 2023 at 13:35):

Hello, I am getting started in kernel development and interested in rust in the kernel.
However I still struggle to understand some aspects.
Particularly how this FFI call works ?

#[doc(hidden)]
#[cfg_attr(not(CONFIG_PRINTK), allow(unused_variables))]
pub unsafe fn call_printk(
    format_string: &[u8; format_strings::LENGTH],
    module_name: &[u8],
    args: fmt::Arguments<'_>,
) {
    // `_printk` does not seem to fail in any path.
    #[cfg(CONFIG_PRINTK)]
    unsafe {
        bindings::_printk(
            format_string.as_ptr() as _,
            module_name.as_ptr(),
            &args as *const _ as *const c_void,
        );
    }
}

How the C code could possibly interpret the pointer to args wich is of type fmt::Aruments ?
I was looking for some documentation or piece of code that do the adaptation but can't find anything.

Miguel Ojeda (Jun 16 2023 at 14:04):

We call _printk with a special format specifier (%pA) which calls back into Rust, so the C code does not need to understand the type. See the pointer and rust_fmt_argument symbols.

Miguel Ojeda (Jun 16 2023 at 14:05):

 * There is also a '%pA' format specifier, but it is only intended to be used
 * from Rust code to format core::fmt::Arguments. Do *not* use it from C.
 * See rust/kernel/print.rs for details.
