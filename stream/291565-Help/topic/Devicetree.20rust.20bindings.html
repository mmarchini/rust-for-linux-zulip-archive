    pub fn property_read_bool(&self, name: &CString) -> bool {

    pub fn property_read<T>(&self, name: &CString) -> Result<T>
    where T: Copy,
    {
        let val = Self::property_read_array::<T>(&self, name, 1)?;
        Ok(val[0])
    }

    pub fn property_read_array<T>(&self, name: &CString, len: usize) -> Result<KVec<T>> {
        let mut val: KVec<T> = KVec::with_capacity(len, GFP_KERNEL)?;
        let val_ptr = val.as_mut_ptr();
        let actual_len;

        unsafe {
            match size_of::<T>() {
                1 => actual_len = bindings::device_property_read_u8_array(
                        self.as_raw(),
                        name.as_ptr() as *const i8,
                        val_ptr as *mut u8,
                        len
                ),
                2 => actual_len = bindings::device_property_read_u16_array(
                    self.as_raw(),
                    name.as_ptr() as *const i8,
                    val_ptr as *mut u16,
                    len
                ),
                4 => actual_len = bindings::device_property_read_u32_array(
                    self.as_raw(),
                    name.as_ptr() as *const i8,
                    val_ptr as *mut u32,
                    len
                ),
                8 => actual_len = bindings::device_property_read_u64_array(
                    self.as_raw(),
                    name.as_ptr() as *const i8,
                    val_ptr as *mut u64,
                    len
                ),
                _ => return Err(EINVAL),

            }
            if actual_len <= 0 {
                return Err(Error::from_errno(actual_len));
            }
            val.set_len(actual_len.try_into().unwrap())
        }
        Ok(val)
    }

Dirk Behme (Nov 11 2024 at 08:51):

@Rob Herring I have a conceptual question regarding the data (the integers or array of data) we read from the device tree: Is this data supposed to be "trusted" or "untrusted"?

  • "trusted" does mean the data read can be used "as is" by the reader. Without further validation. Because it can't do any harm.
  • "untrusted" in this context does mean that the data read should be validated by the reader before using it. For example if the values are in the expected range etc. And only if this validation check is successful the data can be used in the driver.

Some background: Some weeks back @Benno Lossin and @Greg KH discussed about the concept of "Untrusted data abstraction" (and Greg's answer). The idea is to consider data coming from an "external data source" as untrusted and make it available to the reader only via an additional validation step.

So I wonder if this might applicable for device tree data, as well?

Thoughts?

Rob Herring (Nov 11 2024 at 13:39):

Both? The data coming from the bootloader is trusted the same way the kernel itself is and if you want to verify that, then the DTB needs to be signed and checked like the kernel. For overlays which change the DT at runtime, it may be a bit more complicated. However, there's no general purpose interface for loading overlays at runtime and it's only possible with specific usecases on a few devices. I'm not sure what we'd consider "validated". Is that the overall DTB structure is valid? Or all properties only contain valid values? Or both? We can do the latter with schemas, but doing that within the kernel isn't likely feasible.
