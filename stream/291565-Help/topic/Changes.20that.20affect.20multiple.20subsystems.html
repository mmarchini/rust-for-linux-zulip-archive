Topic: Changes that affect multiple subsystems

Konstantin Andrikopoulos (Dec 02 2024 at 00:06):

I am trying to get a patch submitted that affects both rust code, and code in the vsprintf subsystem. I see that I should send my changes to both subsystems. Does this mean that I should create 2 patches, one that contains rust-related changes, and one that contains vsprintf related ones?

Miguel Ojeda (Dec 02 2024 at 07:42):

It depends on the change. In general, if a change can be split into several patches and each patch still makes sense on its own (even if it requires depending on a previous patch), then it should. But you shouldn't split just because it is a different subsystem: when it is possible/reasonable to do so, that is great, because it typically helps upstreaming and reviewing, but if the change cannot be split, then it should not be. For instance, commits must build on their own, i.e. when applied one-by-one, so you cannot split certain changes even if you wanted.

In your particular case, it would be OK to put it in the same patch, since it is, after all, reorganizing the safety comments/preconditions. But it would also be OK to do the C change as a second patch too in the same series if you think the first patch still makes sense on its own.

Either way, I wouldn't worry too much -- maintainers may ask you to split or merge sometimes, when they see the actual patch(es).

I hope that helps!
