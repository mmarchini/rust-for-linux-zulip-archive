Topic: initializing name fields

Valentin Obst (Jan 30 2024 at 08:28):

The structure that I need to supply when registering my module with the kernel has an embedded name field that is just a fixed-size character array. It was a bit of a pain to convert a CStr into the fitting type. I currently do it like that:

const fn to_name_field<const N: usize>() -> [i8; N] {
    let mut name_field: [i8; N] = [0; N];

    build_assert!(T::NAME.len_with_nul() <= N);

    // SAFETY: We just checked that it fits and it is impossible for `T::NAME`
    // to overlap with a local variable.
    unsafe {
        ptr::copy_nonoverlapping(
            T::NAME.as_bytes_with_nul().as_ptr() as *const i8,
            name_field.as_mut_ptr(),
            T::NAME.len_with_nul(),
        )
    };

    name_field
}

ops.name = Self::to_name_field::<16>();

There are a couple of things I dislike about this:

  • as it is a common problem, it fells like a method like this should be on CStr; abstractions for a particular subsystems are not the right place.
  • build_assert fails at the linking stage, but I would like to fail it when compiling; tried static_assert, but it can not deal with the generics.
  • it feels wrong to hard-code the size in the const generic parameter at the call site but I did not find an easy way to get the size of the structure field.
  • it requires an unstable feature to use *mut in const contexts

Did someone here already bump into this kind of problem, is there maybe even a clean solution for that I missed?

Alice Ryhl (Jan 30 2024 at 08:45):

Instead of copy_nonoverlapping, you should be able to manually copy T::NAME.as_bytes_with_nul() over. A loop that copies one byte at the time should work in const.

const fn only means that the function can be used in places where a const fn is expected, so in other const fn's, const variables and array lengths.
