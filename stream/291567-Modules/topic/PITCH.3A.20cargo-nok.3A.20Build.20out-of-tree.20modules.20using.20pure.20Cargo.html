Tiago Dinis (Jun 14 2026 at 12:26):

I wanted to share a project I've been working on called cargo-nok (No-Kbuild)

https://github.com/ardos-os/cargo-nok

It's an experimental tool designed to build out-of-tree Rust kernel modules while keeping Cargo fully in charge of dependency resolution, without invoking the usual external-module Kbuild flow or Makefiles. It discovers the active kernel tree, derives targets/cfgs, and handles the modpost/objtool/linking stages to produce a loadable .ko, and cargo-nok is implemented within a single rust binary.

The RA Problem: Since rust-project.json doesn't play well with cargo, I also built a custom Cargo proxy launcher for rust-analyzer. It intercepts cargo metadata on the fly to inject the kernel crates (kernel, bindings, uapi, etc.) into the workspace graph and feeds the precompiled proc-macro .so artifacts back to RA so macros like module! expand perfectly.

Rust analyzer crate graph:

Currently tested on x86_64, but I also coded support for some other architectures.
Everything works just as you would expect from a no_std rust crate, it integrates with rust analyzer without rust-project.json generation, you can install crates from crates.io such as bitflags, lazy_static and popular crates, and everything just works like a normal userspace crate, all you need to do is create a normal cargo library and set the crate type to rlib and just start coding your module. Hopefully this will reduce by orders of magnitude the barrier of entry for coding for the linux kernel.

Would love to hear your thoughts!

Of course not every crate will be compatible because some rust APIs are different since linux uses custom versions of core, alloc, compiler_builtins and other essential crates, but a lot of crates should work flawlessly.

Tiago Dinis (Jun 14 2026 at 14:20):

If you remove float support from serde (I forked serde to add a floats feature: https://github.com/coffeeispower/serde/) , it also works, you can use it with cargo nok and even use things like postcard which integrates with serde to encode and decode structs from userspace to kernel space and viceversa. This allows removing a lot of boilerplate code to safely read parameters from userspace through syscalls like ioctl, you're not thinking in bytes anymore, you're thinking in types and structs and the encoding becomes buried inside serde which increases productivity too.
