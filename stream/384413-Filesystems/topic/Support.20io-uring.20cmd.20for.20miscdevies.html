Hi, I'm working for supporting io-uring cmd in file operations for miscdevices. in this branch

https://github.com/SidongYang/linux/tree/rust-next-io-uring-cmd

As a newbie, I don't know the best way to make abstraction for address (or memory chunk?) based data struct like cmd in sqe, pdu in uring_cmd. Currently, IoUringCmd::pdu() just returns &[u8;32] and maybe caller should cast it unsafely. Or it would be better to make marker trait like unsafe trait PduLayout {}?

Boqun Feng (Jun 20 2025 at 17:50):

Hi Sidong, thanks a lot for taking a look into this. For your question, I think it actually depends:

  • For cmd in sqe, I think an sqe is accessible to the userspace at all times, right? If so, you either need to provide a customized wrapper or use [Opaque<u8>], because since userspace can change the value anytime it wants, kernel cannot assume it's a valid [u8].
  • For pdu in uring_cmd, I believe io_uring_cmd is a in-kernel only data struct, so it should always be owned by kernel only, however, since different cmds may define it differently (as different data type), so probably you should use &mut MayUninit[u8; 32], but I think pdu() should take a &mut self.

Please notice these are very preliminary answers, depending how you can and want to design the lifetime management of each struct, the answer may vary.
