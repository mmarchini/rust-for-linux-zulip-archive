Miguel Ojeda (May 05 2024 at 22:18):

Hey Thomas, sorry for the late reply. I talked about this briefly with @Wedson Almeida Filho a few days ago.

One quick note on the paths: rust/kernel/ is the current place for abstractions, but eventually the code will be moved into each subsystem when we split the kernel crate. So whether it is in rust/kernel/ or not is not really important (regarding the "WIP" status).

Now, if you think upstreaming without the VFS abstractions is the only sensible way to bootstrap the effort, then as long as the code is properly marked as WIP/staging and Kent and other FS maintainers are happy with that approach, it should be fine. However, I would say it should be made very clear in the code/docs/... that the goal is to eventually use proper abstractions etc.

One way to do that would be putting the "staging" marker directly in the path, like in drivers/staging/. Then essentially you have your staging area.

So, in summary: it is not what we would recommend, but it may be fine if you can justify it, other FS maintainers agree, you commit to eventually use proper abstractions and you are happy tagging it as fs/staging/ or similar meanwhile. In other words, what we are looking for is that subsystem maintainers are aware that the goal is to have the abstractions in place and used, but exceptions sometimes need to be made. :)
