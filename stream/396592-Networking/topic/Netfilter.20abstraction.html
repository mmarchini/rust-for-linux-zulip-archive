Amélie Gonzalez (Feb 21 2025 at 14:15):

Hi!

A bit of a delayed response but i guess if it ever reaches you it might still explain what's going on.

When Rust support was officially merged upstream, circa October 2022, the project left aside a lot of original work that had been done independently in the rust branch of the repo to instead upstream abstractions and users progressively upstream with feedback from other developers. As a result, because nobody focused on it, kernel::net::filter was never upstreamed.

Nowadays, people seemingly develop over rust-next, and you will need a user:

In general, the kernel does not allow to integrate code without users, but exceptions can potentially be made for Rust code to simplify the upstreaming process early on. That is, upstreaming some dependencies first so that it is easier to upstream expected in-tree users later on. However, note that this is not meant to be a way to justify upstreaming APIs that do not have agreed upon in-tree users. In particular, out-of-tree modules do not constitute a user in this context.

This has been a problem in general in networking for a while. All the users we've had have been PHY devices (we can't rewrite drivers either). Adding support for NAT64 could constitute a user, and therefore a reason to finally bring back netfilter abstractions, but i'd leave the appreciation of that bar to people who actually know about these components in depth, and it's likely you would only be allowed (and it'd be preferable that you only write) the bits of abstraction that are useful for your particular user.
