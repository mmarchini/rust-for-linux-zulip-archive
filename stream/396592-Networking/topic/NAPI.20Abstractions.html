Boqun Feng (Aug 16 2023 at 16:44):

I don't think you misunderstand anything. It's just a "as time goes on" thing ;-)

Rust support is still in the experimental stage: no one yet knows the scope of it, will Rust be used for all the drivers or just some? will Rust also be used for core kernel part (e.g. scheduler)? These are still questions required to answer. So it's natural to expect that people ask the hard question "why Rust???". Besides different people have various availability that they can work on Linux kernel, some may have extra time to spend on Rust and some may not (unless it's proven to be useful). So it might seem that Rust abstractions/bindings are treated differently in different subsystem (also some APIs are more general than others, e.g. locks vs sockets), but it's just the nature of decentralized open source projects.

Now I think the important question is "how we should react accordingly", a few thoughts of my own: although the merge (and the review from netdev) won't happen immediately, but as Trevor and I mentioned, nothing could prevent you from sending these bindings, you can keep improving it until the use cases are convincing enough. This will provide the abstractions for potential users and also persistently working on something is a good way to earn trust.

Trevor Gross (Aug 16 2023 at 17:20):

Are you focusing on out-of-tree modules rather than something that is likely to be upstreamed? If so, definitely drop that specific term because it could be a source of confusion if they're thinking you are looking to support a future in-tree module. I think that generally there isn't a lot of code specifically to support OOT modules that isn't used by some IT mods (this is based on the notion that most things should be brought in-tree if they are to be maintained for >1 kernel version) but net may have exceptions since they are higher level. Just reply with the following:

  1. Definitely include something in samples/rust (you should just reply to the header email with the sample patch for now, but include it in your patchset if you send a v2)
  2. If your intended use case is out-of-tree, state that clearly and ask what their policy is for accepting OOT-specific things (and expect that they will likely say they don't accept them)
  3. Regardless of IT or OOT, link to any ongoing work you may have that uses this abstraction. Be clear about whether you eventually plan to try to get this work upstreamed
  4. Ask if they have any specific work that they are looking to get in-tree and would use these interfaces - even if you can't work on it, somebody else on RFL may be able to
  5. Respect that it may be a while before these abstractions are upstreamed. Yes - the long term goal is to have RFL work almost everywhere. But the way we get there isn't just by writing abstractions for everything into the kernel - it's by starting with core abstractions (which we have) and then slowly branching out with (hopefully) abstractions along with uses for them, and with support from the maintainers (who generally don't like maintaining dead or OOT-only code, even though some might be OK with it). Keep in mind that RFL still has a lot of winning over to do

And then open a PR with your bindings to the RFL repo if you haven't already, so others who want to use them can

To be clear, there is no such distinction, i.e. it is not about abstractions vs. full-fledged in-tree use cases.

The repository and the project is intended for in-tree kernel development. It is part of the kernel, e.g. it is in GitHub for convenience, but it could be just as well in git.kernel.org.

This means that, for instance, if we create a rust-net branch (and later perhaps rust-net-next etc.), the intention is to have a single place to collaborate easily on things that are eventually intended to go upstream in some form or another. It does not need to be perfect or final, of course, but it needs to be intended for upstreaming. This implies it cannot just be for out-of-tree purposes.

This does not mean we cannot have a branch for iteration or experimentation. But it means that, if at some point we decide something in the branch will not make it to upstream for whatever reason, we will remove it. That is, out-of-tree users cannot rely on the branch nor expect support from it -- same as for the in-tree kernel APIs.

It also means an experimentation branch on our side like rust-net should also have users in the branch. Exceptions can be made there for new abstractions while in development, but again, the idea is that the experimentation is targeted towards upstream.

Furthermore, sending abstractions to the mailing list is OK, and it is the intended procedure. However, the abstractions sent need to have a clear use case in-tree. Thus, when contacting kernel maintainers, it is important to clarify what that use case is (as well as justifying the use case itself, e.g. why it is worth the maintenance effort and so on), just like for any other patch that goes into the kernel.

I will add a few clarifications to the webpage and perhaps a new section on out-of-tree vs. in-tree.
