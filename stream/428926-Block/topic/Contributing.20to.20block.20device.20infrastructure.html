I'm interested to see the block::mq crate completed to a degree that it is possible to write real block device drivers.

As far as I could determine after the implementation of Request, which allowed to create the rnull driver, the work has stalled.

Thus I am wondering if I could help things move forward with contributions.

To get started I created an implementation of a wrapper around bio and bio_vec which is completed enough that it allows drivers to transfer data from/to the block layer. For most parts it wraps the C structs and implements iterators such that the bio's of a request and the segments of a bio can be walked.

With this - using it together with the existing Page wrapper - I created another proof-of-principle block driver 'rrandom', an extension of the rnull driver that returns random numbers.

Would, generally speaking, contributions along these lines be welcome? Does it make sense for me try to upstream this code, to send patches for review?

Setting up some vocabulary:

  • hw context is the context where hardware completions are processed. This may be irq context. There may be fewer hw contexts than sw contexts, leading to Operations::complete being scheduled on a different core than the hw context via IPI.
  • sw context is the context where requests are issued and completed.

If we pass ARef to Request::complete, we need to synchronize on the refcount to increment from 1 to 2 in the hw context (potentially IRQ). We then do ARef::into_unique when Operations::complete is called in the sw context.

If we pass UniqueRef to Request::complete, we need to synchronize on the refcount to decrement from 1 to 0 in the hw context. We do not need to change the refcount in the sw context. However, we do need to synchronize, otherwise writes to the UniqueRef<Request> might not propagate to the sw context in time. Read acquire until 0 is observed on the refcount would do.
