Benno Lossin (Mar 21 2026 at 08:35):

I have an idea to fix the merged issue. I'll explain it along what I think the workflow should be in general:

  • all normal PRs target the staging branch, we have a github workflow that moves any PRs that don't have the correct target,

  • when a PR is ready for merging, instead of pressing the merge button, we issue a command via a comment that contains just the phrase/stage. another github workflow catches this and then performs the actual merge:

    1. create a new branch staging/pr-{num} that points at staging where {num} is the number of the PR
    2. change the target of the PR from staging to staging/pr-{num}
    3. merge the PR to staging/pr-{num} without a merge commit
    4. rebase staging/pr-{num} applying review trailers, adding a link to the PR, and the SoB of the maintainer who authorized the merge
    5. checkout staging, merge staging/pr-{num}, customizing the merge commit message to include the original PR's branch name. Then push to staging.
    6. Delete the staging/pr-{num} branch.
    • This approach allows us to get both the merged status for the PR and the rewritten commits in staging.
    • The workflow needs a concurrency group to prevent race conditions.
    • I also thought about the alternative of just force-pushing the rewritten commits to the PR's branch, but that will trigger CI to run on unchanged contents and also might dismiss any reviews, unnecessarily alert the contributor and might trigger other workflows.
    • we need to check if this actually works, i.e. if changing the target of a PR also re-triggers CI, then this isn't better than force-pushing.
    • we also need to check if this works when people uncheck the "allow maintainer edits" box on PRs.
    • Now all commits accumulate in staging, when the time of a sync with the kernel in the cycle comes, we will create/move the staging-frozen branch to the current staging branch. Merges to staging continue as normal, but the staging-frozen branch is only used manually and not updated with new merges.
    • from the staging-frozen branch, a maintainer creates the patch series for the kernel to synchronize pin-init.
    • any feedback on the PR such as typos, fixes etc as well as RB's are collected and the staging-frozen branch is rebased. it is then used to send a new version until all feedback is addressed.
    • then the series is picked into the pin-init-next branch in the RfL/linux repo and the PR is made to Miguel a bit later.
    • after Miguel's PR has been merged by Linus, we fast-forward main to staging-frozen, rebase staging on top of main and the process can repeat
