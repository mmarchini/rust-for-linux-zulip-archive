Thanks for doing this! I've tried this myself a while ago, but ENOUSER as well ;-)

I think for an RFC purpose, sending out the patch is fine, but be aware that it may be rejected to merge until there is a user. You may also want to reach out to the maintainer (Liam R. Howlett) to see he would like to see an RFC.

I took a quick looks, a few finding below:

  • It's OK as an initial implementation that some features are missing compared to the C version, but you may want to provide a list of what is missing. For example the C version of maple tree supports external locking, but your Rust version only supports using internal locking. Usually a kernel data structure would support various locking scenarios, so missing these support is worth mentioning.
  • mas_find() requires holding either write_lock or rcu, but in Iter::next() I don't see the corresponding lock holding.
  • This part is correct, but I would like rewording on the rest, because for example, RCU can also make the pointer valid to deref if a) rcu_read_lock() is held and b) updater side uses a synchronize_rcu().
