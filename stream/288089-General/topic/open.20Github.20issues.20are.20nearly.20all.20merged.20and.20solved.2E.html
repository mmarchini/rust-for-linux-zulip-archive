Valentin Obst (Jan 15 2024 at 23:57):

There are three things that I noticed while reading through the kernel crate but did not include in my PR since I think they require some further discussion with developers and maintainers:

  • The inner field of UniqueArc is private, but it is mentioned in the Invariants section of its docs, which is public. Similar issues might exist in other places. (The invariants section is here)
  • Oftentimes trait/struct/enum name::function/method name syntax is used to refer to functions/methods of the type that is being documented. This might be unnecessarily verbose. (Example: ForeignOwnable).
  • Oftentimes doclinks link back to the item that is being documented (Example: Arc, ARef, ...). This might be a bit redundant. The standard library has a paragraph in their coding/contributing guidelines with advises against that.

All of those things are a bit opinionated and should probably be discussed here before starting to work on it. Hope this helps you :)

Wedson Almeida Filho (Jan 26 2024 at 14:45):

Here's one area that we need improvement: making owned C strings "foreign ownable". There are two cases where the C side has a single pointer and we would like to store a CString there; the problem is that the current implementation uses a Vec, which has 3 words in it: pointer, length, and capacity.

I have a patch that changes it to use a Box<[u8]> instead, and I implemented ForeignOwnable by storing just a pointer on the C side. Then to reconstruct the CString, I calculate the length by searching for the null terminator (which is a type invariant). This works ok, but makes borrowing or converting back to CString an O(n) operation. Ideally, we'd like it to be O(1).

Miguel suggested that we use ThinBox instead, but it needs some functions (e.g., from_raw/into_raw). The upside is that we could get O(1) behaviour storing just a pointer on the C side.

Another challange we have with this is having to pre-initialise the string to avoid unsafe blocks. Here's the actual scenario: we need to allocate the buffer then populate it from block storage (different parts may come from different blocks, so the source string is not all available in one shot), then actually convert to CString (e.g., ensure that the string is null-terminated and doesn't contain null elsewhere). If we use [MaybeUninit<u8>], we need an unsafe block to convert it to [u8], which we want to avoid. Otherwise, we need to fill the slice with some value to ensure it's initialised (this is what we do), but it's wasteful because we are soon going to overwrite it with something else.

Anyway, is this something that may interest you?
