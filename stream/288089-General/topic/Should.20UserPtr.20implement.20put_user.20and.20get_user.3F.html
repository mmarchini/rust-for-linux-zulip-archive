I forgot that, In order to make the variable size known at compile time through a FFI, I can't have void* as a parameter. I need to implement a helper for every variable type, ex:

int rust_helper_put_user_u32(u32 x, u32 __user *to)
{
    return __put_user(x, to);
}
int rust_helper_put_user_u64(u64 x, u64 __user *to)
{
    return __put_user(x, to);
}

Another problem is with __get_user(x, ptr). Because it's a macro, when you pass x to it, it can actually reference and mutate it. So, when implementing a helper function, it should look like this:

int rust_helper_get_user_u32(u32 *x, u32 __user *from)
{
    u32 val;
    int res = __get_user(val, from);
    *x = val;
    return res;
}

Which is more overhead.

Boqun Feng said:

in x86 case it's the rep movsb instruction

I ran gdb on the kernel, and when tracing through _copy_to_user, I found out that my virtual machine wasn't running with FSRM support (Fast Short Rep Mov), so rather than using rep movsb, it used an assembly function called rep_movs_alternative which copies the 4 bytes of my s32 just like a classic for-loop. Yet it was faster than the much simpler put_user that was basically movl :sweat_smile:

I think the benchmarks are deceiving since we are trying to measure a difference between a very small amount of intstructions on a GHz CPU.

In theory, put_user should be faster on CPUs that don't support FSRM.
