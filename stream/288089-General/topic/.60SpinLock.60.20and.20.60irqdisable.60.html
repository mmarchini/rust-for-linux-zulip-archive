Topic: `SpinLock` and `irqdisable`

Joshua Liebow-Feeser (Apr 01 2025 at 22:11):

Does the SpinLock type ensure that locks must not be interleaved? For example, is the following permitted to compile?

  • Acquire lock A using lock_irqdisable
  • Acquire lock B using lock_irqdisable
  • Release lock A
  • Release lock B

If I'm not mistaken, this would erroneously result in irqs being unmasked when A is released (since that was the previous state when A was acquired) and possibly spuriously being masked when B is released (though that one I'm less sure about). I believe that, in C, you're required to promise not to acquire/release locks in a pattern like this. Is it prevented in Rust, or is it also just on the programmer to get it right?

Benno Lossin (Apr 01 2025 at 22:14):

yes and no:

  1. if you just call lock on both and then use the guards and let them be dropped automatically by the end of the scope, you're fine.
  2. you can manually call drop on the second guard which will unlock the second lock before the first one
  3. for catching these kinds of mistakes, we have klint, which we aren't using in-tree yet, but will in the near future

Boqun Feng (Apr 01 2025 at 22:40):

Note that your link is an old version of the spinlock API. The current upstream API doesn't have the lock_irqdisable function. And we are proposing the new interrupt disabling API to make it work when irq disabling locks are interleaved.
