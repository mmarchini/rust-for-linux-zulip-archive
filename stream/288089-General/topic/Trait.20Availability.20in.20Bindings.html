Topic: Trait Availability in Bindings

Matthew Maurer (Dec 12 2025 at 18:21):

I'm polishing up some patches to go up, one of which is derive macro for AsBytes and FromBytes. When I went to enable this in the bindgen output in places where it was appropriate, I found that our current dependency graph doesn't support this. We currently have kernel depends-on bindings, where bindings is a separate crate. (It also doesn't have a dep edge on macros right now, but that could be fixed.)

We cannot have dependency cycles, so this means that any trait declared in the kernel is currently unavailable to be derived for any type generated by bindgen in the main bindings.

The two main solutions to this I can see are:

Use a traits crate for types we want available in kernel deps

  • kernel depends-on bindings
  • kernel depends-on traits
  • bindings depends-on traits
  • bindings depends-on macros

This keeps the build roughly the same, but requires us to move traits to a separate crate to make them available to bindings.

Include the source of bindings.rs into a module built as part of kernel
This removes the bindings crate, and by virtue of being part of kernel, can access macros and any crates declared inside.

The main disadvantage I can see to this approach is that it weakens the split between the raw bindings and the rest of the kernel, since you can now end up with e.g. a trait impl in the kernel abstractions directly on a sys type.

We also have some special handling of the bindings.o build that might need to be fiddled with.

Does anyone have opinions on either of these, or another solution?
