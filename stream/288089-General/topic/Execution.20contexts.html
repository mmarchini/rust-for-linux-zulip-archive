Asahi Lina (Sep 27 2022 at 09:37):

I really like the idea of tracking execution context, but I think it should be some kind of Rust language feature that makes it implicit (or more ergonomic to use). Carrying around a cx everywhere might be okay for simple drivers, but having to throw it into every single function call of a complex driver is really a huge usability problem... and even if you can elide it in some code paths that don't call back into kernel core functions, the moment you add one you have to add the argument to the entire call chain to that point.

Linus is against runtime tracking, but not implicit compile-time tracking! I think this should be a Rust compile-time feature request ^^. I'd like to see something like having "sleepable" be the default context, and then functions that promise to be callable from Atomic/RawAtomic should be marked as such via an attribute or something like that (and then can only call functions that also have those attributes). You still need to mark call chains up (no real way around that) and you still need some sort of mechanism to convert the current context when you actually need to make a change, but at least this way you can avoid adding the argument to all function calls...

For reference, the GPU driver I'm working on currently has 304 instances of fn, and I actually don't think there's a single atomic-context call anywhere, since GPU drivers can pretty much always sleep. I really don't want to have to add context arguments to everything... the problem with this argument approach is that the default in the kernel for most drivers is Sleepable (least restrictive), while the only functions that can be called in any context are those that are RawAtomic (most restrictive), and those are the only ones which could elide the context argument. So you end up with the common case being "carry a context everywhere", and I think that's really bad for usability...

Also, how would this work for implicit calls, like Drop? I have Drop implementations that call into the kernel, but I can't pass a context into those... If it's an implicit mechanism it could be made to work, but then it would have to be more deeply built into the type system (propagate upwards through wrapper types somehow).

// assume_sleepable() is `unsafe`

mod token {
    use core::marker::PhantomData;
    // our invariant token
    pub struct Token<'a>(PhantomData<fn(&'a ()) -> &'a ()>);

    impl<'a> Token<'a> {
        // use the token in the closure and return the same token again + some retvalue
        pub fn use_token<'b, R>(&'b mut self, f: impl FnOnce(Token<'b>) -> RetToken<'b, R>) -> R
        where
            'a: 'b,
        {
            f(Token(PhantomData)).0
        }

        pub unsafe fn conjure() -> Self {
            Self(PhantomData)
        }

        pub fn sleep(&self, millis: u64) {
            std::thread::sleep(std::time::Duration::from_millis(millis));
        }
    }

    pub struct RetToken<'a, R>(pub R, pub Token<'a>);
}

use token::*;

macro_rules! ctx {
    ($tok:ident, $f:ident($($arg:expr),*$(,)?)) => {
        $tok.use_token(move |tok| $f(tok, $($arg),*))
    }
}

fn foo<'a>(tok: Token<'a>, typ: i32) -> RetToken<'a, i32> {
    tok.sleep(10);
    RetToken(42 + typ, tok)
}

fn bar<'a>(mut tok: Token<'a>) -> RetToken<'a, i32> {
    tok.sleep(100);
    RetToken(ctx!(tok, foo(42)), tok)
}

fn main() {
    let mut tok = unsafe { Token::conjure() };
    println!("{}", ctx!(tok, foo(0)));
    println!("{}", ctx!(tok, bar()));
}

the ctx! macro is just for demonstration, it should be done by the #[context(Sleepable)] proc-macro by recursing into every function call of the function and add the tok parameter. This seems a little bit too complex atm, especially because we do not have syn...

The reborrow is needed if you want to preserve some previous context. Your solution does't address the case where the context changes momentarily, then goes back to the previous one. For example, if you're in sleepable mode and acquire a spinlock, you transition to atomic context; once you release the lock, you go back to sleepable.

Now, if you were already in atomic context and then acquired a spinlock, it would remain in atomic context even after release the spinlock.

Things get more complicated if you have several transitions within the same scope (e.g., acquiring a mutexes, spin locks, raw spinlocks) and releasing in unconventional order (i.e., not in reverse order).

Demi Obenour (Oct 03 2022 at 16:16):

Not unlocking can also be prevented using (declarative) macros:

macro_rules! guard {
    ($a: ident) => {
        let mut $a: Guard = unsafe { Guard::new() };
        let $a: Pin<&mut Guard> = unsafe { Pin::new(&mut $a) };
    }
}

This prevents forgetting $a because the identifier has been shadowed by a mutable reference to $a, so there is no way to refer to it.
