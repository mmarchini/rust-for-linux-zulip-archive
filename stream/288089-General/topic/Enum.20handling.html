Miguel Ojeda (Sep 23 2023 at 00:23):

I wonder if we should just get bindgen to generate this.

There are rustified-enums enum types already (exhaustive and otherwise), so we could simply get bindgen to generate the conversion methods for us (the safe infallible version that panics, the safe fallible version that tries, and the unsafe one that assumes).

Not sure how we would specify the variant names that we want, but if we can do that, it would be best, since we wouldn't need the lint as what we want would be expressed already in code.

Benno Lossin (Sep 23 2023 at 08:15):

What I am a bit skeptical about is if we always want to set the representation of the enum values to be exactly that of the C side. So if I just take your example from the issue:

enum foo {
  one = 1,
  two = 2,
  three = 3
};

Do we really want the Rust enum to look like this?

#[repr(u32)]
#[derive(Debug, Copy, Clone, Hash, PartialEq, Eq)]
pub enum foo {
    one = 1,
    two = 2,
    three = 3,
}

Because in this case, u32 is not necessary. Also is it really necessary for the values to also have the same representation? What about

#[derive(Debug, Copy, Clone, Hash, PartialEq, Eq)]
pub enum foo {
    one,
    two,
    three,
}

Benno Lossin (Sep 23 2023 at 08:19):

Miguel Ojeda (Apr 13 2026 at 15:14):

I track the progress in our bindgen metalist at https://github.com/Rust-for-Linux/linux/issues/353.

The original unmerged PR is at https://github.com/rust-lang/rust-bindgen/pull/2908 (Cc @John Baublitz @Trevor Gross @Christian Poveda @Emilio Cobos Álvarez).

Then, the RFC at https://github.com/rust-lang/rust-bindgen/issues/3050 from Christian I think came out from a discussion we had in the PR above, which (from a quick look to get this back into my cache) allows to render several representations, which may be more flexible.

On the other hand, the idea with the "new representation" in bindgen that I made was about having one single way that would cover all (or at least most common) use cases (and without risk of UB!) and that could perhaps eventually be made the default in a future bindgen, possibly even deprecating at some point some of the other representations since the new one would be more flexible.

I don't think anyone looked at it recently, but I may be wrong. If you have cycles to help moving that forward, then I would suggest testing the existing PR, and considering whether that would work, and comparing it with the new representation idea as a user. At least that is what I would eventually do, i.e. my idea was to perhaps implement the original suggestion to compare.
