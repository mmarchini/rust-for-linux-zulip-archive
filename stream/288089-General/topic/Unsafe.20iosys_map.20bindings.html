Lyude (Aug 11 2024 at 19:48):

Hi! In my work on the rust vkms driver, I came across another thing it looks like I would need to add rust support for in the kernel: iosys_map bindings. I'm not entirely sure what these would look like yet, but my assumption is because of how general this interface is that it might be a good idea to make it !Send + !Sync, and expose IO operations as unsafe methods so that users implement their own safe abstractions around it to fit their needs. Does this seem like a good plan?

Lyude (Aug 11 2024 at 20:33):

Benno Lossin said:

(disclamer, I don't know what iosys_map is and have just read the doc overview in iosys-map.h)

Do you mind sharing what you need to use iosys_map for? What makes you think that we cannot provide safe abstractions?

The main reason I want to use it is so that we can handle return values from DRM functions like drm_gem_fb_vmap() (drivers/gpu/drm/drm_gem_framebuffer_helper.c) for mapping framebuffers that could live in either system memory or IO memory at runtime.

Also - I actually guess maybe we could expose them as safe? I had assumed this was something that would be an unsafe operation because a generic type for a memory mapping isn't really going to know what access rules ithe memory is governed by. However, it seems like the kernel::io::Io type we currently have in the kernel actually exposes stuff like readw/readl/etc. as safe methods so I guess perhaps safe would be fine here?

Lyude (Aug 13 2024 at 21:02):

Going to start working on this today - I'm thinking of trying to handle this similarly to io.rs, except we use a struct like this:

pub struct IosysMap<const SIZE: usize = 0> {
  map: bindings::iosys_map,
  maxsize: usize,
}

And probably require that any conversions from a raw iosys_map require that the user specify the size to store in maxsize
