Jesung Yang (Jun 16 2025 at 21:23):

Hi all,

I'd like to take on the following task from the nova-core TODO list, if it's open to external contributors:

FromPrimitive API

Sometimes the need arises to convert a number to a value of an enum or a
structure.

A good example from nova-core would be the Chipset enum type, which defines
the value AD102. When probing the GPU the value 0x192 can be read from a
certain register indication the chipset AD102. Hence, the enum value AD102
should be derived from the number 0x192. Currently, nova-core uses a custom
implementation (Chipset::from_u32 for this.

Instead, it would be desirable to have something like the FromPrimitive
trait [1] from the num crate.

Having this generalization also helps with implementing a generic macro that
automatically generates the corresponding mappings between a value and a number.

As far as I can tell from the mailing list, no one is currently working on it.

Here's what I plan to implement:

  1. A FromPrimitive trait
  • excluding float support for now, since it doesn't seem necessary and can be added later if needed
  1. A corresponding derive macro
  • along with lightweight helpers to parse enums, since it looks like such functionality isn't currently available

Some design questions I'd like to get feedback on:

  • Do we also want a ToPrimitive trait?
  • Should the derive macro support single-field tuple structs in addition to enums?

Open to any thoughts or suggestions. Thanks in advance!

By the way, I think it's reasonable for all methods of the FromPrimitive trait to return Option<Self>, so that the trait remains generic and allows users to handle errors in a way that suits their use cases.

However, adopting this design would require a bit of extra cleanup in nova-core. For example, in this code:

     pub(crate) fn architecture(self) -> Result<Architecture> {
-        Architecture::try_from(
+        Architecture::from_u8(
             self.architecture_0() | (self.architecture_1() << Self::ARCHITECTURE_0.len()),
-        )
+        ).ok_or(ENODEV)
     }

If returning Option<Self> sounds reasonable to you as well, and if it feels like an obstacle, I'd be happy to help with the necessary adjustments in nova-core once this patch successfully lands. @Alexandre Courbot

Sounds good to me. I think we need a bit more discussion about the API though.

#[derive(TryFrom, Into)]
#[try_from(u8, /* and more if needed */)]
#[into(u8, /* and more if needed */)]
#[repr(u8)] // Defaults to this if the helper attributes are omitted.
// Falls back to `isize` if nothing is provided.
enum Foo {
    A,
    B = 7,
}

Does this direction seem reasonable to you?

One thing I'm unsure about is the use of the name Into here. It would actually generate impl From<Foo> for u8, since From automatically provides the corresponding Into implementation. So the naming might be slightly misleading at first glance.
