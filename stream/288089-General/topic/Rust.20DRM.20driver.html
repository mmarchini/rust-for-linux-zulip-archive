Asahi Lina (Aug 13 2022 at 17:01):

@bjorn3 Thanks for the reply! Here's roughly how it works (at least for this GPU). GPU kernel objects are always allocated and initialized by the kernel, from regular (pinned) memory pages. They have a CPU VA and a GPU VA (the physical page addresses don't really matter after allocation/mapping). After an object is initialized and mapped to the GPU, its GPU VA is passed to the GPU in some way (usually via some existing object, and the root is the InitData object, which has its GPU VA sent via the firmware mailbox).

Once the object is shared, its usage can vary. Some objects are managed locally by the GPU (CPU does not touch them), some are written by the GPU and potentially read by the CPU, some are written by the CPU and read by the GPU, and some are write-write shared. For those, usually each side is in charge of some fields (e.g. ring buffer state objects where one side writes the rptr and the other the wptr). Sometimes the owner could change at different points in the object's lifetime.

There is no explicit lifecycle management in the interface, so the rules for who owns what at any given point and when it's safe to access or dealloc an object that the GPU was previously using are ad-hoc and part of the firmware ABI. For example, objects related to a render job can be reused or freed once the GPU signals completion of that job, and objects related to a command queue can be freed when the queue doesn't have any pending work, but an explicit context destroy command also has to be sent to the firmware (I think).

The mapping mode can also vary, on both GPU and CPU side (this is set by the kernel in the page tables). For objects that are internally used by the GPU, and sometimes for CPU-w GPU-r objects, they are usually mapped as non-coherent on the GPU side (in the latter case it would do manual cache management as required). For objects that are shared, they get mapped coherently.

The objects are mostly fixed-layout structures, but some can be arrays of structures or of pointers (usually ring buffers).

Usually, GPU objects would be managed by some kind of CPU-side object/manager (the CPU will never put CPU pointers in GPU objects or anything like that). For example, ring buffers would need a CPU manager type/impl that keeps shadow pointers, knows how to copy the data in/out of the buffer, etc. Also, it's generally useful to be able to associate objects with things like names and types at runtime, at least in debug builds (e.g. if the GPU crashes, we can try to map the fault address to what object it was trying to access). So I think there should be some kind of GPUObject abstraction that takes care of encapsulating the shared allocation, knowing its GPU VA as well as having a CPU reference, and probably keeping track of which side supposedly owns it at any given point. Then a type that manages one or more GPU objects would own references to those GPUObjects.

For some patterns, like where the CPU initializes something and never touches it after that, or the CPU initializes it and then only reads after that, that could probably be enforced at the API layer (e.g. the caller can only get a GPU VA for the object after giving up ownership of the internal data, and can only get immutable references or none at all after that). Object destruction to some extent relies on correct firmware interaction too, but at least I definitely want to enforce correctness when it comes to persistent object-to-object pointers (which is not all of them, e.g. an object reference sent out via a ring buffer is effectively gone, if not immediately, then soon as the GPU advances the rptr, but complex pointer trees of objects are common and those should never have dangling internal references).

bjorn3 (Aug 13 2022 at 17:08):

At a minimum I think you did need

struct GpuBox<T>(usize, PhantomData<T>); // Box<T> equivalent
struct GpuRef<'a, T>(usize, PhantomData<&'a T>); // &'a T equivalent
struct GpuRefMut<'a, T>(usize, PhantomData<&'a mut T>); // &'a mut T equivalent

struct GpuVec<T>(usize, PhantomData<T>); // Box<[T]> equivalent
struct GpuSlice<'a, T>(usize, usize, PhantomData<&'a T>); // &'a [T] equivalent
struct GpuSliceMut<'a, T>(usize, usize, PhantomData<&'a mut T>); // &'a mut [T] equivalent

As for how to handle migrating between cpu and gpu owned. For converting from cpu to gpu owned, you can probably have a safe method converting Box<T> to GpuBox<T> and Box<[T]> to GpuBox<[T]> which maps the memory to the gpu and returns the gpu address. As for going the other way, I think the core interface will have to be unsafe (as you have to prove that the gpu isn't accessing it), but it should be possible to have safe methods for specific cases, like consuming already processed ring buffer elements.

Asahi Lina (Aug 13 2022 at 17:16):

There's also the question of what exactly is unsafe. In principle, for safety, we'd want to consider anything mapped to the GPU as untrusted (and since the objects come from DMA pools and are not mapped at the object level, that means every GPU object, for its entire lifetime). Is there some way of telling the compiler "this data structure may be written to by other threads at any time and you should not make assumptions/optimizations that would violate safety guarantees if it changes out from under you"? If there's some way of doing that (without a big performance cost, and it only has to work for simple primitive types/structs), that would effectively make the "take ownership back from the GPU" operation safe, since the CPU isn't making assumptions that it worked. Worst case you crash the GPU, not the driver.

bjorn3 (Aug 13 2022 at 17:26):

Yeah, volatile is a big hammer. What you want is probably the unordered memory ordering for atomics of LLVM, which rustc doesn't expose. As I understand it, it makes data races not UB, but doesn't imply any synchronization. It was introduced for Java. I believe the reason rustc doesn't have it is because the exact operational semantics of unordered are not clear. Also it is LLVM only AFAIK.

bjorn3 (Aug 13 2022 at 17:27):

https://llvm.org/docs/LangRef.html#atomic-memory-ordering-constraints

unordered
The set of values that can be read is governed by the happens-before partial order. A value cannot be read unless some operation wrote it. This is intended to provide a guarantee strong enough to model Java’s non-volatile shared variables. This ordering cannot be specified for read-modify-write operations; it is not strong enough to make them atomic in any interesting way.

Asahi Lina (Aug 19 2022 at 13:18):

Here's the GPUObject design I ended up with! It's just a dummy userspace implementation right now, but it should work in the kernel (just replacing the Box<> with a real GPU shared memory allocator, and using the real GPUVAs).

https://github.com/asahilina/gpu-rust-playground/blob/main/playground/src/main.rs

I had a lot of people help me out design it (and I still don't quite understand some of the really deep details), but I think what we ended up with ends up being reasonably ergonomic to work with and quite safe. It encodes the concept of GPU objects owning other objects (so they all get dropped together) and being able to contain references to those objects, with compiler-enforced lifetimes. There are also weak pointers for things like circular references or other cases where lifetimes don't really apply (I think it's okay to not try to encode absolutely everything in the type system, since the goal of this is to make it easier to write a correct GPU driver, and it's not possible to enforce everything at compile time anyway since the correct semantics depend on the GPU firmware ABI anyway). I also ended up with a cute macro to get GPU pointer references to inner fields of an object, with pointer/field type safety (based on the kernel's offset_of!() but with some clever type inference so it works directly on a GPUObject value).

There's no explicit concept of object GPU vs CPU ownership state right now, but what I do have is that GPUObjects can only be accessed via callbacks right now after construction, and they take an immutable ref, so the idea is that any CPU-mutable fields should be atomics (which is what we'd want for shared mutable data anyway). The GPUArray type doesn't enforce this right now, but I might add an immutable variant just to keep things more kosher.
