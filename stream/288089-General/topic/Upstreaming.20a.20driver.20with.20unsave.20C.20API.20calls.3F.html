Topic: Upstreaming a driver with unsave C API calls?

Remo Senekowitsch (Sep 20 2024 at 08:14):

I'm a master student and it was just decided that for my semester project I will be working on rewriting a C driver in Rust. The driver works but hasn't been upstreamed yet, so the "no duplicate drivers" rule doesn't prevent my Rust rewrite from being upstreamed.

It's a well-known problem that upstreaming Rust drivers is a bit of a chicken-and-egg problem: APIs need users to be upstreamed and drivers need their APIs to be upstreamed. This means several unrelated tasks need to be completed and upstreamed at once.

An idea my advisor and I had was to upstream a version of the Rust driver that simply calls unsafe, generated bindings to the C APIs. This would be much easier because safe API wrappers in Rust don't have to be upstreamed at the same time. After that, safe API wrappers can be upstreamed more easily because a potential users already exists in-tree.

I'm assuming this approach has already been considered. What was the conclusion on that? Is there a rule against using unsafe in Rust drivers? Or is it a bad idea for other reasons?

Andreas Hindborg (Sep 20 2024 at 09:40):

I would not advice having direct unsafe calls to C code in your driver. I think there are two viable approoaches.

1) Submit your driver as an RFC. Base the patches on a git tree that includes abstractions for the subsystems your driver interfaces. Then start upstreaming the subsystem abstractions, pointing the RFC for your driver as the user. This is the approach taken by rust binder.

2) Submit your driver as a stub to be included in tree. Then submit your subsystem abstractions little by little, pointing to your in-tree stub driver as the user. When subsystem abstractions are merged, grow your stub driver to use them. This is the approach taken by nova.

Andreas Hindborg (Sep 20 2024 at 09:40):

Rust null block is also taking approach 2.
