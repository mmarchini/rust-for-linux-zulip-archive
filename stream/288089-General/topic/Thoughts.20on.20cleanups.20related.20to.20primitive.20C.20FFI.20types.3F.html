Jesung Yang (May 26 2025 at 13:25):

I'm wondering if the following changes would be worth submitting as a patch:

  1. core::ffi::c_void in rust/kernel/kunit.rs
use core::{ffi::c_void, fmt};
// ...
#[doc(hidden)]
pub fn err(args: fmt::Arguments<'_>) {
    // SAFETY: The format string is null-terminated and the `%pA` specifier matches the argument we
    // are passing.
    #[cfg(CONFIG_PRINTK)]
    unsafe {
        bindings::_printk(
            c"\x013%pA".as_ptr() as _,
            &args as *const _ as *const c_void,
        );
    }
}
// ...
#[doc(hidden)]
pub fn info(args: fmt::Arguments<'_>) {
    // SAFETY: The format string is null-terminated and the `%pA` specifier matches the argument we
    // are passing.
    #[cfg(CONFIG_PRINTK)]
    unsafe {
        bindings::_printk(
            c"\x016%pA".as_ptr() as _,
            &args as *const _ as *const c_void,
        );
    }
}

Ideally, crate::ffi::c_void would be used, but it's just a simple wrapper around core::ffi::c_void. They're equivalent for now, so it doesn't make a big difference, but this might help with consistency and would account for any future changes (well, it seems to be pretty unlikely that c_void would ever become anything other than core::ffi::c_void).

I believe use core::ffi::c_void can be simply removed, because commit 3d5bef5d47c371e1704 ("rust: add C FFI types to the prelude") made all the FFI types available through the prelude.

  1. doctest in rust/kernel/init.rs
//! ```rust,ignore
//! # #![allow(unreachable_pub, clippy::disallowed_names)]
//! use kernel::{prelude::*, types::Opaque};
//! use core::{ptr::addr_of_mut, marker::PhantomPinned, pin::Pin};
//! # mod bindings {
//! #     #![allow(non_camel_case_types)]
//! #     pub struct foo;
//! #     pub unsafe fn init_foo(_ptr: *mut foo) {}
//! #     pub unsafe fn destroy_foo(_ptr: *mut foo) {}
//! #     pub unsafe fn enable_foo(_ptr: *mut foo, _flags: u32) -> i32 { 0 }
//! # }
//! # // `Error::from_errno` is `pub(crate)` in the `kernel` crate, thus provide a workaround.
//! # trait FromErrno {
//! #     fn from_errno(errno: core::ffi::c_int) -> Error {
//! #         // Dummy error that can be constructed outside the `kernel` crate.
//! #         Error::from(core::fmt::Error)
//! #     }

Similarly, in this doctest, it might make sense to use c_int instead of core::ffi::c_int in the signature fn from_errno(errno: core::ffi::c_int) -> Error. That said, this doctest is marked with ignore attribute, so the practical impact is minimal.
