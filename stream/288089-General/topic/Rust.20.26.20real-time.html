During building RROS, one problem we met was that Rust may alloc memory implicitly. This could cause latency in the real-time code path. It needs careful implementation to avoid such situations. Another problem is that sometimes it's hard to keep pure safe code and low latency at the same time when manipulating a list.

Hongyu Li (Mar 06 2024 at 03:45):

Do you mean concerns like .pushing to a vector automatically resizing? Or just having a tough time of tracking which functions or types may allocate ever

It's kinda similar to the first case. For us, this occurs when we're adding a node into an rbtree. In Linux, adding the node to the tree doesn't involve any memory allocation since the rb_node is already allocated within the structure. But in Rust, when we try to insert using try_insert, it calls try_allocate_node, which then attempts to allocate memory.

We've also encountered the latter scenario. We tackled it by introducing a flag that indicates whether we're operating in the real-time code path. Then, within the memory allocation function, we check the state of this flag.
