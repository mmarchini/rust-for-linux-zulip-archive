Topic: `bit` function

Wedson Almeida Filho (Old Google account) (Jul 15 2021 at 15:09):

I'm trying to write a type-safe function that given a bit index, generates a mask with all zeroes except the index, which should be 1.

Here's what I have:

fn bit<T: core::convert::From<u8> + core::ops::Shl<U, Output = T>, U>(offset: U) -> T {
    T::from(1u8) << offset
}

It works well in most cases, but in some cases the compiler is not able to infer T, for example:

fn test2(x: u32) -> bool {
    x & bit(2) != 0
}

This fails with:

error[E0284]: type annotations needed: cannot satisfy `<u32 as BitAnd<_>>::Output == _`
  --> src/lib.rs:12:7
   |
12 |     x & bit(2) != 0
   |       ^ cannot satisfy `<u32 as BitAnd<_>>::Output == _`

I have the code here: https://play.rust-lang.org/?version=stable&mode=debug&edition=2018&gist=65f4c87be0974ee1b04b7d11e9ef54ca

If I use a macro instead, it works in both cases: https://play.rust-lang.org/?version=stable&mode=debug&edition=2018&gist=5d3a4a037e8dfec0084cb050bd93174e

Any ideas on how to make this work?

Wedson Almeida Filho (Old Google account) (Jul 15 2021 at 16:33):

Yes, we could do that. But then people would have to memorise those function names, it doesn't seem like an improvement.

For some context: the kernel has a BIT macro and I'm converting a driver to rust. I'd like the bit-banging parts to look as similar as possible to C to reduce distraction.

For example, there's a line in C that reads x &= ~BIT(y), in Rust with a macro it would be x &= !BIT!(y). I'm positive people are not going to be happy about it. I think x &= !bit(y) would be slightly more palatable.

Also, it looks like Rust defaults to i32 when an unsuffixed integer type is under-constrained so it likely to lead to unexpected behaviour.
