Jubilee (Oct 12 2022 at 07:11):

Some ground details insofar as I understand them:

My knowledge of aarch64's floating point capabilities is that the floating point registers are mandatory on aarch64 (they are exempted only for very specialized architectures that will NOT be made generally available as APIs). They must be initialized and made ready at boot. There is only one FPU on aarch64. It is the Neon unit. The aarch64 Neon registers have a few very subtly different characteristics than Armv7 Neon registers, due to some hardware enhancements, the absence of the Armv7's VFP, and different register mappings. However, they conceptually respond to the ~same instruction set that Armv7's Neon support did.

I know that generally the kernel doesn't get to do a lot with floating point because it tries to avoid saving the FPU register context. Smart move, tbh, when the FP registers are also packed SIMD registers are also variable length vector registers.

rustc has support for carefully toggling features, including aarch64 ones, on and off, including Neon, but this can ALSO get... complicated.

Asahi Lina (Oct 12 2022 at 17:42):

I'm actually not sure how much float support I'll need yet! The GPU initialization data has a whole bunch of float constants, but we can cast those to u32 at compile time. Right now that's all I do, no real float support required. But some of those constants have to come from the device tree eventually, which requires adding support for float literals to dtc. And then the remaining question is... do I also need to do some basic math on these values? I haven't looked into exactly how they vary and how they map to what needs to be in the device tree. It's possible I can just pre-compute everything in its final form and won't need any float calculations at all. But if I do need calculations, it's going to be some very simple add/sub/mul/div stuff, only computed at driver initialization time, and using well-behaved values... so I think there is room for doing the math with software emulation (either the existing full-fat soft-float implementation in the kernel, or an even simpler, bespoke one in Rust). This would avoid using the actual FPU entirely, which some people might find more palatable than diving into getting the compiler and kernel_neon_begin/end() to play nicely. But of course, I won't complain if I get real float support! ^^
