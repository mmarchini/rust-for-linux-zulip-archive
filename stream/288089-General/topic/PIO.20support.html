In response to the PIO topic you brought up during today's meeting.

I think that on the Rust side we shouldn't use iowrite*/ioread*. The reason is that those functions always do the if-else dance in IO_COND(), which in Rust isn't necessary, since we can do this check once on creation of the Io type.

We can introduce a trait IoBackend with generic read / write functions, e.g. read8 or write32 and then implement an MMIO and a PIO type implementing those functions with the corresponding MMIO and PIO accessors. The generic Io type can then instantiate either one or the other on creation avoiding the duplicate checks we'd get with iowrite*/ioread*.

If you look at pci_iomap(), it either returns the result of ioremap() which is a real memory mapping or the result of ioport_map(), which indeed is some kind of cookie. But in this case cookie means that the port is embedded in a "fake" address.

When you call e.g. ioread32() [1] the "fake" address is converted to the port again [2]. We should be able to do the same thing once on creation of Io, rather than on every read / write.

[1] https://elixir.bootlin.com/linux/v6.13.1/source/lib/iomap.c#L225

Danilo Krummrich (Feb 20 2025 at 14:16):

@Rob Herring

I don't really see how this is related to PCI? The relaxed semantics is about instruction ordering at compile time and CPU ordering at runtime. As you mentioned for ARM / ARM64 both are relevant, for x86 the relaxed semantics only omits the compiler barrier as in the non-relaxed version.

The PCI bus itself has strictly ordered reads (i.e., a read cannot pass a previous write), but may have unordered writes, e.g. posted writes (buffering in root complexes or switches) or write combining. But this is unrelated to the ordering of the CPU or the compiler.

So, on x86 it's probably not that interesting to be able to omit the compiler barrier. But on ARM / ARM64, PCI devices are typically mapped through a PCI root complex, i.e. PCI BAR accesses happens through accesses to the root complex and there we might care about unnecessary CPU barriers, no?
