Demi Obenour (Feb 04 2023 at 02:54):

Are drivers written in Rust generally going to be secure against malicious devices? I ask because drivers written in C typically are not, and in Qubes OS there are several cases where the lack of a hardened driver is causing problems.

Asahi Lina (Feb 05 2023 at 12:56):

This is something that came up in a discussion with some of the DRM folks... Personally, I think they should be, because otherwise that just opens up a lot of attack surface.

In general my GPU driver tries not to trust the GPU firmware, though there is one big fundamental problem that makes it impractical to be provably secure against a malicious device with today's Rust support (C sort of makes it possible but not ergonomic). It's somewhat academic though, in that I think it's unlikely that the compiler will generate code that would introduce vulnerabilities, but it could...

Asahi Lina (Feb 06 2023 at 01:04):

The issue is less the atomicity itself and more that the compiler shouldn't expect data to not change out from under it. For example, let's say you have this kind of code:

1    let mut buf = [0; 4];
2    if count > 4 { return Err(EINVAL); }
3    gpu_shared_struct.foo = count;
4    // some other code
5    buf[..count].fill(1);

The compiler could elide any bounds checking in line 5, because the bound was already checked in line 2. But let's say line 4 causes register pressure, so the compiler decides to load back gpu_shared_struct.foo into count. Now the GPU could cause the CPU to overflow buf by changing that value.

In C we have READ_ONCE and WRITE_ONCE to avoid this. Volatile also basically avoids this but it's a bigger hammer since it also forbids reordering and other things...

Asahi Lina (Feb 06 2023 at 01:06):

What I really want is some kind of qualifier, ideally something that can be applied ergonomically to references, that tells the compiler to do exactly the reads and writes shown (reordering allowed but not adding more).

Alice Ryhl (Feb 06 2023 at 08:04):

Asahi Lina (Feb 06 2023 at 08:18):

Asahi Lina (Feb 06 2023 at 08:22):

The tricky thing is I don't necessarily want to make everything volatile... I do want the compiler to optimize writes however it wants, what I don't want is for it to read things back or expect them to be the way it left them!

Most of what I do with GPU memory is write (sometimes very large) structs, so it seems silly to use such a big hammer for that. For example, it's perfectly fine to reorder everything, split or merge writes, etc... I wouldn't want to lose performance by marking every field of multi-kilobyte structs as volatile. I already use actual atomics where ordering and access type strictly matters (things like ring buffer pointers).

Asahi Lina (Feb 06 2023 at 08:24):

So I guess what I really want is some kind of "read-guarded" value: reads should be volatile, but not writes (and writes should never cause reads to be synthesized)
