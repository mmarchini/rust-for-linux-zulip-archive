Miguel Ojeda (Jul 29 2024 at 09:33):

Welcome!

So I was wondering if there was a way to gauge this and if there were some things I need to know or that are good to know before I start working on issues

There are no prerequisites -- you can contribute to whatever you want to! What is important is that the patches you submit are well thought out :)

what kind of commitment starting one of them would entail

If it is for something that goes into the kernel, then if it gets accepted, it typically means people may ping you in the future about questions on it via email and/or to help maintaining what you submitted.

Like, for sure I need to know Rust and C, but would I need to know more about the linux kernel, OS, drivers or other systems stuff?

It depends in what you want to contribute to. Rust for Linux is a very wide project -- for instance, some things may require changes on the kernel, others on the Rust compiler or another tool, others may require changes in several projects, etc. (to the point that you could see RFL as a "metaproject" :)

So you don't necessarily even need to know about C, or Rust! For instance, if you were to fix an issue or implement a feature in a Rust tool that the kernel needs, it could be the case that C is not needed, and vice versa.

Of course, for kernel code, it helps to have a background on kernel development (at least for the time being, since most of the work is writing the abstractions).

In summary, if you already have an area you like, I would recommend finding something around that area that could help RFL, and start there.

(+ perhaps some resources like textbooks or papers I can go through to help me understand better what needs to be done)

For kernel-related things, you may find https://docs.kernel.org/process/kernel-docs.html useful. But, again, you can contribute to RFL without being a kernel developer.

I hope this helps!
