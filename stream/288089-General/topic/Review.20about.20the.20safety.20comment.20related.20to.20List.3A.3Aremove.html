I put forward my question in https://rust-lang.zulipchat.com/#narrow/channel/425075-rust-for-linux/topic/Review.20about.20the.20safety.20comment.20related.20to.20List.3A.3Aremove/with/567645786, and maybe it's better to post my question in Rust-for-linux zulip, so I copy my question as follows, but the above link already has some discussion.

Several weeks ago, the Rust code in the Linux Kernel saw its first CVE vulnerability. After reading the patch for this CVE, I think the safety comment for List::remove could be modified to prevent potential misuse.

A post on Reddit introduces the related code for this CVE. This CVE involved a possible data race condition when traversing and simultaneously removing elements from the same list. The main reason is that the buggy function drops the list's lock while the list is being traversed:

impl Node {
    // [Other bits omitted]
    pub(crate) fn release(&self) {
        let mut guard = self.owner.inner.lock();
        // [omitted]

        let death_list = core::mem::take(&mut self.inner.access_mut(&mut guard).death_list);
        drop(guard);
        for death in death_list {
            death.into_arc().set_dead();
        }
    }
}

Thus, the removal and traversal operations can happen simultaneously.

The CVE patch fixes the data race by acquiring the lock before every pop operation:

pub(crate) fn release(&self) {
    let mut guard = self.owner.inner.lock();
    // [Other bits omitted]

    while let Some(death) = self.inner.access_mut(&mut guard).death_list.pop_front() {
        drop(guard);
        death.into_arc().set_dead();
        guard = self.owner.inner.lock();
    }
}

However, the remove operation itself satisfied its safety comment, and the maintainer provided the #SAFETY comment at the call site:

// The source code of List::remove
    /// Removes the provided item from this list and returns it.
    ///
    /// This returns `None` if the item is not in the list. (Note that by the safety requirements,
    /// this means that the item is not in any list.)
    ///
    /// When using this method, be careful with using `mem::take` on the same list as that may
    /// result in violating the safety requirements of this method.
    ///
    /// # Safety
    ///
    /// `item` must not be in a different linked list (with the same id).
    pub unsafe fn remove(&mut self, item: &T) -> Option<ListArc<T, ID>> {
    // omit the implementation
}

// The call site of List::remove
impl NodeDeath {
    // [Other bits omitted]
    /// Sets the cleared flag to `true`.
    ///
    /// It removes `self` from the node's death notification list if needed.
    ///
    /// Returns whether it needs to be queued.
    pub(crate) fn set_cleared(self: &DArc<Self>, abort: bool) -> bool {
        // [Removed some hopefully-not-relevant code]

        // Remove death notification from node.
        if needs_removal {
            let mut owner_inner = self.node.owner.inner.lock();
            let node_inner = self.node.inner.access_mut(&mut owner_inner);
            // SAFETY: A `NodeDeath` is never inserted into the death list of any node other than
            // its owner, so it is either in this death list or in no death list.
            unsafe { node_inner.death_list.remove(self) };
        }
        needs_queueing
    }
}

Currently, the safety comment for List::remove reminds users to be careful with using mem::take, but it seems the main reason for this CVE was that the lack of a lock caused the traversal and removal operations to happen simultaneously.

So I have the following questions:

  1. Why might mem::take result in violating the safety requirements of List::remove? It seems that mem::take simply takes the list from Self and stores it on the stack, and the item intended for removal is in the same list.

  2. I propose adding a safety requirement entry for List::remove: When performing the removal operation, the caller must hold the lock for this list. This will guarantee that no data race can occur.
