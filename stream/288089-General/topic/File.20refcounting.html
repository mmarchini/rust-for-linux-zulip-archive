I was thinking about how to justify the file descriptor stuff that I'm doing in my file patchset and I came up with the following comment:

Wraps the kernel's struct file.

Refcounting

Instances of this type are ref-counted according to the following rules. There are three kinds of refcounts:

  • Normal refcounts. These are created with fget/get_file and dropped with fput.
  • Fd refcounts. These correspond to an fd that a process has. Dropped when the fd is closed.
  • A light refcount. Created with fdget, dropped with fdput.

The two first kinds of refcounts are tracked using the f_count atomic in struct file. Both types of refcount correspond to one increment of f_count. Light refcounts are not tracked anywhere. The ARef<File> type corresponds to a normal refcount.

Fd refcounts are normally held by a struct files_struct for a process. Closing a file is done by removing the file from the struct files_struct with close_fd_get_file and then using filp_close to actually close the file. The filp_close method takes ownership of a refcount and drops it. Normally, filp_close is given the fd refcount being closed, but this isn't a requirement: The filp_close method can also accept a normal refcount instead to keep the fd refcount alive for longer.

Light refcounts are always associated with an fd refcount. The idea is that we will not close the file between the fdget and fdput calls, which means that the fd refcount is guaranteed to be active between those two calls. Note that this doesn't work if another thread could close the file in that period, so fdget only creates a light refcount if the struct files_struct is not shared with other threads. Otherwise, fdget creates a normal refcount.

Invariants

  • Instances of this type are refcounted with three types of refcounts: normal refcounts, fd refcounts, light refcounts.
  • The f_count field holds the total number of normal and fd refcounts.
  • Each light refcount is associated with an fd refcount and can only be created when the associated struct files_struct is not shared.
  • A light refcount must be dropped before returning to userspace.
  • An fd refcount must not be dropped if there are active light refcounts associated with it.

Thoughts?

Alice Ryhl (Jan 16 2024 at 11:50):

New proposed wording:

Wraps the kernel's struct file.

Refcounting

Instances of this type are reference-counted. The reference count is incremented by the fget/get_file functions and decremented by fput. The Rust type ARef<File> represents a pointer that owns a reference count on the file.

Whenever a process opens a file descriptor (fd), it stores a pointer to the file in its struct files_struct. This pointer owns a reference count to the file, ensuring the file isn't prematurely deleted while the file descriptor is open. In Rust terminology, the pointers in struct files_struct are ARef<File> pointers.

Light refcounts

Whenever a process has an fd to a file, it may use something called a "light refcount" as a performance optimization. Light refcounts are acquired by calling fdget and released with fdput. The idea behind light refcounts is that if the fd is not closed between the calls to fdget and fdput, then the refcount cannot hit zero during that time, as the struct files_struct holds a reference until the fd is closed. This means that it's safe to access the file even if fdget does not increment the refcount.

The requirement that the fd is not closed during a light refcount applies globally across all threads - not just on the thread using the light refcount. For this reason, light refcounts are only used when the struct files_struct is not shared with other threads, since this ensures that other unrelated threads cannot suddenly start using the fd and close it. Therefore, calling fdget on a shared struct files_struct creates a normal refcount instead of a light refcount.

Note that if the struct files_struct becomes shared during a light refcount, then the requirement that the fd is not closed before the fdput call also applies to the other threads that it was shared with. That is, making a struct files_struct shared does not "upgrade" existing light refcounts to full refcounts; it just prevents the creation of new light refcounts. This also means that checking whether the struct files_struct is shared cannot be used to determine whether an fd has active light refcounts, since it could have light refcounts that were created before it was shared.

Light reference counts must be released with fdput before the system call returns to userspace. This means that if you wait until the current system call returns to userspace, then all light refcounts that existed at the time have gone away.

Rust references

The reference type &File is similar to light refcounts:

  • &File references don't own a reference count. They can only exist as long as the reference count stays positive, and can only be created when there is some mechanism in place to ensure this.
  • The Rust borrow-checker normally ensures this by enforcing that the ARef<File> from which a &File is created outlives the &File.
  • Using the unsafe [File::from_ptr] means that it is up to the caller to ensure that the &File only exists while the reference count is positive.
  • You can think of fdget as using an fd to look up an ARef<File> in the struct files_struct and create an &File from it. The "fd cannot be closed" rule is like the Rust rule "the ARef<File> must outlive the &File".
  • Light refcounts can only be created from ARef<File> pointers stored in a struct files_struct, whereas &File references can be created from any ARef<File>.

Invariants

  • Instances of this type are refcounted using the f_count field.
  • If an fd with active light refcounts is closed, then it must be the case that the file refcount is positive until there are no more light refcounts created from the fd that got closed.
  • A light refcount must be dropped before returning to userspace.
