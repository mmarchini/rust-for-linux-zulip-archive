Topic: core::ops::Receiver

Asahi Lina (Feb 08 2023 at 16:22):

Is there precedent for using core::ops::Receiver to create ergonomic subclassable kernel types?

A pattern I keep running into over and over again while writing abstractions is that I have some kernel object O which needs to be subclassable by the user (from the C side, via struct composition). So I end up with O<T> containing the raw kernel object and a T. Then I add a Deref<O<T>> -> T impl, so given an O<T> you can call things on T. But if those methods take just the &self as the receiver, they can only interact with the user-implemented T type, not the actual O<T> kernel object. So I wind up with stuff like fn foo(this: &O<T>) when I need to give the methods access to the outer object, which is quite ugly (especially to call). What I really want is fn foo(self: &O<T>), which works if the type implements core::ops::Receiver.

Here's an example of not using Receiver: https://github.com/AsahiLinux/linux/blob/070e841af7bdb37ea554e61667360004cf44d06f/drivers/gpu/drm/asahi/queue/mod.rs#L164
And here's one using it (a trivial one, which I just implemented): https://github.com/AsahiLinux/linux/blob/070e841af7bdb37ea554e61667360004cf44d06f/drivers/gpu/drm/asahi/queue/mod.rs#L128

Is there some other pattern to solve this issue I don't know about? Should I just start switching things to use Receiver impls this way?

Boqun Feng (Feb 08 2023 at 16:39):

I think that Deref could only be used for smart pointers: https://doc.rust-lang.org/std/ops/trait.Deref.html, in other words, using it for subclass (or inheritance) is not idiomatic, at least that's what I've heard.

Boqun Feng (Feb 08 2023 at 16:40):

Implementing Deref for smart pointers makes accessing the data behind them convenient, which is why they implement Deref. On the other hand, the rules regarding Deref and DerefMut were designed specifically to accommodate smart pointers. Because of this, Deref should only be implemented for smart pointers to avoid confusion.

Boqun Feng (Feb 08 2023 at 16:50):

Just FYI: https://github.com/rust-unofficial/patterns/blob/main/anti_patterns/deref.md

Boqun Feng (Feb 08 2023 at 16:56):

And even for impl Deref, using regular parameter for &Self is recommended: https://marabos.nl/atomics/building-arc.html#mutation

This function does not take a self argument, but takes a regular argument (named arc) instead. This means it can only be called as Arc::get_mut(&mut a), and not as a.get_mut(). This is advisable for types that implement Deref, to avoid ambiguity with a similarly named method on the underlying T.

Asahi Lina (Feb 08 2023 at 17:09):

I understand how it's an antipattern in standard Rust, but in the kernel the problem is that the composition order is inverted in order to interact with C. In idiomatic Rust you'd use T contains O, but the kernel needs to carry around pointers to O, not T, and there is no sensible way to do that with a user-defined T as the outer struct as far as I know. And once you have O<T> contains T, it's very unergonomic not to be able to directly access methods in T from the outer O<T>.

I'd love to hear of a better solution if there is one ^^
