Hi! I just counted up all the instances of unsafe in my drm/asahi driver and I thought others might find the stats interesting.

UserPtr                     20
Object system internals     19
Zeroed/mem::zeroed()        18
Impl Sync/Send              15
Memory allocator internals  9
asm!()                      6
place!()                    6
Unions                      6
Bad/missing abstractions    5
MMU/Handoff internals       5
Pin projections             4
Cast to bytes               2
MMIO                        2
GpuWeakPointer.upgrade()    1
Pin::new_unchecked()        1
mem::transmute()            1
Misc raw pointer usage      1
pin_init limitations        1
  • UserPtr: Each read from userspace is 3 unsafe blocks. I'll probably abstract it out soon within the driver so I can at least avoid the duplication, but in general making this API safe requires something like qubes-castable so we can mark structs that are "bag of bytes" compatible. Once we have that, we can have a safe API for copies to/from userspace.
  • Object/Allocator/MMU internals: This is the kind of code that would be expected to use unsafe all over the place, so not much to do here. It's quite self-contained.
  • Zeroed: This is basically a duplicate of the Zeroable trait already upstream, so most of this will go away except for a few unsafe impl Zeroable instances. It would be nice to be able to auto-derive this, which would be easy with the Castable implementation.
  • place!(): Will be superseded by pin-init once it gains I think just one more feature I need.
  • Unions, asm!(), impl Sync/Send: These are quite fundamental, not sure we can do much about it. Unions consisting entirely of bag-of-bytes members are probably safe, but I'm not sure if there's a good way to implement an abstraction for that? It'd be neat if you could unsafely mark a union as "safe to access" the normal way with fields, instead of having to create accessor methods for each field to contain the unsafety...
  • Cast to bytes: Same story as before, needs some kind of Castable thing.
  • Bad/missing abstractions: Mostly the memremap() stuff in the MMU code, I'll write a proper abstraction for this before upstreaming.
  • MMIO: Fundamentally unsafe.
  • pin_init limitations: That's one where I have an array I need to pin_init, so I had to open-code that. It would be nice to have a pinned array initializer (that takes a closure that returns a PinInit<> per element).
  • transmute: Waiting on const_float_bits_conv becoming stable.
  • GpuWeakPointer.upgrade(): Unsafe by choice (this isn't actually unsafe by normal Rust rules, I just consider it unsafe in the context of the GPU object/pointer model)

The good news is basically everything is "boring"! It's either APIs which are fundamentally unsafe but where it's easy to verify they are being used safely, simple raw pointer manipulation in allocator/memory management code, etc. I only do "interesting" things with raw pointers in one or two places (the kind of situation where you actually need a long SAFETY comment to explain why the logic is sound).
