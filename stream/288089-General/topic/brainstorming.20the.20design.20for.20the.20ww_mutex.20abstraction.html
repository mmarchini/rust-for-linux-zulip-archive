Topic: brainstorming the design for the ww_mutex abstraction

Onur Özkan (Oct 23 2025 at 16:05):

I come up with some design options for ww_mutex and decided to share them here in a dedicated thread to ensure we are on the same page before I send the next patch as it will require a massive refactoring of the latest patch on the ML.

Currently, we have the ww_mutex.rs module, which is a bare (maybe a little bit more than that) wrapper with some type-level safeties and a higher-level exec.rs module. The exec module provides an automatic EDEADLK handling mechanism that cleans up the acquired locks and rolls back to try locking them again, all automatically.

However, both modules are pub(this is the first problem) and users of ww_mutex will be responsible for most of the safety concerns. For example, dropping WwAcquireCtx without releasing all the locks acquired by that context is incorrect (see this).

Therefore, we should release all the locks before calling ww_acquire_fini in PinnedDrop, but we are not doing this explicitly yet. While the lifetime bounds help, I am not sure if this is good/future-proof enough for safety. Furthermore, users have to handle EDEADLK manually if they use the low-level ww_mutex module and I believe they shouldn't be responsible for this, at least for the Rust side, and I think there could be more (potential) issues we have not realized yet.

The exec module on the other hand, provides a much safer API for the user. It might not be sufficient for all scenarios (yet), but it can definitely be improved to cover them.

There are two paths we can follow and I wouldd appreciate any feedback on them.

Option 1: Consolidate Logic into WwAcquireCtx

Dropping the exec module entirely and folding its logic into the ww_mutex types (e.g., by improving WwAcquireCtx). Doing this means we will need to add a mutable field to the pinned WwAcquireCtx, such as:

#[pin_data(PinnedDrop)]
pub struct WwAcquireCtx<'a, T: ?Sized> {
    #[pin]
    inner: Opaque<bindings::ww_acquire_ctx>,
    class: &'a Class,
    taken: KVec<MutexGuard<'a, T>>,
}

This then means we would have to use unsafe get_unchecked_mut to modify taken in various functions (like in drop and lock functions), like this:

// SAFETY: `self` is not moved and `taken` is unpinned.
let this = unsafe { self.get_unchecked_mut() };
this.taken.push/clear()...

Another thing is that we have a from_raw function (in the latest unpublished version) that takes a raw pointer and creates WwAcquireCtx. This makes me very uncomfortable because it taken field is becoming to not work in every case. If a user creates WwAcquireCtx from a C pointer, the taken field will obviously have no knowledge of the locks acquired from the C code. This makes me to believe that WwAcquireCtx is not the right type to contain the taken field and it makes the implementation less stable and convenient.

Option 2: Enforcing for higher-level API

Keeping ww_mutex as abstract and low-level as it is, but making it private (sealed) for internal use only (i.e., for the exec module). This would enforce users to use the safe, high-level API and allow us to improve that API for cases where it's currently insufficient for user needs. With this approach, no user can accidentally encounter invalid or unsafe cases (including ones we haven't yet recognized). Also from a maintenance perspective, I think this is much easier, it creates two levels of abstraction: one is a simple wrapper of the C side and the other wraps the Rust wrapper for actual use cases.

I am happy to hear alternative options, but for now I feel that option 2 is far better than option 1.

Lyude (Oct 23 2025 at 18:37):

so I was going to suggest making a raw version of the acquire context and a high-level version of it and just keeping them separate like you're doing, but I think there's an option 3 here.

So - I don't think we need to actually track the guards at all on the rust side of things! If we drop the KVec, the duplicate class ref, and the lifetime:

#[pin_data(PinnedDrop)]
pub struct WwAcquireCtx<T: ?Sized> {
    #[pin]
    inner: Opaque<bindings::ww_acquire_ctx>,
}

Then all we need to do is have MutexGuard<'a, T> be created by &'a WwAcquireCtx. We might need to make sure the lifetime encompasses both the lifetime of WwAcquireCtx and the lifetime of the mutex itself, but that should be easy ('a + 'b I think?). If we do that though, lifetimes will actually ensure that the drop order is correct and that any guards will always be dropped before the context. This also would still remain true even in situations where we received the WwAcquireCtx from a reference derived from a C pointer, as long as the reference lifetime is bound to something that outlives the guard.

Regarding handling EDEADLK, I think the easiest way of handling this might be to make it so that we only can create a lock context in two ways. The first being through a closure interface, that keeps trying the closure until it doesn't return a deadlock (or returns another error), the second being deriving the context from a C pointer. I think we do need both of these, because occasionally deadlock recovery is handled by a C function that's invoking a callback rather than something that happens within the callback itself - which means that we don't want special handling on EDEADLK In that case and want to return it like a normal error.
