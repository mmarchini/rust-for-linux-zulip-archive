The problem I'm hitting is that macro_rules! hygiene honestly seems to get in the way far more often then it helps, particularly because there are plenty of situations where I would like to pass an expression and explicitly have it evaluated during macro expansion and not at the callsite. This started with me trying to write this, rather crazy but with a reason macro:

/// Implement the repetitive from_opaque/try_from_opaque methods for all mode object and state
/// types.
///
/// Because there are so many different ways of accessing mode objects, their states, etc. we need a
/// macro that we can use for consistently implementing try_from_opaque()/from_opaque() functions to
/// convert from Opaque mode objects to fully typed mode objects. This macro handles that, and can
/// generate said functions for any kind of type which the original mode object driver trait can be
/// derived from. All conversions check the mode object's vtable. For example:
///
/// ```
/// impl<'a, T: DriverConnectorState> ConnectorState<T> {
///     impl_from_opaque_mode_obj! {
///         // | An optional lifetime and meta-variables to declare for each function
///         // |          | The type of the input parameter `opaque`
///         // |          |                               | The return type for try_from_opaque()
///         // |          |                               | from_opaque() will return &'a Self
///         // v          v                               v
///         fn <'a, D, C>(&'a OpaqueConnectorState<T>) -> Option<&'a Self>
///         where // <-- An optional set of additional trait bounds to match against
///             T: DriverConnectorState<Connector = C>
///         use
///         //  | Meta-variable that will contain ::OPS (the auto-generated vtable)
///         //  |    | The driver trait implemented by the driver for generating the vtable
///         //  |    | It will add the bound C: DriverConnector<Driver = D> to the function
///         //  v    v
///             C as DriverConnector,
///         //  | Meta-variable to assign to KmsDriver
///         //  |    | This must always be KmsDriver, it's just here for clarity
///         //  v    v
///             D as KmsDriver,
///             |o| o.connector() // The closure used for obtaining a reference to the mode object
///         return
///         //  | The value to return on success, must be a pattern like Ok(full) or Some(full)
///         //  | `full` is defined as the fully qualified type from the onversion
///         //  |        | The value to return if the conversion fails, must also be a pattern
///         //  |        | `opaque` is defined here and can be used with Err() for functions that
///         //  |        | want to return the original un-converted input value

This macro doesn't work because if you try something like:

    /* … */
    return Ok(full), Err(opaque)

rust gets upset that it can't evaluate full or opaque at the call-site, which isn't what we want. As many devs would point out: why not use a procedural macro?

My reasoning is that I'd like to find a workaround for this because unfortunately I kind of think if the purpose of macro_rules! hygiene is to make code more legible, boiler-plate code like what's above does not in any way seem more legible as a complex block of AST-generating code rather than existing in a form that closely resembles the actual functions it's generating. I don't have anything wrong with procedural macros btw, I think they're great for a lot of things! But it really does not seem like the right tool for something as simple as this, where the boilerplate just happens to be varied enough I don't see any reasonable way of doing this without macros. So, do we have any kind of "please don't evaluate this at the callsite" escape hatch, or is anyone aware of any ongoing rust work to add support for this? Could we even try to make a case to rust upstream to add something like this?

The reason for macro hygiene is not to make code more legible, but to prevent that macros aren't general enough. So for example you have a macro:

macro_rules! foo {
    ($value:expr) => {
        let mut foo = 0;
        foo += $value;
        println!("{foo}");
    }
}
// then you can use it like this:
let x = 42;
foo!(x); // this prints 42

// but if we had no hygiene, then this would be different:
let foo = 42;
foo!(foo); // this prints 0!

So essentially macro hygiene prevents renaming of variables, constants and functions from affecting program behavior. It is very important, as this is a huge problem in macro systems that don't have it, because you have to name all of your in-macro variables "__macro_internal_var..." (and then still you have the problem of what happens if a macro-generated variable is input into a macro...).

I think it would be OK. The compiler assigns a parse context to the identifier. If the fullidentifier is created in the macro context, the return expression you pass in cannot reference it. But if the context of full is the outside context, you should be able to reference it from the return expression you pass in. But you would have to let $full_id = ... in the macro.
