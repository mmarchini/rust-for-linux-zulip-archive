Topic: Attribute macros for kernel version conditional compilation

Gilberto Bertin (Mar 16 2026 at 20:49):

I played a bit with implementing attribute macros for conditional compilation based on kernel versions:

use kernel::macros::*;

#[before(7, 0, 0)]
fn call_kernel_fn() {
    // implementation for kernels < 7.0.0
}

#[since(7, 0, 0)]
fn call_kernel_fn() {
    // implementation for kernels >= 7.0.0
}

I think this can be useful for out-of-tree modules, as these can then target particular kernel versions' features, especially as more and more kernel APIs gets ported or wrapped with every new release.

As this would be my first contribution here, before trying to send the patch I wanted to ask some questions and gather some preliminary feedback.

First of all: do you also think there can be some value in this change?

Next: at the moment everything lives in the macros crate: a couple of macros, an extra binding with related Makefile changes for LINUX_VERSION_CODE and some helpers to handle conversions between the (major, patchlevel, sublevel) and the u32 kernel version format.

For completeness I think these bindings and helpers should be exposed also in the kernel crate, so that a module can interact directly with the kernel version, but I'm unsure which approach would be best to follow:

  • Duplicate the bindings and helpers in both the kernel and macros crates? Perhaps not, even if it's just a few LOCs
  • Extract that logic into a separate crate shared between kernel and macros? Not sure if that would be useful for anything else in the future?
  • Some other way to keep the bindings and helpers in the kernel crate and access them in the attribute macros in macros?

Gary Guo (Mar 17 2026 at 02:49):

Upstream kernel won't accept code (and the associated maintenance burden) just to make out-of-tree modules easier. Note that this macro will be completely dead-code for the upstream.

Gilberto Bertin (Mar 17 2026 at 07:25):

Makes sense, I thought this could be justified with the argument of porting the KERNEL_VERSION()/ LINUX_VERSION_CODE API to Rust, as without a macro we can't really replicate the behavior of the preprocessor eliminating unsupported code (and I don't think we can define attribute macros in out-of-tree modules?)

David Rheinsberg (Mar 17 2026 at 08:44):

You can always carry those in out-of-tree modules. But apart from symbol-licensing, I cannot think of any APIs that were merged exclusively for out-of-tree modules. There is just a general dislike of those in the kernel community.
