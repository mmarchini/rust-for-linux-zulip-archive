Gary Guo (Jan 27 2026 at 14:50):

I am reviewing the THIS_MODULE series sent by @Kari Argillander and I think I understand why generics is needed here.

The problem is that a constant is needed for module pointer, and whether this is &__this_module or NULL depends on whether the crate is built-in or compiled as a module.

As we build VTABLE using generics, this needs to be something exposed via the type system, and cannot just be a reference to THIS_MODULE (which is a static that we generate per crate).

However, I think the approach in the series is quite verbose and thus non-ideal. I think there're two simpler approaches:

  1. Have #[vtable] macro implicitly add a reference to THIS_MODULE (Just the constant, so no trait is needed)
  2. Change the C side so that instead of NULL, we have a __this_module for built-in modules. Then we can unconditionally reference __this_module symbol, and it will resolve to either the special built-in __this_module, or the specific one for removable modules.
