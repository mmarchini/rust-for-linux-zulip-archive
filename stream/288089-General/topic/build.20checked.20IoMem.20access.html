Alice Ryhl (Feb 25 2025 at 08:01):

Try marking sgx_read32 inline or making the offset into a const generic.

fn sgx_read32<const OFF: usize>(&self) -> u32 {
    let io = self.sgx.try_access().unwrap()
    io.readq_relaxed(OFF)
}

I wonder if the build_assert! is too fragile to use in practice.

Janne Grunau (Feb 25 2025 at 09:04):

both inline and const generics avoid the issue.

My main complaint against build_assert! is against the ergonomics. Failing at link time takes ~26 seconds for single change in the asahi driver trying to fix the build_assert!. The reporting of the issue is very vague. When compiling the driver as module only the missing rust_build_error symbol is reported (gcc). Built-in at least the function referencing the symbol is reported but if it is a leaf function like kernel::io::Io<16777216>>::io_addr_assert::<u32> it's not very helpful. <asahi::regs::Resources>::sgx_read32 which I've seen as well is not much better. Depending on compiler optimizations for how to report build time assert is not user friendly.

Alice Ryhl (Feb 25 2025 at 09:07):

It might make sense to make the offset into a const generic and never perform the build-time checks with the methods that don't take a non-const-generic.
