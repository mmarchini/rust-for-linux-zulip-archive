Topic: Unintended rust-analyzer behavior after update

Jesung Yang (Dec 03 2025 at 10:18):

After the commit 50384460c68f ("Rewrite method resolution to follow rustc more closely") to rust-analyzer (included in release 2025-11-24 (v0.3.2693)), our current rust-project.json setup does not provide proper IDE support for inherent methods of primitive types:

fn foo() {
    // rust-analyzer does not provide LSP features like autocomplete,
    // go-to-definition, etc. on `rotate_left`.
    let _ = 0_i32.rotate_left(0);
}

This is because the method resolution logic has changed. A detailed description of the root cause can be found in this thread [1].

To fix this issue without touching the rust-analyzer side, I think the following steps are needed:

  1. Re-enable sysroot_src in rust-project.json.
  2. Remove -Zcrate-attr=no_std from scripts/Makefile.build.
  3. Explicitly add #![no_std] to all kernel modules.

... or simply downgrade rust-analyzer.

Appreciate any thoughts on this issue. Specifically, I'm curious about why the unstable -Zcrate-attr=no_std is used instead of #![no_std].

[1] https://rust-lang.zulipchat.com/#narrow/channel/185405-t-compiler.2Frust-analyzer/topic/Primitive.20type.20inherent.20method.20lookup.20fails/with/561607963
