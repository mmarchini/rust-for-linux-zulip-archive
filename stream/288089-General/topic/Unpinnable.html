Benno Lossin (Mar 05 2023 at 20:34):

I think that there are two different notions of "move" that are used in the docs/here, I will call them:

  • "rust" move: "x = y;" a move, i.e. dropping x, copying y into x and then dropping y
  • semantic move: if we have some pinned data x and then use ptr::read to copy it into some other location calling it y and then call some function on y. Then this object can observe the change in its memory location, i.e. it has moved.

When talking about "pinned data is not allowed to be moved" I think of the semantic kind.

Alice Ryhl (Mar 05 2023 at 20:50):

Pinning is a safety invariant and not a validity invariant, so breaking its promises does not, strictly speaking, lead to language UB. You only get true UB if you write unsafe code that relies on the pinning guarantees, and then break the pinning guarantees in a way that the unsafe code did not expect.

Alice Ryhl (Mar 05 2023 at 21:16):

Using terminology from Two Kinds of Invariants: Safety and Validity 3, basically every aspect of Pin is in the realm of "safety invariants", not "validity invariants". This means that violating these rules might not immediately cause UB in a way that Miri can detect or that could go wrong at runtime, it "just" means that invariants are broken and if control enters any code that depends on these invariants, it's your fault when that code goes wrong.
