Miguel Ojeda (Jan 14 2025 at 12:21):

Thanks for letting us know about it!

A few high-level points:

  • Does this target upstream Comprehensive Rust? They have Chromium there, but I am not sure if they would like to have other projects in general, or whether it applies for a book like that.

  • Our "entry point" is https://rust-for-linux.com -- I would recommend checking the subpages/content there and referencing them (e.g. on contributing). I see you have linked a couple subpages, but it may be useful to link others too (e.g. for the unstable features and nightly discussion: https://rust-for-linux.com/unstable-features and https://rust-for-linux.com/rust-version-policy), or to refer to those (e.g. the Contact page vs. just the Zulip in "Next Steps"). It may be a good idea to refer to the main page in other places like "Welcome", since the website has most of the important links already.

  • If this is meant as a text for kernel hackers as you say, i.e. people that already have some experience with the kernel (but not Rust), then it would be great to focus more on abstractions and their relation with the C side, rather than e.g. how to set things up which is already covered in the upstream docs. By the way, for some things, perhaps it may be a good idea to try to contribute to the upstream documentation, i.e. to https://docs.kernel.org/rust/ or even to the code documentation directly (e.g. as examples or more in-depth explanations, maybe).

  • In general, I would suggest focusing on in-tree things vs. out-of-tree. That makes it more relevant, and perhaps easier to track for others. For instance, I saw links to out-of-tree abstractions, but at this time you could probably base the book on things that are already in-tree or at least in the mailing list. That may help keeping things up-to-date too.

  • If this is intended as an introductory text, I would avoid mentioning implementation details (e.g. I noticed a section about how to implement a module, but module writers should not really need to care about that, especially in an introductory text).

  • There are some bits I saw while scrolling that should probably be updated, e.g. the bit about needing nightly (we have used stable compilers since very early on, and since half a year ago we even have an MSRV! :) or the line that seems to be taken from the out-of-tree template but that is used to explain the in-tree rust-analyzer setup (by the way, it may be best to use our tracking issue #2, not the Rust project goal, as the link there).

I hope that helps!

Frances W (Jan 14 2025 at 12:39):

  • Yes, this targets upstream Comprehensive Rust; I don't think there's an explicit policy stating for which projects they accept coverage, but I've heard enthusiasm about this from Martin and other Comp Rust folks. At some level Google has the final say about what goes upstream, and I think want to help people get about the Rust for Linux train.
  • I'll link more of the rust-for-linux.com pages. I wasn't sure how much was necessary vs. letting people browse themselves, but links are free!
  • The impetus for writing was an audience of this type, but I don't think it's necessary to limit the scope to only experienced kernel developers since Rust has the potential to make kernel dev much more accessible. I'd love to have more to say about the relation between abstractions and C--are there particular resources beyond the code that I should look at? I do want to avoid simply duplicating docs that should be part of the code or kernel doc in a location that's more likely to go out of sync.
  • I'll look into replacing some of the out-of-tree references with in-tree/ML ones.
  • W/r/t implementation details, in my experience people learning Rust often have questions about the plumbing between things they know (e.g. low-level C details) and a new "shiny" interface as Rust presents, so I did want to address those questions directly in a kernel context. But maybe it would be good to separate the discussion of implementation details from the interface so that readers can focus on the side that concerns them.
  • I'll update the nightly reference and line from the out-of-tree template--I forgot about the escape hatch, so I just figured unstable features meant nightly compiler. These are the kinds of mixup that I really appreciate feedback on, as I don't want to mislead folks by synthesizing conflicting sources wrongly as a relative outsider not working on RfL on a daily basis.

Thanks for the feedback! It's very much appreciated.
