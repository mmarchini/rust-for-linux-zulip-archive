Should the default be to require bounce buffers for DMA, and to only avoid bounce buffers if it is absolutely necessary for performance? This allows true IOMMU protection from DMA attacks, which is otherwise almost impossible.

Asahi Lina (Oct 03 2022 at 05:09):

Should the default be to require bounce buffers for DMA, and to only avoid bounce buffers if it is absolutely necessary for performance? This allows true IOMMU protection from DMA attacks, which is otherwise almost impossible.

You don't need bounce buffers for DMA protection as long as your allocations come from a DMA pool or are page-aligned, assuming there is an IOMMU in the system. Unless you're talking about the subtle memory model issues (the fact that on paper it's UB for a device to mutate memory out from under the Rust memory model), but that's largely a theoretical problem, and it's something to be fixed in the language/compiler, not a reason to start using bounce buffers everywhere...

Asahi Lina (Oct 03 2022 at 17:20):

Demi Obenour said:

Ah, sorry, I think I confused you with my driver which is a very special case. GPUs have their own MMU (and in this case there is no extra IOMMU level behind it), and so my driver has to do the duties of both an IOMMU driver (though we don't call it an IOMMU, it basically is one) and the consumer driver. To do that it uses the io_pgtable API which just manages a page table, it doesn't actually map anything to hardware (IOMMU drivers themselves are consumers of that same API). It shouldn't be safe to map arbitrary pages into an actual IOMMU instance through a public interface. What is, technically, safe is putting mapping entries into a page table that isn't, a priori, in use by any IOMMU, which is what io_pgtable does.

The codepath in my driver actually does ultimately go through the DRM core helpers which handle any real, standalone IOMMU that could exist, and will only map pages belonging to objects allocated through those APIs properly, and will unmap them when those objects are freed. So if the platform had a standalone, separate IOMMU protecting the GPU, all that would be handled transparently and safely already. It's just that this platform doesn't have one, it relies on the internal GPU MMU only for safety, and that has to be managed by the GPU driver itself.

Asahi Lina (Oct 03 2022 at 17:28):

Demi Obenour said:

That’s fine. It does, however, mean that pages accessible to a device need to be treated just as carefully as pages accessible to userspace.

Yes, and one of the questions came up was whether there is some way of forbidding drivers from, for example, putting Rust references or CPU pointers in structures mapped to those pages. Though from a soundness perspective, whatever internal mechanism ends up pointing or moving Rust structs into such device-accessible pages has to be unsafe anyway.

I do agree that post-facto mapping of memory to a device should be unsafe, but there are actually many footguns with trying to do that and it's not normally how you'd do it either way! Normally you allocate memory from the get-go with the intent of mapping it to a device (which could come from special memory ranges and things like that), and the underlying DMA APIs handle IOMMUs for you. So then it's up to you what you do with what, at that point, has to be a raw pointer.

Some simple safe abstractions over that could allow copying byte data in, but if you want to access complex structures in DMA memory safely, you'd need type system features that assert those things (mainly that all bit patterns have to be valid for all types contained within any structs writable by the device, with the corollary that that forbids any CPU-side pointers or refs, and pretty much any complex Rust type). I'm not sure if Rust has anything for that, though... my driver doesn't attempt to fix this problem internally, so in other words, if its internal object management system were a public interface shared with other drivers, it would have to be marked unsafe since it has extra requirements for soundness that it cannot enforce automatically.
