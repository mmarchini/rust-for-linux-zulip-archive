Would it be worth adding a macro like this to utils?

/// If the first expression is true, return `Some(second_expression)`. Otherwise
/// return `None`.
///
/// ```
/// let t = true;
/// let f = false;
/// assert_eq!(if_then_some(t, "hello"), Some("hello"));
/// assert_eq!(if_then_some(f, "hello"), None);
/// ```
///
/// This is a version of `bool::then`that can be used in const contexts
macro_rules! if_then_some {
    ($if_this:expr, $then_some:expr) => {
        if $if_this {
            Some($then_some)
        } else {
            None
        }
    };
}

Wedson Almeida Filho (Oct 09 2023 at 12:59):

Benno Lossin said:

Ariel Miculas said:

Wedson already answered this question.

I do not know how the C side looks like, but maybe we could just call the C function that provides the default behavior.

I wish it were that simple. I'll show you a couple of examples.

Consider do_dentry_open: https://elixir.bootlin.com/linux/latest/source/fs/open.c#L919 -- here it modifies flags based on the function pointers being non-null. We can't really replicate this by "calling the C function that provides the default behaviour". With some luck we could potentially find a place where we could replicate this behaviour, but even this wouldn't be ideal because we would have to keep it in sync with the C side.

Let's look at __kernel_read (https://elixir.bootlin.com/linux/latest/source/fs/read_write.c#L418): 

    /*
     * Also fail if ->read_iter and ->read are both wired up as that
     * implies very convoluted semantics.
     */
    if (unlikely(!file->f_op->read_iter || file->f_op->read))
        return warn_unsupported(file, "read");

vfs_read, however, allows both of them to coexist (https://elixir.bootlin.com/linux/latest/source/fs/read_write.c#L467) and uses a static (new_sync_read) function to emulate read from read_iter. do_iter_read does something similar but the other way around, it uses a static (do_loop_readv_writev) function to emulate read_iter from read. Let's say we convince that C maintainers to expose those functions to us so that we can have default implementations. If a module were to leave both of them unimplemented (i.e., using the default implementation), we would have an infinite recursion.

file_needs_f_pos_lock checks iterate_shared (https://elixir.bootlin.com/linux/latest/source/fs/file.c#L1052):

static inline bool file_needs_f_pos_lock(struct file *file)
{
        return (file->f_mode & FMODE_ATOMIC_POS) &&
                (file_count(file) > 1 || file->f_op->iterate_shared);
}
