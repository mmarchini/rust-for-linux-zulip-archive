Topic: Splitting `ClockSource` into `TimeSource` and `ClockSource`?

Lyude (Jul 11 2025 at 19:40):

(cc: @Andreas Hindborg @FUJITA Tomonori )
Hey! So I mentioned in one of the RFL meetings that I've been rebasing rvkms and trying to figure out how to handle all of the time related changes, and one of the issues I hit was whether or not to actually be using Instant types in the modesetting vblank API - since timestamps for vblanks usually come from a timing source on the modesetting device itself and not one of the standard system clocks, but the C side of DRM this uses ktime_t for vblank timestamps.
ClockSource on its own though doesn't really seem completely appropriate though, we can't provide a clock ID for a source like this since none exists.
So I had the idea of potentially splitting ClockSource into two traits to deal with this: a TimeSource that any kind of timestamp providing source can implement (-maybe- with ktime_get(), but I'm not sure yet), and then ClockSource which would be a supertrait of TimeSource and handle all of the kernel timer specific bits, e.g. ID. Then we would be using Instant<T: TimeSource> and only binding T to ClockSource as needed. Does this seem reasonable?
