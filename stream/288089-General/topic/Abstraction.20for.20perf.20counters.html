Rob Herring (Feb 20 2025 at 21:23):

@Dennis Kobert
What h/w or arch are you looking to do this on? The performance counters are typically already consumed by 'perf' drivers and exposed by the perf core to userspace. If you are not using any of that, it is unlikely that a 2nd driver for the same h/w would be accepted upstream. If this is just a new perf driver in Rust, I'm not aware of any efforts on bindings. The perf subsystem is pretty unique compared to most kernel subsystems.

Dennis Kobert (Feb 20 2025 at 21:34):

This is basically how using the abstraction currently looks like:

        let counter = EventBuilder::new(EventType::Hardware(HardwareEvent::CpuCycles))
            .cpu(CPU_NUM)
            .sample_period(SAMPLE_PERIOD)
            .on_overflow(|_, _, _| pr_info!("Perf counter has overflowed"))
            .disabled(); // Start disabled so we can control measurement
        counter.enable();
        counter.read();
        counter.disable();
