Linux kernel – From C to Rust for Linux: Semaphores and Mutexes

This text compares some aspects of Linux kernel programming differences between C and Rust for Linux (RFL) for lock handling with Mutexes or Semaphores. It is intended for in-kernel-users of these functionalities like device driver or subsystem authors who are familiar with the C programming concepts but want to switch to RFL, now. It looks at the differences in the programming concepts and is not intended as an API reference.

  • The C side
  • The RFL side
    • Data protection
    • Guards
    • No explicit unlock
    • Irqsave (preliminary, not mainlined yet)
    • debug_assert!()

The C side

On the well-known C side lock handling like with Mutexes or Semaphores is using the concept of critical sections. If you want to protect something regarding concurrency issues, you do this in a lock protected section:

lock_get();                                                |--
/* Do everything what needs concurrency protection here    |
 * (as long as it is allowed)                              | critical section
 */                                                        |
lock_release();                                            |--

Between lock_get() and lock_release() you can do everything. Independent on what data is processed here. As long as it’s not forbidden to be done in the selected type of critical section protection. The topic to stress is that a code section is protected here.

The RFL side

While the RFL implementation of Mutexes and Semaphores in its abstraction uses the C’s side mutex/spin_lock()/unlock() via bindings as well, the result is quite different. For a nice discussion of how and why this design was chosen see for example the chapter 4: Building Our Own Spin Lock from the O’REILLY book “Rust Atomics and Locks”.
On the RFL side thanks to Rust’s object orientation locking is based on critical data. That means that the lock is bound to the data which needs to be concurrency protected. And not to a code section. See the RFL SpinLock and Mutex documentation for the detailed API reference and usage examples.

Data protection

Both of the given examples use a data structure

struct Inner {                                         |-
    a: u32,                                            | critical data
    b: u32,                                            |
}                                                      |-

being protected by the lock. That means that in contrast to the C critical section here the critical data is protected or synchronized for concurrent access.

This contrasts to C where a section of code is protected.

Guards

Taking the lock the return value is guard. This

  • is used to access the locked elements/data
  • ensures that as long as it lives the lock is held
let mut guard = m.lock();
guard.a += 10;
guard.b += 20;

This contrasts to C where everything within the protected code section can be accessed “normally”.

No explicit unlock

The above section about the guard implies it, already: If the guard is living, the lock is held. That means that as soon as the guard is not living any more, the lock is released. “Automatically”! By the guard’s drop()):

    let mut guard = m.lock();
    guard.a += 10;
    guard.b += 20;
}                              <= lock released here (“automatically”)!

This contrasts to C where an explicit lock release in the code is required.

Irqsave (preliminary, not mainlined yet)

In C there are explicit functions for spin lock with controlling the interrupt state. For spin locks with controlling the interrupts there are spin_lock_irqsave()/spin_unlock_irqrestore(). Using them will lock the critical section and at the same time (!) control the interrupt state. Typically disable the interrupts at critical section entry and re-enable them at critical section exit with taking/releasing the lock.
In RFL we have functionality to create an execution section with interrupts disabled:

with_irqs_disabled(|irq| {      <= Interrupt state change (disable)
       /* Add the code to run with disabled interrupts here */
});                             <= Interrupt state change (re-enable)

In this disabled interrupt section code which needs this context can be executed. For irqsave locks in this section locking can be executed nearly the same way like non-irqsave locking. Just use lock_with(irq) instead just lock():

with_irqs_disabled(|irq| {
    let mut guard = m.lock_with(irq);
    guard.a += 10;
    guard.b += 20;
});

With this interrupt disable locks have to be in a with_irq_disabled() section. But on the other hand, a with_irq_disabled() section does not need to have a lock. It can be whatever other code which needs interrupts disabled.
This contrasts to C where combined explicit lock-irqsave functions are provided.

debug_assert!()

If RUST_DEBUG_ASSERTIONS is enabled in the Kernel’s Kconfig system, the RFL (lock) abstraction includes some additional checks which will assert at runtime in case there is some mis-use. So it is a good idea once the development has finished to do a runtime verification with that assertions being enabled. Enabling this for a production system is not recommended, though.

Linux kernel – From C to Rust for Linux: Conditional compilation

The C side

On the well-known C side we have the preprocessor. Besides a lot of other things, it enables us to use conditional compilation like

#ifdef CONFIG_X
  g1();
  g2();
  g3();
#endif

where the functions g1() to g3() are called (compiles in) only if CONFIG_X is defined.

The RFL side

That heading of this section is misleading here. This is not a Rust for Linux feature, but a Rust feature. This Rust feature is called Attributes in general. One use case of these attributes is the conditional compilation. Even though not specific to RFL, it is mentioned here to explain the difference to the C usage.
Important here is that we don't have any kind of end marker like the #endif. The attribute, here specifically the #cfg attribute, is just valid for the item immediately following it. Without end marker:

#[cfg(CONFIG_X)]
g1();
#[cfg(CONFIG_X)]
g2();
#[cfg(CONFIG_X)]
g3();

This could be simplified by:

#[cfg(CONFIG_X)]
{
  g1();
  g2();
  g3();
}

This contrasts to C where a code section with #ifdef and #endif is "valid" for everything what is encapsulated by it.

See the discussion of inner and outer attributes and cfg in Rust By Example.
