Topic: alloc_hooks and statics duplicated per function

Alice Ryhl (Dec 03 2025 at 13:56):

If you look in include/linux/alloc_tag.h you will find that many allocation functions are wrapped in alloc_hooks() using a #define. This macro is declared as follows:

// a few different variants of this one depending on config
#define DEFINE_ALLOC_TAG(_alloc_tag)                        \
    static struct alloc_tag _alloc_tag __used __aligned(8)          \
    __section(ALLOC_TAG_SECTION_NAME) = {                   \
        .ct = CODE_TAG_INIT,                        \
        .counters = NULL };

#define alloc_hooks_tag(_tag, _do_alloc)                \
({                                  \
    typeof(_do_alloc) _res;                     \
    if (mem_alloc_profiling_enabled()) {                \
        struct alloc_tag * __maybe_unused _old;         \
        _old = alloc_tag_save(_tag);                \
        _res = _do_alloc;                   \
        alloc_tag_restore(_tag, _old);              \
    } else                              \
        _res = _do_alloc;                   \
    _res;                               \
})

#define alloc_hooks(_do_alloc)                      \
({                                  \
    DEFINE_ALLOC_TAG(_alloc_tag);                   \
    alloc_hooks_tag(&_alloc_tag, _do_alloc);            \
})

So this means that each call to kmalloc expands to declaring a new global variable for counting allocations from this call-site.

Right now, all Rust code is going to share a single global declared in rust/helpers/, so alloc_hooks() combines all Rust code throughout the entire kernel. The INLINE_HELPERS patch doesn't help here (ok actually it helps a bit, because .ko modules get a separately compiled copy of helpers, so each module has its own global). To really fix it, we would need some way for things such as KBox::new to expand to a new instance of this static.

But we can't really create these statics using Rust statics, afaict. We want them duplicated per callsite, including when generics is in play.

What's the solution here? Is the only way to convert the C static into an inline asm block containing a section block? I think those do get duplicated by inlining and generics ...

Thoughts?

I made this about alloc_hooks(), but I don't think it's the only case in the kernel where this pattern occurs.
