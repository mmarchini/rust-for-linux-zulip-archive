...

    pub fn begin_signalling(&mut self) {
        // FIXME: this needs to be mutable, obviously, but we can't borrow mutably. *sigh*
        self.signalling = true;
        // TODO: Should we warn if a user tries to do this several times for the same

        // SAFETY: `dma_fence_begin_signalling()` works on global lockdep data and calling it is

        self.signalling_cookie = unsafe { bindings::dma_fence_begin_signalling() };
    }

The signalling boolean is needed to mark whenever someone in Rust has already called begin_signalling(). The cookie is for the C backend.

Setting those can be asynchronous. So I can't make that mut. I need to protect them with a lock.

The lock protecting the fence is in the fence context, fctx.
Having a second lock in the Rust DmaFence would be racy, because the inner lock is what prevents fences from being signalled.

Long story short, I need to take the fctx lock in begin_signalling(). Once I have the lock, I know that no one can touch those booleans. Rust, however, doesn't know that this is OK and doesn't allow me to modify that if it's not mut, and mut it cannot be because it can be accessed asynchronously.

The normal Rust way of doing things would be to have the SpinLock contain the data I protect, and then access the data through a guard. In this case, however, the lock is effectively embedded into what I want to protect, not the other way around.

(I know that I should use SpinLock<()> btw, Boqun and others told me. Will do. But that won't change the fundamental problem)

I suppose I'm not the first person with such a problem so I suspect there is a canonical solution for issues like that.

Any tips or hints?

I don't think so. I think you only really want Opaque or UnsafeCell (I mention the latter since it's how the former is implemented, and the same rules apply) in situations where you aren't able to guarantee that rust's data aliasing rules will be upheld. A common situation I see this in is where you have a C structure where you know when it's safe to access/mutate individual fields in the struct, but you can't guarantee that another thread won't mutate other parts of the struct. This is in addition to what Alice said about allowing undefined values of course.

Anyway - I think if rust code is the only one reading/writing from that field (I'd be surprised if it wsn't just from that excerpt) you're fine not using an Opaque. If you don't want users to be able to touch that field without going through your interfaces, that's more of a matter of ensuring two things:

  • That you don't expose accessors for signalling_cookie to users
  • That you fulfill one of these two conditions:
    - Make sure &mut references to the struct are never exposed to users (since this would allow mem::swap(), which would potentially allow mutating the field without going through your interfaces)
    - That you do expose &mut references, but that mem::swap() would not leave the struct in a bad state. More often then not if the struct you're working with doesn't need to be pinned, it's probably fine to allow mem::swap().
