Topic: Unsoundness in `container_of!`

Léo Lanteri Thauvin (Jul 31 2021 at 18:16):

While working on #351, I noticed that container_of! is unsound, i.e. you can trigger UB from safe code. Here's an example:

struct Foo {
    a: (),
}

fn main() {
    container_of!(core::ptr::null(), Foo, a);
}

This is already UB because it violates the following condition for pointer::offset:

Both the starting and resulting pointer must be either in bounds or one byte past the end of the same allocated object.

And sure enough Miri wil complain:

error: Undefined Behavior: pointer arithmetic failed: 0x0 is not a valid pointer
   --> /playground/.rustup/toolchains/nightly-x86_64-unknown-linux-gnu/lib/rustlib/src/rust/library/core/src/ptr/const_ptr.rs:235:18
    |
235 |         unsafe { intrinsics::offset(self, count) }
    |                  ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ pointer arithmetic failed: 0x0 is not a valid pointer
    |
    = help: this indicates a bug in the program: it performed an invalid operation, and caused Undefined Behavior

So I was wonder if this is a known problem, if so why that is (is it because it is not high priority?), and if not how it should be fixed. If it needs to be fixed I see two options, which is preferred?

  • make container_of! unsafe to use, so you need an unsafe block to use it
  • make container_of! safe to use by replacing offset with wrapping_offset, which is safe to call but doesn't allow for as aggressive compiler optimizations as offset.

Léo Lanteri Thauvin (Jul 31 2021 at 18:53):

Also, here's a relevant passage from the wrapping_offset docs:

Compared to offset, this method basically delays the requirement of staying within the same allocated object: offset is immediate Undefined Behavior when crossing object boundaries; wrapping_offset produces a pointer but still leads to Undefined Behavior if a pointer is dereferenced when it is out-of-bounds of the object it is attached to. offset can be optimized better and is thus preferable in performance-sensitive code.

Wedson Almeida Filho (Old Google account) (Jul 31 2021 at 19:03):

using wrapping_offset is not sufficient to make container_of safe. Consider the following case:

struct Foo {
    a: u32,
    b: u32,
}

fn main() {
    let x = 0u32;
    let y = container_of!(&x, Foo, b);
}

The violation here is that x is not a field of Foo, and the compiler can't enforce this for us, thus the need for unsafe and the safety requirement in the docs: Callers must ensure that the pointer to the field is in fact a pointer to the specified field, as opposed to a pointer to another object of the same type.
