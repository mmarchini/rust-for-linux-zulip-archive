Topic: usize for large memory regions

Joel Fernandes (Dec 06 2025 at 22:31):

Hi all,
I wanted to discuss support for 32-bit architectures in the Rust core kernel code.

While drivers (such as Nova) target 64-bit only, the core Rust abstractions are intended for 32-bit as well if I am not mistaken. This creates an issue when dealing with large resources such as GPU video memory.

I recently ran into this with the GPU buddy allocator bindings. These bindings need to address VRAM that can exceed 4GB. On a 32-bit architecture, usize is 32-bit, which means using it for these "size" or "offset" values would result in data loss and not work.

Currently, I use u64 explicitly instead of usize to represent these values.

My question is: What is the recommended way to handle this in Rust in core kernel? Is there a preferred abstraction or best practice for supporting large address spaces on 32-bit targets without having to resort to using u64 everywhere usize would normally be used?

Alice Ryhl (Dec 07 2025 at 12:58):

If the underlying C api doesn't support 64-bit numbers, then you're out of luck. But I would expect a GPU-specific allocator to use 64-bit always?

Alexandre Courbot (Dec 08 2025 at 08:58):

I would expect sizes that are explicitly 64-bit (due to the hardware device working with that size) to use u64 in the code, with usize being used when dealing with the size of the CPU address bus

Joel Fernandes (Dec 08 2025 at 15:56):

Alex, Sure. For nova, considering it is 64-bit only, do we use usize or u64 for large size/offset data. It would be good to be consistent with rust core kernel code though.

Joel Fernandes (Dec 08 2025 at 15:58):

Alice, yes the allocator in C uses u64. I am ok with sticking with u64 then. I have already converted to using u64.

Alice Ryhl (Dec 09 2025 at 03:29):

For GPU memory that's 64-bit, using u64 seems like the way to go.
