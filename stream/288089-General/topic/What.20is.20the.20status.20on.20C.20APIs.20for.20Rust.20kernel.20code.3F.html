Topic: What is the status on C APIs for Rust kernel code?

Daniel Almeida (Feb 09 2024 at 16:57):

Hi guys,

I wonder what is the status of writing C APIs for Rust code in the kernel?

I know cbindgen is not going to work since it requires cargo, but I wonder if the usual approach of declaring functions with C ABI (pub extern "C" fn foo()) coupled with a compatible C header file will work?

My intention here is to rewrite a small kernel library in Rust and then provide a C API for existing users.

Alice Ryhl (Feb 09 2024 at 18:44):

We don't have cbindgen, but you can still do it manually. Rust Binder does that to let binderfs call into Rust Binder.

Miguel Ojeda (Feb 09 2024 at 18:57):

Yeah, we don't have cbindgen because we do not really need it yet, but we may want to explore it at some point. For the moment, as Alice says, you can write your own header.

However:

I know cbindgen is not going to work since it requires cargo

What do you mean? It does not require Cargo to use it, i.e. they offer a CLI tool like bindgen does.

Miguel Ojeda (Feb 09 2024 at 19:07):

Perhaps you mean that they offer the option to fetch the metadata of a crate from Cargo (because it is the only example they give in their front page), but you can also pass it your own crate root, e.g. cbindgen --lang c -- x.rs.

Hongyu Li (Mar 06 2024 at 03:58):

We have used cbingen to generate the header files for the Rust crate by command line. One problem is that cbindgen can not handle structures like Arc and Spinlock. The C side only has a typedef which can only be used as a pointer. Thus we have to manually write a structure in the C side that has the same size and alignment as the Rust.
