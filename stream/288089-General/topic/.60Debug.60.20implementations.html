Trevor Gross (Sep 10 2024 at 10:15):

I think this already works with the pr_* macros, like pr_info!(“foo {:?}”, [1, 2, 3]) should print [1, 2, 3]. But we don’t seem to have it for any of our types, so you can’t pr_info!(“foo {:?}”, my_rbtree) and print its contents.

rbtree might not be the best example, but it is pretty handy for debugging to have the implementation on almost everything (even if it just writes the type name when it doesn’t make sense to print contents). Since you can easily #[derive(Debug)] on any struct and get a default implementation as long as all struct members also have Debug. Then you have an easy way to print it in logs or (I assume) write it to debugfs.

So I was just wondering if there is a specific reason we don’t have these - if not, I can probably send an RFC once I’m back.

Trevor Gross (Sep 10 2024 at 11:30):

There is no reason they would wind up in the final binary right? I think the convenience is easily worth some increase in intermediate library size, or just mark them inline if that is really a problem

bjorn3 (Sep 10 2024 at 15:29):

All public symbols of the kernel crate are exported and thus end up in vmlinux without an option to garbage collect them. Since rustc 1.75 all #[derive(Debug)] do get their Debug impl marked as #[inline] and thus only gets codegened when actually called, so adding #[derive(Debug)] shouldn't cause any bloat outside of the rmeta files necessary to compile kernel modules.
