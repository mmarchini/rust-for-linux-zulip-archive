Alice Ryhl (Apr 19 2024 at 07:18):

It looks like SGTable::alloc_table is the real constructor, not the new function. Generally, the goal should be that safe code can't use your struct wrong, but it looks like bad things will happen if you try to use the SGTable before you call alloc. So you probably want to use pin-init to have a constructor for SGTable where you call sg_alloc_table inside the constructor.

Alice Ryhl (Apr 19 2024 at 07:35):

Such a constructor would probably look something along these lines:

use crate::alloc::Flags;

#[repr(transparent)]
pub struct SGTable {
    inner: Opaque<bindings::sg_table>,
}

impl SGTable {
    pub fn new(nents: u32, flags: Flags) -> impl PinInit<Self, Error> {
        try_pin_init!(SGTable {
            inner <- Opaque::try_ffi_init(|slot| {
                // SAFETY: `slot` is a valid pointer that can hold an
                // `sg_table`. If this call returns an error, we do not use the
                // `slot` anymore.
                //
                // If it returns success, then since this type is pinned, `slot`
                // will hold an `sg_table` until the destructor of `SGTable`
                // runs, at which point we destroy the sg table before the
                // address becomes invalid.
                let ret = unsafe {
                    bindings::sg_alloc_table(slot, nents, flags.as_raw())
                };

                to_result(ret)
            })
        })
    }
}

Some notes:

  • The Flags parameter is from the recent Allocation API patchset, which is now in rust-next and will be merged for the next kernel version.
  • The safety comment should explain why the unsafe block is okay. C doesn't explicitly document what the method requires, but I reasoned that it's okay as long as:
    • The pointer is okay for writing an sg table to.
    • If sg_alloc_table fails, then we do not need to clean it up.
    • If sg_table_alloc succeeds, then we must ensure that we call sg_free_table before the address is freed or otherwise reused. Pinning guarantees that the destructor runs before the address is freed or reused.
  • to_result is a convenience method in the error module for converting an integer to an error.
  • It looks like Opaque::try_ffi_init doesn't exist yet. I guess nobody has used Opaque with a type that has a fallible constructor yet. You will need to add it, I'm happy to show you in the right direction there as well.

Alice Ryhl (Apr 24 2024 at 12:33):

It is awkward that sg_table is both long-lived and pinned for this. Typically, the way you model this kind of thing is that you have two types (e.g., UnmappedSgTable and MappedSgTable), and whenever you call dev_map or dev_unmap, then this takes a table of one type and returns one of the other. But if the table is a pinned field of a long-lived struct, then this design does not work, because it requires that you can "take" the table out of the struct, and put it back afterwards once you have the other type.

Alice Ryhl (Apr 24 2024 at 12:35):

One thing you can do is to have the sg table in its own allocation and then the two types can carry around owned pointers to it. But this prevents you from storing the sg table inline in another struct.

Alice Ryhl (Apr 24 2024 at 12:36):

Another possibility is that dma_map and dma_unmap are unsafe functions and require the caller to not mess it up. Then you could have a safe wrapper on top of it. For example, store a boolean next to it keeping track of whether it is mapped, and use the boolean to prevent incorrect calls (and use it in the destructor too to unmap first before cleanup).
