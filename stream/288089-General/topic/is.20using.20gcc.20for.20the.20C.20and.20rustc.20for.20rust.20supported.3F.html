Yeah, LLVM=1 implies CC=clang among others.

I'm not sure that will be always be doable. Is there any known problems (assume I don't care about LTO) with using gcc for C and rustc for rust? is this documented anywhere?

It "works", but it is very hacky. We have it as a proof-of-concept, but we are waiting for a better approach involving rustc_codegen_gcc and/or GCC Rust. Until then, we don't recommend distributions that use GCC to enable Rust (assuming they have a module they want to use), though one actually did in the past (non-LTS Ubuntu).

https://rust-for-linux.com/rustc_codegen_gcc
https://rust-for-linux.com/gccrs
https://docs.kernel.org/rust/quick-start.html#building

Andreas Hindborg (May 29 2024 at 08:45):

You would probably also loose the option to inline C helpers into Rust code. @Gary Guo has a patch in the works for that. We have seen 3-5 % performance difference in binder and null block for that feature.

Janne Grunau (May 30 2024 at 16:40):

The Fedora Asahi Remix kernel with the rust asahi GPU driver uses gcc for the C parts. We haven't encountered any issues with that combination as far as I'm aware.
