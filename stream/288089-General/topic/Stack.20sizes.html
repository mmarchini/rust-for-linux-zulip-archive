Gary Guo said:

You can always create an uninitialised box and initialise it unsafely. Sure it's not very elegant and require unsafe, but from my impression in the kangrejos workshop, kernel people seems not very repellent to the usage of unsafe compared to Rust people.

I'm happy with the crazy macro solution for now because it solves the safety issue (by requiring that all fields are initialized) and keeps normal syntax, but I would not be happy if there were no way to do that. It's not about avoiding unsafe code altogether, but rather about keeping unsafe code in "core" code or abstractions, and have the bulk of the logic be safe. I have to deal with a lot of complex structures, and one of the big advantages of Rust is that I was able to build an object system that uses the Rust ownership and lifetime rules to enforce the relationships between those structures, not just for safety on the kernel side, but also to keep things consistent with GPU firmware. If I had to use open-coded unsafe code every time I initialize a struct, that would be ugly and also remove a lot of the advantages of Rust here, since there would be many more places in the code that could have a mistake...

For context, the GPU I'm writing a driver has an unfortunate design quirk that if we crash its firmware, the only way to recover is rebooting the entire machine (I couldn't find a way around this, it has to do with the firmware being loaded by the bootloader). So any driver bug that causes the structure lifetimes to be managed wrong, even one that does not cause any unsoundness from the kernel's point of view, is as bad as a kernel panic since it forces users to reboot. That's one big reason why writing this in Rust is beneficial, because it helps get both of those things right. And that relies on the core object system being sound, so that the rest of the driver can safely work with all these structures and have the GPU pointer validity and lifetimes checked at compile time.

To give an example: we have to support multiple firmware versions, and some of them add new pointers. If structure initialization didn't have static checking that all fields are initialized, then adding a new pointer field for a firmware version change would mean that all initializers of that struct are suddenly UB (at least on paper - in practice probably not since these aren't real pointers, but it would still be incorrect and crash the firmware), until we go through and check all of them. That's a really big downgrade from Rust's usual safety guarantees. So I really think being able to do "placement new" in Rust is critical for these kinds of use cases. Right now we can get most of the way there with macros, so that's okay, but I hope the language gets that feature some day!

Alice Ryhl (Sep 12 2022 at 08:41):

How about using a macro like this? It only compiles if all fields are mentioned.

macro_rules! box_and_init {
    ($type:ident { $( $(#[$attr:meta])* $field:ident: $val:expr),* $(,)? }) => {
        {
            #[allow(unreachable_code)]
            fn _field_check() {
                let _: $type = $type {
                    $(
                        $( #[$attr] )*
                        $field: todo!(),
                    )*
                };
            }

            match Box::<$type>::try_new_uninit() {
                Err(e) => Err(e),
                Ok(v) => unsafe {
                    let ptr = Box::into_raw(v) as *mut $type;
                    $(
                        $( #[$attr] )*
                        core::ptr::addr_of_mut!((*ptr).$field).write($val);
                    )*
                    Ok(Box::from_raw(ptr))
                }
            }
        }
    };
}

#[derive(Debug)]
struct Foo {
    a: i32,
    b: i32,
    c: i32,
}

fn main() {
    let x = box_and_init!(Foo {
        a: 10,
        b: 30,
        c: 45,
    }).unwrap();

    println!("{:?}", x);
}

Alice Ryhl (Sep 12 2022 at 08:47):

This macro leaks memory if any of the $val expressions contain a return, but that can be fixed using some structs with destructors.

Gary Guo (Sep 12 2022 at 12:13):

Alice Ryhl said:

What if you allocate the memory before creating the stack temporary? I've seen people claim that the allocation failure path is a common cause for it being spilled on the stack.

let uninit_box = Box::try_new_uninit()?;
*uninit_box = MaybeUninit::new(MyStruct { ... });
let my_box = uninit_box.assume_init();

This is a very good point. Box::new gets some extra optimisation from LLVM because it is assumed that allocation will succeed, so the construction of the struct can be moved after the allocation. This matters a lot especially if any of the field (or the whole struct) contains destructor. Moving the construction of things with dtor after Box::try_new will change the semantics and therefore not allowed.

BTW, you can do this totally safely:

let uninit_box = Box::try_new_uninit()?;
let my_box = Box::write(uninit_box, MyStruct { ... });
