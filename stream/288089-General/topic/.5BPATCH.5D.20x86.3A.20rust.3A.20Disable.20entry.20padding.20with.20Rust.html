Matthew Maurer (Jan 22 2024 at 15:24):

The problem is that this does not just affect ftrace, it affects how the kernel hotpatching system deals with preludes. For example, KCFI is broken if the kernel tries to enable both KCFI and pachable-function-entry with Rust, because the KCFI header doesn't get moved the way the kernel expects. There are similar results for IBT.

Additional patches to prevent other architectures from using this seems fine, I was just fixing the one I knew was broken and could test.

Re: Exempting from ftrace, that is approximately the behavior Rust gets from ftrace today - patchable-function-entry creates a section of function addresses that can be traced this way, and Rust doesn't end up in them. The brokenness isn't coming from ftrace, that's just another nice thing that might start working once we have this fixed.
