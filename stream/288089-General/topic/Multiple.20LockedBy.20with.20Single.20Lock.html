Topic: Multiple LockedBy with Single Lock

David Rheinsberg (Mar 12 2026 at 13:21):

Is there a general recommendation how to use LockedBy::access_mut() on multiple objects simultaneously, protected by the same lock? access_mut() takes a mutable reference to ensure safe access, but I obviously cannot provide the same reference to multiple calls at the same time.

I have a patch (see below) to reduce the requirement to just providing the mutable reference once, without borrowing it for the entire lifetime (obviously making the interface unsafe). This makes it very flexible to use.

But I would also be open to other suggestions like LockedBy::access_mut_multiple(v: [&'a LockedBy<...>; N], owner: &'a mut U) -> [&'a mut T; N].

Let me know whether you have better ideas! Or whether I missed something.

We could also use field projections from @Benno Lossin to make LockedBy generic over a field underneath the lock. Then split-borrows would be able to avoid sharing a mutable reference. But it does not solve the problem where one has to lock n instances of the same object underneath a single lock.

Subject: [PATCH] rust/sync: add LockedBy::access_mut_unchecked()

Add a new accessor to `LockedBy`, which allows getting mutable access
without mutably borrowing the owning object.

This is particularly useful when having to lock multiple objects under
different instances of `LockedBy`, but protected by the same lock. In
those cases, the caller needs to retain access to the mutable reference,
so it can continue calling `access_mut()` on the other instances.

It is now up to the caller to ensure the provided reference is held long
enough, and a single `LockedBy` instance is not accessed multiple times.

Signed-off-by: David Rheinsberg <david@readahead.eu>

+    /// Returns a mutable reference to the protected data when the caller
+    /// provides evidence (via a mutable reference) that the owner is locked.
+    ///
+    /// Unlike [`Self::access_mut()`] this does not require the mutable
+    /// reference to be borrowed for the requested lifetime, and thus multiple
+    /// different [`LockedBy`] objects can be acquired simultaneously.
+    ///
+    /// # Panics
+    ///
+    /// Panics if `owner` is different from the data protected by the lock used
+    /// in [`new`](LockedBy::new).
+    ///
+    /// # Safety
+    ///
+    /// The caller must hold `owner` for `'a` and must not call into this
+    /// function more than once under the same lifetime `'a`.
+    pub fn access_mut_unchecked<'a>(&'a self, owner: &mut U) -> &'a mut T {
+        build_assert!(
+            size_of::<U>() > 0,
+            "`U` cannot be a ZST because `owner` wouldn't be unique"
+        );
+        if !ptr::eq(owner, self.owner) {
+            panic!("mismatched owners");
+        }
+
+        // SAFETY: `owner` is evidence that there is only one reference to the owner.
+        unsafe { &mut *self.data.get() }
+    }
 }

David Rheinsberg (Mar 12 2026 at 13:52):

Benno Lossin said:

I think it would be sad if we used unsafe for this. the access_mut_multiple sounds like a good idea, but you probably want to use a tuple of &'a LockedBy<...> instead of an array to allow different contents between the locks.

I don't think field projection play a role in this, but maybe I misunderstood you.

Think of something like this:

struct PlatformDev {
    inner: Mutex<PlatformInner>,
}

struct PlatformInner {
    sound: Arc<SoundDev>,
    input: Arc<InputDev>,
}

struct SoundDev {
    inner: LockedBy<SoundInner, PlatformInner>,
}

struct InputDev {
    inner: LockedBy<InputInner, PlatformInner>,
}

Imagine you want to lock both PlatformInner.{sound, input}.inner at the same time. You cannot do that, since both would need &mut PlatformInner. But you could instead use field-projections to say either only needs access to a field in PlatformInner, like this:

struct SoundDev {
    inner: LockedBy<SoundInner, PlatformInner, field_of!(PlatformInner, sound)>,
}

struct InputDev {
    inner: LockedBy<InputInner, PlatformInner, field_of!(PlatformInner, input)>,
}

Now you can lock both at the same time. LockedBy can verify that you hold a reference to a field of the correct locked PlatformInner.

Maybe a bit over the top, but I have run into situations like this multiple times.
