Topic: Pin<_> in method signature

Boqun Feng (Mar 03 2023 at 18:09):

I'd like to have a discussion about using (or not) Pin<_> in method signature, hopefully we can reach out to a conclusion and document it somewhere.

My understanding of Pin is largely based on the following conceptual model:

  • "pinned" is a non-reversible state of a memory object, once pinned, stay pinned until drop
  • Of course moving a pinned !Unpin object is UB, no matter the move itself is a safe API or not.

, and comes to Pin<>:

  • Having a Pin<_> proves the object is pinned.
  • Not having a Pin<_> (i.e. having a normal & or &mut or DerefMut) CANNOT prove the object is not pinned. The object may be pinned previously but the Pin<_> pointer is dropped.

Now if the above conceptual model is right, having a Pin<_> in the method signature means at least two things (for the sake of ease, let's say the object is !Unpin):

  1. Don't move inside:

    • the method won't move the object inside, since the method was only provides a Pin<_> there is no safe way to move the object. This is especially useful for trait (e.g. Future::polll()) or generic function.

  2. Don't move outside:

    • after the method called, the moving the object is UB, since Pin<_> proves the object was pinned when the method called. And moving something after pinned is UB.

The above is good enough to guide myself for whether use Pin<_> or not if I design something. Does it sound reasonable? (I do have some follow-ups, but in case I'm wrong, let me pause here for a while ;-))

Boqun Feng (Mar 06 2023 at 04:49):

Below is the current version I'm considering to add as some sort of guidelines for using Pin. I probably open a Github issue first for boarder discussion before I send a patch. It's still need some polishing, but I want to see whether we can get on the same page.

When do you need the function to take Pin<_> as the argument? If any of the following is true:

  • The function expects the object pinned (i.e. the object should have a fixed address), this is a sign of starting relying on pin safety guarantee.
    • There are rare cases that the pin safety guarantee (once pinned, always pinned) can be broken by users, if these are allowed, see UNPIN comments below
  • The function is a mutable one (i.e. it would take a &mut self) and may be called after the object is pinned. Reason: otherwise users may need to use unsafe to call these function.
    • What about the function may be called before pinned? A duplicate &mut self version or rely on pin-init so all mutable functions take Pin<&mut Self> ???

UNPIN comments:

  • Put an # Unpin comments on the functions that make the object movable again (e.g. removing the object from a global list), notes these functions themselves have to be a Pin<_> function. For example,
struct A {
    ...
    _pin: PhantomPinned,
}

impl A {
    /// Puts on a list for execution.
    pub fn queue(self: Pin<&mut Self>)
    /// # Unpin
    ///
    /// Removes the object from the global list when return,
    /// can be moved after this.
    pub fn wait_for_finish(self: Pin<&mut Self>)
}

Then if these # Unpin functions are used to make the objects movable. Add a UNSOUND: UNPIN: comment as the move point, e.g.

let a = A::new(..);

// UNSOUND: SAFETY: `a` may be moved afterwards, but that's after
// a Unpin function.
let pin = unsafe { Pin::new_unchecked(&mut a)};

pin.as_mut().queue();

// UNSOUND: UNPIN: `wait_for_finish()` makes `a` movable again.
pin.wait_for_finish();

// UNSOUND: Moves `a`.
let b = Box::new(a);
