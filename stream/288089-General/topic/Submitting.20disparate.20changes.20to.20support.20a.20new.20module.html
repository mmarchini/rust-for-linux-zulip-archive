Miguel Ojeda (Feb 14 2026 at 02:06):

Essentially, most things should go through the relevant subsystems (i.e. the usual ones you would have normally send it to if it was written in C).

So even if your changes are within rust/, in most cases they would still go through another subsystem. For instance, even the module bits should likely go through "MODULE SUPPORT".

There are exceptions, e.g. sometimes we may act as a fallback, and sometimes different arrangements may be made, but in general, the above is what applies in most cases.

Please take a look at some entries around MAINTAINERS that have rust/ in their F: fields. Also, in case you didn't see it, I have some notes at https://rust-for-linux.com/contributing#the-rust-subsystem.

I hope that helps!

Miguel Ojeda (Feb 14 2026 at 22:27):

I am a little confused by the GitHub merges though, do those end up in rust-next or will they still need to go through the mailing list?

PRs in GitHub (including those linked in the lore.kernel.org message I linked above) are (most of them) for work that happened before Rust was merged into the kernel. The reason is that, when Rust support was merged, we were asked to remove most of the code we had, and instead upstream it piece by piece when users appeared. So some code is still out there that was never needed yet. I have some more context at https://rust-for-linux.com/past-branches#rust.

In other words, in the thread I linked above from lore.kernel.org, I was mostly referencing that old work that happened in the past, and that was being upstream by Andreas. So I was giving Andreas the references to the original authors/work so that they could ideally be included in the Git log.

Nowadays all reviews happen in the mailing list, and that issue of "old things in GitHub PRs" almost never happens anymore, since most of the basic bits were already needed one way or the other by the first major users in the upstream kernel.

Miguel Ojeda (Feb 14 2026 at 22:33):

In summary, you should not submit things via PRs, and the existing PRs are usually very old work. However, if you know about work that overlaps those PRs, please try to reference the old work (both for giving credit to original contributors and as a reference/comparison if you decided on a different approach etc.).
