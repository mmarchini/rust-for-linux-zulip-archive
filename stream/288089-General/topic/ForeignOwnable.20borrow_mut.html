It looks to me like this might have some issues in light of core::mem::replace()/swap()/take().
If you replace the Arc<T> that is behind the &mut you can obtain through borrow_mut(), the Arc that was there will be dropped. Refcount might go to zero and leave a dangling pointer in C land.

Further, when ScopeGuard guarding the &mut is dropped, it will into_foreign() the new Arc<T> and it will leak.

This is not the intention as far as I know. I don't think we considered this.

Andreas Hindborg (Jul 04 2023 at 17:29):

    /// Search for `key` in the map. Return a mutable reference to the
    /// associated value if found.
    pub fn get_mut(&mut self, key: Key) -> Option<MutBorrow<'_, V>> {
        let item =
            core::ptr::NonNull::new(unsafe { bindings::radix_tree_lookup(self.tree.get(), key) })?;

        // SAFETY: `item` was created by a call to
        // `ForeignOwnable::into_foreign()`. As `get()` takes a `&self` and
        // `remove()` takes a `&mut self`, no borrows for `item` can exist and
        // `ForeignOwnable::from_foreign()` cannot be called until this borrow
        // is dropped.
        Some(MutBorrow {
            guard: unsafe { V::borrow_mut(item.as_ptr()) },
            _marker: core::marker::PhantomData,
        })
    }

Maybe "borrow" is the wrong part of the name.

If I remember correctly, the usage is during a series of callbacks, the foreign owned data needs to be accessed multiple times (so manually from_foreign and into_foreign is a bit annoying), and some of the callback implementation may modify the underlying data, so a deref to &mut ForeignOwnable is returned, of course that only works if Box<..> is used, but the different implementations may use either Arc or Box, and having a borrow_mut makes Box case and Arc case share the generic callback handling code.

For example: https://github.com/Rust-for-Linux/linux/blob/rust/rust/kernel/fs.rs#L139

Alice Ryhl (Jul 06 2023 at 08:33):

A had some interesting thoughts regarding borrow_mut and xarray. See:

This probably depends on changes to borrow_mut. I think ideally we want the trait to have a BorrowedMut type that represents the kind of access you can get to the inner value given mutable access to the T, then return Option<T::BorrowedMut<'a>>.

For an Arc, the BorrowedMut type would still be ArcBorrow, but that actually still gives this extra powers over the two other methods. It lets you access the Arcs contents without modifying the refcount, and without a closure.

And of course, with a Box, the BorrowedMut type would be &mut T, so this type would return Option<&mut T> when the pointer type is Box.

<https://github.com/Rust-for-Linux/linux/pull/1019#discussion_r1254112511>
