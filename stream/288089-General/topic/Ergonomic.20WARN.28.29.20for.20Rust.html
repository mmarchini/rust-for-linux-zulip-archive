Topic: Ergonomic WARN() for Rust

Asahi Lina (May 02 2023 at 12:57):

Right now Rust can ergonomically propagate errors silently (? operator), or noisily oops (.unwrap(), .expect()). However, in the kernel, we often have situations where certain conditions indicate a bug, but we don't want to oops since that would guarantee a crash. That's what the C WARN() macro is for, but it's not abstracted yet.

Since Rust has excellent error handling, we want to propagate as much as possible, but we also want to be noisy when "impossible" things happen. Right now there is no ergonomic way to do that. At most you can use if is_err() { warn(...) } or map_err(|_| warn(...)) but those aren't great...

How about adding a WARN equivalent to expect()? It could be implemented both as a trait and as a macro:

macro_rules! require {
    ($($i:tt)*) => {
        ($($i)*).map_err(|_| {
            println!("WARNING! {}:{}", file!(), line!()); // Would be bindings::warn_slowpath_fmt() or whatever
        })
    };
}

impl<T, E> Require for Result<T, E> {
    fn require(self, msg: &str) -> Self {
        if self.is_err() {
            println!("WARNING! {}", msg); // Would be bindings::warn_slowpath_fmt() or whatever
        }
        self
    }
}

trait Require {
    fn require(self, msg: &str) -> Self;
}

fn bar() -> Result<(), ()> {
    let a = foo().require("expected something")?;
    let b = require!(foo())?;

    Ok(())
}

The trait option looks more ergonomic and directly mirrors expect(), which is nice. The downside is that since it's not a macro, it can't capture file and line information (which in principle the C backend for WARN wants, though we could just let that default to something). The macro option can capture file and line information, so it's more suitable for cases without a message where the file and line information is the only thing leading you directly to the line of code it came from. Personally I would kind of want to have both (method version with a custom message and no file:line, and macro version with no message but automatic file:line)...

Any thoughts on the idea and the name?

Asahi Lina (May 03 2023 at 04:32):

The WARN implementation on most architectures uses assembly to build up a data structure in a special section, and then reference it from the warn location. That data structure needs a pointer to the file string (which can be done with asm sym), and a literal line number which needs asm const.

Asahi Lina (May 03 2023 at 04:33):

You'd use file!() to create a static string with the filename, and then reference it via asm sym, and then use line!() as an argument to asm const.

Boqun Feng (May 03 2023 at 05:14):

I think Lina's point is most archs implement WARN()/BUG() as some sort of trapping (e.g. on ARM64 is brk instruction), and in order to pass the pointer to file!() string literal, an asm section is used per WARN()/BUG() "callsite", since that's how to pass parameters to trapping.
