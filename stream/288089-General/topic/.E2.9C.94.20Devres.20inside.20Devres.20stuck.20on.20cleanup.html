Topic: ✔ Devres inside Devres stuck on cleanup

Markus Probst (Jan 31 2026 at 22:45):

If we put a Devres inside another Devres, it gets stuck on the cleanup (waits indefinitely).

Steps to reproduce:

Add

kernel::devres::register(
    pdev.as_ref(),
    kernel::devres::Devres::new::<Error>(pdev.as_ref(), Ok::<(), Error>(())),
    GFP_KERNEL,
)?;

to the probe function in samples/rust/rust_driver_platform.rs.

Follow the instructions on the top of samples/rust/rust_driver_platform.rs to run the sample driver in qemu.

Run echo LNUXBEEF:00 > /sys/bus/platform/drivers/rust_driver_platform/unbind inside qemu.

This process never returns.

Explanation

On cleanup, the outer Devres starts to drop the inner Devres.
The inner Devres is unable to remove the devm action from itself, because the cleanup is running.
It waits for self.inner().devm.wait_for_completion() under the assumption that the devres callback will be run.
(See https://github.com/Rust-for-Linux/linux/blob/08afcc38a64cec3d6065b90391afebfde686a69a/rust/kernel/devres.rs#L305 )
This never happens, as it waits for the cleanup of the outer Devres to finish.

I have not come up with a solution yet.
The issue is, if we let the Drop proceed, the devm_action is still there and will try to perform the cleanup on an already dropped Devres.
