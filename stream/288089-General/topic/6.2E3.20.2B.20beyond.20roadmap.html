Andreas Hindborg (Mar 13 2023 at 09:06):

Demi Obenour said:

Miguel Ojeda said:

Yeah, some of the drivers will be upstreamed, and we have been discussing which abstractions to submit next so that the modules can follow afterwards.

What advantage does the Rust NVMe driver have over its C counterpart? Presumably it is secure against malicious NVMe devices, which would be useful in confidential computing environments. Anything else?

First I would like to state that the Rust NVMe driver is not currently intended to replace the C version. Right now it is primarily a feasibility study to show that Rust is suitable for high performance drivers such as NVMe. It helps us look into design of ergonomic wrappers for the C APIs of the block layer.

I do not think that Rust will help us significantly against malicious devices in general. If we trust data delivered from a peripheral, we have to verify no matter what language we use. One thing we get with Rust is array bound checks, so there is that. In the case of devices that can access host memory directly, we have to rely on hardware mechanisms (IOMMU) to prevent access to unauthorized host memory. And we have to be very careful about how we use the data delivered by the device. Also, I don't think malicious NVMe devices is a very big threat. Even in data center environments with NVMe over Fabrics, people tend to connect to trusted resources. There are easier and cheaper ways to get into peoples systems. But if one were to compromise an NVMe drive, it would be a really good place to hide.

The question of advantages of the Rust NVMe driver over the C implementation is really a generic question about Rust vs C in general. There are tonnes of discussion and blogs posts on the topic. In general I would say that Rust prevents a large amount of undefined behavior that is allowed in C, and it makes some other problems more easy to avoid. Quantitative data for the effect of this is becoming publicly available, such as Google correlating drops in vulnerability count and severity with adoption of Rust in Android for system programming https://security.googleblog.com/2022/12/memory-safe-languages-in-android-13.html . I think that this alone is enough to justify using Rust rather than C.

Besides avoiding undefined behavior, there is another aspect which is arguably much more subjective, and that is the developer experience. I think Rust is easier to read, easier to discover, and easier to maintain. It is easier to construct reusable code with Rust. The tooling around the language is much better than the tooling for C (lints, IDE support). In my opinion, the sum of all these subjective elements is greater than the parts. For me, reading and understanding a driver written in Rust is much easier than one written in C. I think it is also easier to implement a driver in Rust against ergonomic wrappers than it is to implement one in C. The Rust types used in the Rust APIs convey a lot more information than what you often get with a void* ctx in a C API.

What advantage does the Rust NVMe driver have over its C counterpart? Presumably it is secure against malicious NVMe devices, which would be useful in confidential computing environments. Anything else?

A good way to look at this sort of thing (imho) is that code writtin in Rust will be equivalent to correct code written in C. That is, it can't give you anything that you couldn't write in C, and kernel contributors/reviwers are usually quite good programmers. However, (safe) Rust ensures that you wind up with that correct code every time it compiles, so it's just easier to not introduce vulnerabilities. Which isn't the same as eliminating vulnerabilities that somehow couldn't be eliminated in C.

First of all it is not ready for upstreaming yet. Second, there seems to be no interest for this from the block layer maintainers. There is already a very good NVMe driver in the tree that is in active use and is well maintained. The current maintainers of the block layer and of the C NVMe driver is not interested in having another NVMe driver in the tree, and for reasons that I do not understand, they really do not want a driver written in Rust. These are my observations. Better ask the actual maintainers directly if you need clarification :)
