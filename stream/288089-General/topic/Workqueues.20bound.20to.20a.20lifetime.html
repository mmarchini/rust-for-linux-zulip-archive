Topic: Workqueues bound to a lifetime

Alice Ryhl (Jun 04 2026 at 15:06):

I'm thinking about how to work in workqueues with the new lifetime model. Let's say we have device private data that looks like this:

#[pin_data]
struct MyGpuPrivateData {
    wq: OwnedWorkqueue<'foo>,
    foo: Foo,
}

and that Workqueue::<'a>::spawn accepts any closure that is 'a. (Note contravariance in 'a.) Then we want to be able to do this:

impl MyGpuPrivateData {
    fn spawn_work(&self) -> Result {
        self.wq.spawn(|| {
            // ok because `wq` borrows from `foo`
            println!("{}", self.foo);
        });
    }
}

But my understanding is that the self-referential pin-init stuff only lets you create such self-referential values when constructing MyGpuPrivateData and not after the fact.

Maybe you can do this by stashing a &Workqueue elsewhere in MyGpuPrivateData?

Thoughts?

Benno Lossin (Jun 04 2026 at 15:24):

Benno Lossin said:

I feel like this needs more than just FP, since we have no way of naming the 'foo lifetime

  • if we had a places as lifetimes, we could have self: &Self -> &'{self.foo} Foo, which the borrow checker could permit using in a closure that's spawned on the workqueue.
  • however that signature is only valid if foo is not dropped before the workqueue is finished. currently it seems to me that this is solved by dropping the workqueue first and then foo.

Because of this, std can't improve upon the output lifetime of borrowing unless it also supports self-referential types directly (and then it can check that foo is only ever dropped after wq).
