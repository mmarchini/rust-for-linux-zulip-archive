Danilo Krummrich (Nov 29 2025 at 05:45):

(@Daniel Almeida , @Minkyu Jung , Cc: @Alexandre Courbot )

Yeah, by itself I don't see a huge advantage in a new type for DmaAddr.

However - and I think I proposed this in some other context already - I think it would be great to have a DmaRange type that captures both, the base DmaAddr and a usize for the length of the DMA mapped region.

In this context a DmaAddr type that is just an opaque token would make much more sense, while the DmaRange can provide math methods including boundary checks.

Currently, drivers just do arbitrary calculations with DmaAddr taking care of boundary checks themselves, which is not that great.

Minkyu Jung (Dec 02 2025 at 08:07):

That's a great point about DmaRange. But I still see the value in DmaAddr as a newtype.

Since DMA requires a valid mapping when using IOMMU, a DmaAddr type serves as proof that the mapping has actually occurred (e.g., iommu_dma_map_sg).
The newtype can encode the invariant that the address is safe to use for hardware I/O, which a raw integer or a simple token cannot guarantee.

Alice Ryhl (Dec 02 2025 at 08:33):

A type that serves as proof that the mapping occurred is a much more complex proposition. After all, how do you ensure that the DmaAddr is not kept around after the mapping is removed?

Minkyu Jung (Dec 02 2025 at 15:33):

Ideally, DmaAddress could capture the lifetime of the mapping object (e.g., CoherentAllocation<T>) with PhantomData to ensure liveness, since dma unmapping only happens on Drop in the current rust api.

However, even as a simple opaque type without lifetimes, it can prevent mistakes that pass PhysAddr without mapping it.
