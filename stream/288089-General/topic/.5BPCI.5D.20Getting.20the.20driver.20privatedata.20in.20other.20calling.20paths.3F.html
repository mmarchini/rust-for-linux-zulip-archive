Danilo Krummrich (Oct 31 2025 at 14:14):

Indeed, what Alice points out works and is technically correct to obtain the driver's bus device private data.

However, the question here is whether you actually want the drivers's bus device private data or whether you should leverage the private data of the class device you receive the callback from, fwctl in this case.

The driver's device private data should only be used in the context of the driver model (i.e. bus callbacks, driver to driver interaction, etc.). If you are within another scope, e.g. file operations, class device callbacks, etc. you should use the private data from those.

Danilo Krummrich (Oct 31 2025 at 14:15):

If there is data structures needed in both the private data of the device and the private data of something else, then this sounds like shared ownership and you may want to consider reference counting with Arc.

Danilo Krummrich (Oct 31 2025 at 14:17):

Regarding a mutable reference to the driver's private data, you won't ever get one since it can be accessed from multiple tasks concurrently. That means, if you need to modify a member within, you have to protect it with a lock, which if held will give you mutable access. In other words, this is the Rust compiler preventing you from implementing race conditions. :)
