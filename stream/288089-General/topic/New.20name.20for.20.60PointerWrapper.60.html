Topic: New name for `PointerWrapper`

Wedson Almeida Filho (Jan 03 2023 at 04:56):

Hey folks, now that we're slowly upstreaming our changes, I'd like to hear opinions on renaming PointerWrapper.

When I first introduced it over two years ago, it was basically just wrapping a pointer that was returned by open, used in a bunch of file callbacks, then given back to release.

Now we rightly use it to transfer ownership to C of objects that have a pointer-sized representation. While C owns it, it can be borrowed (and different original objects have different borrowed representations, GATs held us here!), mutably borrowed, and eventually can be given back to Rust.

So I was thinking of something like COwnable -- prefixing something with C isn't unprecedented (see CStr). But I'm not completely sold on the name. Any suggestions? Comments?

Wedson Almeida Filho (Jan 05 2023 at 17:28):

The trait has to do with ownership because Rust gives up ownership to some C code, which is responsible for eventually giving ownership back -- if this doesn't happen, we have a leak. The C code also owns in the sense that Rust doesn't have any more references to it, so the C code may, for example, get mutable references to the object as long as it knows there are no other references on the C side alone (as the Rust side doesn't).

So saying that the object is representable with a usize is correct, but I feel is not a sufficient description of the trait.

The reason I liked FFIOwnable was that the it conveys the idea of ownership plus that the object needs to representable by something that is understood/compatible with FFI (implicitly a usize at the moment, things may change if/when we adopt CHERI)

Wedson Almeida Filho (Jan 05 2023 at 19:52):

I see your argument but I'm not sure I agree that one needs to access or free the object to be considered the "owner". Though I'm open to being convinced otherwise :)

Let's look at the concrete struct file example: the open method returns a Result<T> where T: PointerWrapper. While the file is referenced, other methods on the trait (e.g., read/write/seek) are called with a <T as PointerWrapper>::Borrowed type (because they can be called concurrently, so this is a shared reference); eventually when the C side drops the file refcount to zero, it calls the release method, which takes a T as argument. The C side of course just stores an opaque value in file::private_data, but it manages the lifetime of that instance of T (handing ownership back to Rust when file is about to be freed). Doesn't it make sense to say that C owns the object despite not accessing it directly?
