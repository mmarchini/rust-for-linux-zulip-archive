Topic: Unstable features, compiler versions, and `alloc`

marcan (Nov 25 2023 at 07:50):

Hi Rust folks, you probably know me as the guy maintaining the Asahi Linux kernel tree and merging Lina's Rust GPU driver branch. I wanted to ask about the long-term plan for reducing unstable feature usage so that the compiler version can eventually be unpinned.

Over the past few kernel versions, in the Asahi tree, we've been adding compiler forward compatibility version bumps ahead of mainline. Instead of doing this by pulling in all the alloc stuff wholesale, we make the minimum number of changes to make things work while keeping backwards compatibility (with cfg(version()) conditionals). We do this to ease the job of distributions and users, so they don't have to pin old Rust versions all the time (with varying degrees of success). So we have a pretty good idea of what has been breaking and what hasn't, with version bumps.

I understand that there is a good plan to eventually whittle down the unstable feature list used in the kernel crate and general driver crates. That part doesn't worry me. However, I'm afraid about the elephant in the room. These are the unstable feature counts used in our tree right now:

  • kernel crate: 11 features
  • Driver/user crates: 4 features
  • alloc crate: 105 features

As you might expect, this makes alloc the biggest problem when dealing with forwards compatibility with compiler versions. Digging back through kernel tags past, here's a bit of an overview:

  • In 6.3 I added a hack to support Rust 1.70 (up from 1.66). That only affected alloc
  • In 6.4 I bumped that to 1.71. 84% of the patch by line count was hacking up alloc (the rest was fairly trivial changes to buildsystem/kernel stuff)
  • In 6.5 Lina took care of porting to 1.72. That was, again, alloc only.
  • In 6.6, we had a big issue with const_trait_impl going away in 1.73. That affected the OF platform device stuff. Lina figured out a workaround so we could drop that feature. The rest: more alloc.

So... what's the plan for alloc? As part of the standard library, it uses unstable features with impunity, and that makes it intrinsically tied to compiler version. I haven't seen any discussion of what the plan is there. As long as Linux is vendoring alloc and attempting to keep it in sync with upstream, I don't see how we can ever move away from pinned compiler versions, regardless of whether the feature count for kernel and the rest drops to zero.

Supporting unpinned compiler versions is very important to make life easy for distributions and to be able to support Rust-enabled builds without undue pain for everyone involved. Enabling RfL when you can compile with regular unpinned rustc is just a few package dependencies. Enabling RfL with a pinned compiler version requires distros to fork off the rust package into something specific for the kernel only, and then you have to deal with being unable to co-install it (we do that in Fedora Asahi with the rust-kernel package, it does the job, but it sucks that it's either or, you can't install it together with regular rust).

Have I missed something? Is there a long-term plan for alloc?

Neal Gompa (Nov 25 2023 at 08:55):

Andreas Hindborg said:

I think the plan is that eventually we will not have to patch alloc

I don't think that is going to be likely. As @marcan stated, the alloc crate being part of the Rust standard library means that they are free to use unstable features. Realistically, our choices are the following:

  • Fork alloc completely and gut it of what we don't need and stabilize the rest
  • Vendor multiple versions of alloc and using that to define a supported rust version range
  • Drop alloc and replace it with something custom for the Linux kernel

None of these are particularly awesome choices, but I suspect that the first or last options are more viable than the second one.

Boqun Feng (Nov 25 2023 at 17:57):

This may be an unpopular opinion, but I think it's too early to talk about avoiding using unstable features (even just in kernel crate), mostly due to no evidence that the Rust language is mature enough for kernel development. In other words, the unstable features wanted by kernel will increase, IMHO.

I understand that's a pain for everyone working on a stable version of kernel if the upstream rustc version keeps bumping, but replacing an unstable feature with a workaround might not be a win, most of the time is a tech debt. Looking at C side, we use a few compiler extensions, which will never be stabilized in C standard. But it's just fine, because we have some confidence that their interface are stable. Maybe we can try the similar approach for Rust, but it would require the support from Rust community, namely, when we pick up an unstable feature, we can know whether its interface is (relatively) stable or not (we kinda already do this), and when an unstable feature used by us gets changed, we are looped-in to see how to handle that (or they just use our usage as a test case).

Neal Gompa (Nov 26 2023 at 11:42):

Well, as long as you don't find a reason to need both at once, it should be okay. But the issue with it is that it's a question of maintenance. Having two packages of the compiler means you need to do security maintenance for two code streams of the Rust compiler. And with the Linux kernel especially, being stuck on older Rust versions can be dangerous. Additionally, upstream Rust is fairly aggressive in their support schedule, so it winds up being a double-whammy of problems.
