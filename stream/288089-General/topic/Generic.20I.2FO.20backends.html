Topic: Generic I/O backends

Danilo Krummrich (Jan 31 2026 at 13:52):

The goal for the I/O infrastructure is to generalize its API to be able to represent arbitrary backends. The first step in this direction has been done with [1].

This has the advantage that:

  1. We can build a common register abstraction (with the register!() macro [2]) on top of this generic interface, making it accessible for all busses and memory backed backends.

  2. We can share infrastructure that otherwise would be duplicated over and over again. For instance, dma_read!() and dma_write!() can be generalized over I/O backends with trait methods for read_field() and write_field().

The duplication we would have without (2) came up for iosys_map for instance [3]. With generic I/O backends we have a generalization that is much more powerful than things like iosys_map though.

As we have more people working on contributions towards those goals, let's use this thread for coordination.

[1] https://lore.kernel.org/all/20260121202212.4438-1-zhiw@nvidia.com/
[2] https://lore.kernel.org/all/20260129-register-v5-5-c4587c902514@nvidia.com/
[3] https://lore.kernel.org/all/DFOP5BY09539.AFY5L5FV1HNV@kernel.org/

It's the exact same concept, I/O memory and DMA memory is in the end just an abstract backing storage that is shared between a device and the CPU. Same goes for any other device memory, regardless of through which bus it is connected to the CPU.

I.e. for all of those the concept of registers, field accesses and "unsafe slices" is applicable.

Look at C infrastructure like iosys_map [1] which builds exactly this abstraction (of course without all the nice advantages we can get from Rust's type system for building such an abstraction).

I did some experiment to see if the untyped Io design we have today can be made compatible with the strongly typed dma::CoherentAllocation design. This is indeed possible. Some snippets:

/// Trait to represent compile-time known size information.
trait KnownSize {
    /// Minimum size of object of this type.
    const MIN_SIZE: usize;

    /// Maximum size of object of this type.
    fn maxsize(p: *const Self) -> usize;
}

// For sized types, everything is known.
impl<T> KnownSize for T {
    const MIN_SIZE: usize = core::mem::size_of::<T>();

    fn maxsize(_: *const Self) -> usize {
        Self::MIN_SIZE
    }
}

impl<T> KnownSize for [T] {
    const MIN_SIZE: usize = 0;

    fn maxsize(p: *const Self) -> usize {
        p.len() * core::mem::size_of::<T>()
    }
}

#[repr(transparent)]
struct SliceLargerThan<const SIZE: usize, T>([T]);

impl<const SIZE: usize, T> KnownSize for SliceLargerThan<SIZE, T> {
    const MIN_SIZE: usize = SIZE * core::mem::size_of::<T>();

    fn maxsize(p: *const Self) -> usize {
        (p as *const [T]).len() * core::mem::size_of::<T>()
    }
}

struct TypedMmio<T>(*const T); // aka __iomem T*

type Mmio<const SIZE: usize> = TypedMmio<SliceLargerThan<SIZE, u8>>;

Danilo Krummrich (Feb 03 2026 at 22:51):

I think there is no work in progress yet, but I think @Eliot Courtney and @Gary Guo are looking into this as well.

The main idea of IoSlice would be to have something that deals with the fact that we can't take a "real" slice of I/O memory, DMA memory, etc. You could also say UnsafeSlice. I think it should be a borrow of the I/O backend and provide methods that forward to the I/O backend implementation depending on what source or destination we are copying from / to.

So, let's say you have a dma::CoherentAllocation<T>, then we could have a method as_slice() (or as_io_slice()) that returns a IoSlice<'io, T>. IoSlice<T> could have a method copy_from_slice()that takes a "normal" Rust slice.

Similarly, you could also have methods that copy from a UserSlice or interact with UserSliceReader and UserSliceWriter directly. We could also have methods that allow us to copy back and forth between two IoSlice<T> with different I/O backends.

This way users never have to think about what exact lower level accessors we need to copy from or to certain memory, as everything is abstracted in I/O backends and IoSlice.
