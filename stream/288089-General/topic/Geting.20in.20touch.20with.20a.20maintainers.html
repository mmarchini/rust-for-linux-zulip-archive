Miguel Ojeda (Jun 19 2023 at 09:25):

Welcome back Esteban!

It is great to hear you will have time to work again on SPI.

The emails in the list have been mostly RFCs with some code attached, but they may give you some ideas from their cover letter. These are some of the recent ones that also had discussion around them with maintainers etc., so that you see what is usually discussed with them too:

https://lore.kernel.org/rust-for-linux/20230307-rust-drm-v1-0-917ff5bc80a8@asahilina.net/
https://lore.kernel.org/rust-for-linux/20230406215615.122099-1-daniel.almeida@collabora.com/
https://lore.kernel.org/rust-for-linux/20230503090708.2524310-1-nmi@metaspace.dk/
https://lore.kernel.org/rust-for-linux/20230609063118.24852-1-amiculas@cisco.com/
https://lore.kernel.org/rust-for-linux/20230613045326.3938283-1-fujita.tomonori@gmail.com/

One possibility is reaching out to them in private first, if that is easier. Or you could simply wait until you have the RFC, and then have the discussion in the mailing list about the code.

As for the contents, in essence, you need to open the discussion on why you think it is a good idea for the kernel to have such abstractions. So you may want to talk about things like (I am writing this as a generic list -- feel free to use the parts that apply to your case):

  • Why do you think the abstractions, for that subsystem in particular, are a good idea? For instance, have the C drivers (or code in the subsystem in general) been historically buggy or perhaps hard to write or to work with? Does Rust offer some particular feature that you think will make it easier to write that kind of code? (e.g. enums, type safety, perhaps async...).

  • How do you plan to use the abstractions? i.e. are you planning to submit the drivers too? Which ones / when? Otherwise, without an in-tree kernel user, it can be hard to justify. Does your company need the driver?

  • How is it going to be maintained if merged, e.g. would you be willing to be the maintainer if needed? To which degree? Is a company involved? (e.g. will they allow you to spend some time maintaining them, perhaps?). Here it may help to introduce yourself, if you have never talked to them before, so that they have some context.

  • A description of what you have so far: what works, what doesn't. How confident you are in it.

  • Some statistics, benchmarks, etc. about your code, if you have them.

  • If maintainers do not have Rust experience, but are willing to learn, would you be able to help them? In particular, if they need to change something in the C side, will you be available to react to the change on the Rust side or at least guide them on how to do the change? Or perhaps you can just commit to those changes yourself.

Hope that helps!
