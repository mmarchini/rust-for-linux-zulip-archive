Topic: `Arc` and `ARef`

Wedson Almeida Filho (Feb 13 2023 at 05:06):

I have a way to unify Arc and ARef but it requires min_specialization.

Currently Rust ref-counted objects are Arc<T> and C (always) ref-counted objects are ARef<T> (e.g., ARef<Task>). With min_specialization we can make them both Arc<T>, which I think is a nice unification of the concepts. This is also something that @Gary Guo has been asking for for a long time.

How do you all feel about this? Is it worth taking a new unstable feature?

Wedson Almeida Filho (Feb 13 2023 at 15:04):

Gary Guo said:

That's what I suggested in the last meeting

Indeed, but I don't think this is ergonomic because we go from Arc<T> to Arc<WithRef<T>> for Rust types.

My approach with min_specialization is to have a RefCounted trait with unsafe inc_refand dec_ref that require callers to ensure that that instance of Self is ref-counted. There would be a default implementation for all types that assumes that the object was allocated as an ArcInner (or WithRef, however we decide to call it). The C wrappers that are always refcounted specialize RefCounted to call the appropriate C functions (e.g., get_task_struct/put_task_struct for Task).

Then Arc<T> requires T: RefCounted and its constructors always allocate with ArcInner (as it is today), so it's safe to call inc_ref/dec_ref because the object was either allocated with ArcInner and uses the default impl of RefCounted, or it came from C and has a specialized RefCounted.
