Topic: Returning a variable number of values

Remo Senekowitsch (Mar 18 2025 at 15:27):

While writing a safe wrapper around fwnode_property_get_reference_args, I had to think about the best way to return a variable number of values from a function. I have several ideas and it seems like a general problem, so I'd like to get other people's opinion.

In "normal" Rust, you'd return a Vec, and if you can't afford heap allocation, you'd probably use arrayvec. For the kernel, my first approach was to basically vendor the code of arrayvec and have normal function signatures like fn foo() -> ArrayVec<T, N>. But now I came up with a second, more "lightweight" approach: The user of a function passes in a &mut [MaybeUninit<T>; N] and receives a &[T] in return. The lifetime of these two references are bound to each other.

fn get_dyn_values<'a>(out_param: &'a mut [MaybeUninit<i32>; 8]) -> &'a [i32] {
    // TODO: fill some elements... (needs unsafe)
    let elems = 4;
    unsafe { slice::from_raw_parts(out_param.as_ptr().cast(), elems) }
}
fn main() {
    let mut memory = [MaybeUninit::uninit(); 8];
    let elements = get_dyn_values(&mut memory);
    for x in elements {
        println!("number is {x}");
    }
}

Some of the tradeoffs that come to mind:

  • no need to vendor a big blob of boring library code
  • user has to allocate the uninitialized array, which is slightly unergonomic
  • requires use of unsafe within the abstraction (where we probably need it anyway)
  • arrayvec may be useful for drivers too, not just writing abstractions, so we may want to vendor it anyway
  • generated assembly is maybe better (?) or at least more predictable

About the generated assembly: I have no confidence in my ability to inspect assembly and how it is generated, so most of this is just guessing. Passing an out-parameter via pointer to a function generates predictably good assembly. Returning something like -> ArrayVec<T, SomeVeryBigNumber> at least superficially carries the risk of moving all that data around on the stack. I would not be surprised if LLVM can optimize this, but I have no idea if that can be relied upon.

Does anybody have insights / opinions about this topic?
