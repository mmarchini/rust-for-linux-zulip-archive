Follow up from the discussion yesterday; notes from the meeting:

struct MyDriver;

struct MyModule {
   _driver: Registration<pci::Adapter<MyDriver>,
};

impl pci::Driver for MyDriver { ... }

impl InPlaceModule for MyModule { ... }

module! {
   type: MyModule,
   ...
}

module_pci_driver! {
    type: MyDriver,
    ...
}
  • module_*_driver!'s type argument is the actual driver type, whereas
  • module!'s type argument is the type implementing InPlaceModule

impl ModuleMetadata for {type_} {
    const NAME: &'static CStr = c_str!(env!("KBUILD_MODNAME"));
}

impl PartOfModule for MyDriver {
    type Module = MyModule;
}

impl PartOfModule for MyModule {
    type Module = MyModule;
}

There is a little bit of an inconvenience with the discussed solution:

Drivers using the module_*_driver! macros all end up with the exact same type implementing InPlaceModule, see [1].

Obviously, we can't implement ModuleMetadata in module! for this type, since module! is called for every driver. Or in other words, the discussed solution requires a unique module type per module, but the module_*_driver! macros use a common type.

The obvious solution would be to get rid of kernel::driver::Module and instead generate a new module type within the module_*_driver! macros, which kinda sucks a bit.

(I wonder if passing through the module name in Module::init was that bad of an idea.)

Any ideas?

[1] https://github.com/Rust-for-Linux/linux/commit/0fcde4fb43f36673cac188e339f09503cf109b18#diff-4318897edaeb5a4bc0ad9547ff7a7219463aee9c44bab91f13c8bf543bc62926R89
