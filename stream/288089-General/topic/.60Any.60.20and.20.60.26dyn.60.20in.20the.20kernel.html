Topic: `Any` and `&dyn` in the kernel

Lyude (Apr 25 2025 at 19:55):

Hi! So to provide context to start off: in the KMS bindings I've been working on, we have two different types of most modesetting objects:

Crtc<T: DriverCrtc>
OpaqueCrtc<T: KmsDriver>

Where T as DriverCrtc can hold private data specific to the driver, and OpaqueCrtc is the "opaque" variant of a Crtc that doesn't know the specific DriverCrtc being implemented and cannot access the private data. Both of these implement AsRawCrtc which implements RawCrtc for them, which allows both of them to use a common subset of methods that don't rely on knowing what DriverCrtc is being used. We do this same pattern for pretty much every modesetting object (so, also Encoder, Connector, and Plane). And, you can fallibly cast from a OpaqueCrtc to a Crtc.

Now - when I posted these patches last one of the questions I got was why we didn't use &dyn and to be honest, I don't actually know why! Which brings me to ask: with the kernel, would it actually be possible for us to implement a similar scheme where instead of having OpaqueCrtc we just replace it with some sort of &dyn type? And would we need to use Any for enabling casting like this?

Hello, I think yes you can make a similar abstraction with &dyn trait but trait objects use runtime dispatch and compile-time dispatch problem arises i guess and if you use &dyn again and return to t again you need to downcast in rust. in my opinion this may cause problems with kernel-level bindings and again i think &dyn is a runtime dispatch, which will cause complexity.

again, this doesnt prevent writing Crtc<T> and OpaqueCrtc structure with &dyn and Any.
