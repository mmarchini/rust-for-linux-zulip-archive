Asahi Lina (Dec 05 2022 at 15:27):

I just realized that there is no support for module ID tables yet, so modules don't autoload. I tried to implement it but I ended up giving up... modpost wants a static symbol containing the raw ID table, and the IdTable stuff in driver.rs doesn't really have any obvious way of getting a sized array out, since the size information is statically lost as it only keeps a reference to the first element, and modpost wants a correctly sized object (integer multiple of single ID size).

I ended up manually encoding the module aliases (where I also had to add support for more than one module alias, because that's a multivalued property)... ^^;

Does anyone have any idea of how this ought to be implemented / is anything planned here? I get the feeling the whole IdTable mechanism needs a significant refactor to make this possible, but I don't know what direction it should go in. It might be that part of the answer involves changing modpost to play more nicely with what Rust is doing?

I did figure out how to get the right symbol name generated for modpost (just with the wrong data). In case it's useful, here's the magic incantation: 

macro_rules! module_id_table {
    ($table_name:ident, $table_type:literal, ...) => {
        #[export_name = concat!("__mod_", $table_type, "__", stringify!($table_name), "_device_table")]
        static $table_name: type... = ...;
    };
}

// Sample call: module_id_table!(some_id_table, "of", ...);

Asahi Lina (Dec 08 2022 at 05:53):

In the case of ID tables specifically, the way C does it the same structures get used at runtime and for modpost. So while it could be done by generating a C file, you'd have to have that link back to Rust somehow for the Rust-side driver_data machinery attached to device IDs to work, which sounds painful.

In general, the current Rust abstraction doesn't really work generally for device tables. In C, you can define an arbitrary number of global device tables, you can reference them from explicit lookup functions, you can pick one to be the DT match table for a specific driver (I think similarly for other ID types), and you can mark an arbitrary number of them to be picked up by modpost to generate module aliases. Rust at the very least needs to support decoupling ID tables from the driver, which means the macros that generate tables should be stand-alone (not attached to the device driver associated const), and then the driver should probably just take a reference to the table instead. I started implementing this and that part works fine, but then I ran into not being able to get the actual table struct out as a stand-alone symbol the way the table machinery works.

One way out would be to just change modpost to not care about the overall symbol size, and just look for a terminating entry the same way the runtime code does. Then we could pass the raw Rust table object directly (which has some appended stuff) and it would work. But then we still need to figure out how to attach a symbol to the real table object. Right now Rust ID tables have an indirection, and I'm not sure how practical it would be to make modpost follow pointers (across relocations and all that...).

If folks think this direction is good I can try implementing it again, though it's probably going to need turning the ID table stuff on its head quite a bit...

Asahi Lina (Dec 10 2022 at 11:10):

modpost expects symbols with a specific name to contain ID match tables, then extracts them and converts them to textual module aliases and generates the alias= modinfo fields. So the Rust side has to generate those symbols. The tables are the same the Rust code already generates for runtime APIs, they are just not exported the right way for modpost to work right now.

The patch I proposed doesn't need any C side changes, but it has the disadvantage that you end up with two copies of the table, one for just modpost, because the way Rust currently handles ID tables has it appending its own metadata into the same constant symbol, and modpost expects the table symbol to be exactly sized. To avoid the duplication would require either telling modpost to not check sizes strictly for Rust (for example, we could have a convention that all the Rust modpost tables start with _rust and skip the length check for those), or refactoring the Rust table handling not to require directly appended metadata but instead reference that some other way.

This is about compile-time generation, so you can't just call the C APIs (they're macros, not APIs)!
