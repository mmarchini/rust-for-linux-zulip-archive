Asahi Lina (Dec 05 2024 at 19:21):

Brian Misiak said:

Reading Asahi Lina 's description of the drm scheduler and fences, it kinda sounded like something that could be made very ergonomic if it could be abstracted away via futures/async :thinking: That being said I know next to nothing about the kernel really so here I am perusing zulip for whether or not this idea has been discussed, heh.

I don't think it would be that useful because you already have a job concept and the point of the scheduler is to do the work scheduling around fences, which kind of duplicates an async runtime. In general fences aren't used for kernel work, they are used for hardware work, and the chaining is userspace-driven, so I don't see how an async programming style would help much. I guess you could do it for job dependency/resource acquisition? But I don't even do that in my driver right now (resources are acquired synchronously at submit time).

There's also the thing where fence signaling cannot do any allocations, so if the async runtime does any allocations behind the scenes it's not suitable in that context either.

Asahi Lina (Dec 05 2024 at 20:17):

The thing is there isn't really a variable state machine. The lifecycle of the DRM job is mostly fixed and driven by the scheduler, you just provide callbacks to do certain things. 90% of the code is just blocking and runs on the submission ioctl, then the other callbacks just check for prerequisites, submit the job to the hardware, and when it completes you signal the fence. You could write it out as an async function, sure, but it wouldn't make it any easier to understand, and it would allow for misuse because then you could choose not to perform operations in the right expected order.

Asahi Lina (Dec 05 2024 at 20:21):

I think the only part that could be usefully written as async code is the prepare_job callback that is supposed to acquire hardware resources and return a new fence for each one. You'd need a DRM fence-driven async runtime for that. But as I said... my driver doesn't even do that in that callback, it just checks for space in the hw job queue and if not returns the oldest available fence, so there isn't any open-coded state machine there either, just a single piece of code that does one thing.

Asahi Lina (Dec 05 2024 at 20:24):

I acquire resources in the blocking path ahead of time instead, and that works fine because the GPU has ~64 slots for every exhaustible resource so it's very unlikely to block in real-world scenarios because nobody ever has 64 threads/queues competing for GPU resources all at once (and if it does happen it'll still make forward progress fine, there's no fairness guarantee but an async runtime wouldn't necessarily guarantee that either on its own. I actually have a mode where I serialize all GPU execution by limiting one resource to 1 slot and it runs fine and barely has a performance impact for regular usage).
