Topic: Naming conventions?

Daniel Almeida (Mar 14 2023 at 12:33):

Hi all, I have been working on some virtio and V4L2 bindings recently.

I don't think it's clear when to wrap over a pointer vs when to wrap over the bindgen struct directly, i.e.:

 struct Wrapper(<some pointer>)

vs

struct Wrapper(UnsafeCell<bindings::some_struct>)

Both approaches are used in the kernel crate, so I wonder if we could have some guidelines on how to choose between these two and also on how to choose proper names for these.

I'm asking because when prototyping a driver, at times I was passed in a pointer by some kernel callback, but there were instances where I found myself trying to instantiate the same object on the stack.

I started naming the struct Wrapper(<some pointer>) variant as struct FooRef(<some_pointer>) to distinguish it from struct Foo(UnsafeCell<bindings::some_struct>).

Thoughts?

Boqun Feng (Mar 14 2023 at 14:26):

I think overall the approach you are using is preferred, just note that if some_struct itself has a reference count, you may be able to impl AlwaysRefCounted for Foo, and FooRef can be just ARef<Foo>.

Trevor Gross (Mar 15 2023 at 00:02):

I'm not involved with RFL, but here's my 2cents from other FFI work:

#[repr(transparent)]
struct SomeStruct(UnsafeCell<bindings::some_struct>)

This is typically the pattern to use when you want a representation of a C structure in Rust. If you're often working with pointers, you typically will want something like:

impl SomeStruct{
    pub(crate) unsafe fn from_raw_ptr<'p>(ptr: *mut bindings::some_struct) -> &'p Self {
        &*ptr.cast()
    }
}

The &Self is equivalent to some_struct* or struct Wrapper(<some pointer>) and can be passed to C functions taking a pointer (with a cast). Writing the methods is more ergonmic this way.

I think that something like struct FooRef(<some_pointer>)is typically only useful for cases where the pointer may not be valid and has to be checked somewhere else. But usually something like this should have a PhantomData lifetime - becauase otherwise, you could pass it around for use anywhere, even after the pointer is invalid
