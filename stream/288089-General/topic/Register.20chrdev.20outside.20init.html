Louis Chauvet (Mar 15 2023 at 14:52):

Hi again,
I try to write a driver which does:

  • Create a platform device
  • Register chrdev_region of MAX_CHRDEV devices
  • On probe, afer reading some I/O, register a chrdev in the allocated region

But I'm stuck on the step "during probe, get access to the chrdev_region"

Actually I have this:

struct MyDriver {
    _char_dev: Pin<Box<chrdev::Registration<MAX_BUSSES>>>,
    _pdev: Pin<Box<platform::Registration<Self>>>,
}

struct Bus {
    base: IoMem<252>,
}

type BusData = device::Data<Bus, (), ()>;

impl platform::Driver for MyDriver {
    fn probe(
        dev: &mut platform::Device,
        _id_info: core::option::Option<&Self::IdInfo>,
    ) -> Result<Self::Data> {
        let res_mem: IoMem<252> = unsafe { dev.ioremap_resource(0) }?;
        let minor = res_mem.readl(REG_USER_ID);
        // Here I want to access MyDriver._char_dev to register a device with the minor
        // The C version of this driver use a static global "major" and cdev_init/cdev_add/device_create to achieve that, but there may be a better way to do that in rust
    }
}


impl kernel::Module for MyDriver {
    fn init(name: &'static CStr, module: &'static ThisModule) -> Result<Self> {
         let char_dev = chrdev::Registration::new_pinned(name, 0, module).map_err(|e| {
            pr_err!("Failed to register char device: {:?}\n", e);
            e
        })?;
        let pdev = platform::Registration::new_pinned(name, module).map_err(|e| {
            pr_err!("Failed to register platform: {:?}\n", e);
            e
        })?;
        Ok(MyDriver {
            _char_dev: char_dev,
            _pdev: pdev,
        })
    }
}

Louis Chauvet (Mar 16 2023 at 15:04):

I found a way to do that, but it's ugly I think, and maybe not safe:

        let module = unsafe {
            match &mut __MOD {
                Some(module) => module,
                None => return Err(ENODEV),
            }
        };

It allows me to register on existing chardevice, but I think it's not a good idea to do that (what happens if two probes happens in the same time ? I don't know enough about the kernel to know if it is necessary for the probe function to be reentrant)
I may need something like a mutex to avoid those problem.

For my tests it will be sufficient (I have only one device), but if you have a better solution I will be very happy to use it :
