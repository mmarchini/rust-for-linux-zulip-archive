Miguel Ojeda (Jan 15 2024 at 12:04):

Vadzim Dambrouski said:

What if we use neither of them, not libclang and not gcc. It should be possible to implement a simple C language parser in Rust in a reasonable amount time since there are no C++ in the kernel. The only issue that I see is that we have to know the precise ABI information to generate correct struct sizes. Is there any way to get this info without linking to gcc or clang?

There are many C parsers out there (e.g. sparse that Linus started), but the issue is really interpreting the code exactly as GCC does: it is not just GNU extensions or conditional compilation, but everything that may change the ABI as you say. For instance, one may have enabled layout randomization or one may be using a plugin or a fork etc.

So one way would be to "just" ask GCC to give us the information. Otherwise, if we are able to use another parser, one could simply use Clang (perhaps with extra information that we lack). If we do the latter (i.e. what we are doing now), one very good thing to have (suggested by Nick) is checking after the fact (via e.g. DWARF) that things match.

Vadzim Dambrouski (Feb 11 2024 at 21:50):

The new approach seems to be working well for now. The idea is to parse the file in 2 stages, first stage only collects tokens into groups separated by semicolons and nested by (){}[]. Then on the second stage we can parse each group of tokens separately to extract the declarations. The first stage can't fail unless you have unbalanced parenthesis (which is not valid in C anyway). This allows us to precisely skip over all the complicated syntax inside the function bodies and other declarations we don't care about and only parse what we need. The second advantage is that on the second stage if we failed to parse some declaration we can just ignore it and skip over to the next one. It's not a big deal to skip some struct which failed to parse, we don't need to abort the whole process.

This weekend I implemented the first stage parser. It's able to parse all of the bindings_helper.h file into a tree and then serialize it back to the stage1.c file without introducing any syntax errors. I put the code on github if anyone interested

https://github.com/pftbest/kbindgen

I also started parsing some things for the second stage but there is nothing concrete for now, hopefully I will have more time next weekend.

Then on the second stage we can parse each group of tokens separately to extract the declarations. The first stage can't fail unless you have unbalanced parenthesis (which is not valid in C anyway).

Btw, looks like you might have already figured this out but the first stage is known as "lexing" (splitting identifiers, punctuation, string literals), "parsing" is the second stage where you turn it into a usable AST/CST
