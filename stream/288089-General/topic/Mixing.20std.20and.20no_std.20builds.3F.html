Topic: Mixing std and no_std builds?

Richard Weinberger (Sep 24 2023 at 09:19):

I'm currently evaluating implementing a simplified variant of UserModeLinux's hostfs in rust. UML drivers always consist of a kernel and a userspace component. So to I'd have to use std (for the userspace parts to interact with the host) and no_std for the kernel side.
Is mixing std and non_std possible?

bjorn3 (Sep 24 2023 at 09:42):

Nornally you can have crates that don't depend on libstd and that do together, however rust-for-linux doesn't work with libstd. It uses a fork of liballoc which conflicts with the liballoc used by libstd and doesn't link in the allocator shim that is necessary to successfully link anything that uses libstd. It also defines a panic handler that conflicta with libstd's panic handler.

Richard Weinberger (Sep 24 2023 at 10:01):

That was my fear. rust-for-linux defines it's only std functions which will then conflict with libstd. So, this makes it currently impossible to utilize rust for UML. Unless you want only the kernel side.

Benno Lossin (Sep 24 2023 at 10:18):

You still could try to create a common library that is no_std and also alloc-agnostic. It could be used by both the kernel and the userspace parts. I would have to see the code to fully understand if this is possible, but it might be worth trying.
