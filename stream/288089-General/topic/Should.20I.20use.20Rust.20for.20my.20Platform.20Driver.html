bjorn3 (May 09 2023 at 18:24):

I believe building the rest of the kernel with GCC is possible, but it tends to break as new options are passed to GCC that clang doesn't understand. We use rust-bindgen for generating bindings which is based on libclang and as such inherits all clang C compiler flags. I believe we have a list of options only GCC supports to skip when invoking rust-bindgen, but this list is not always up to date.

bjorn3 (May 09 2023 at 18:24):

Rustc itself uses LLVM unconditionally currently. There is an experimental GCC backend though.

bjorn3 (May 09 2023 at 18:35):

Part of the rust infrastructure (like libcore and libkernel) is linked into the main kernel image. As such you need to build the kernel itself with rust support to be able to load rust modules.

Asahi Lina (May 10 2023 at 05:24):

Ayush Singh said:

I know that Asahi Linux has drivers written in Rust but I'm not sure how that works. Does it build the kernel using LLVM?

The Arch builds are using GCC for the C part of the kernel, and yes the GPU driver is a loadable module. However, you can't have Rust support itself as a loadable module right now (with the upstream trees anyway). So the kernel itself and all the core abstractions are always built-in, then your driver can be a module like any other driver.

Right now upstream doesn't have enough abstractions for platform support though, so you will have to use downstream trees. Hopefully this will change in the next few kernel versions ^^

bjorn3 (May 13 2023 at 20:31):

Rustc has an entirely different commandline interface from clang anyway, so it doesn't matter that kbuild passes different args to the C compiler when using GCC vs LLVM. (except for in rust-bindgen, but there we can and do filter out unsupported args on a best effort basis) Linking itself shouldn't matter unless LTO is used.
