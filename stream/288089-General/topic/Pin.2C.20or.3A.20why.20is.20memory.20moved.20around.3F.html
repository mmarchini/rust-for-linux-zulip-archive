Topic: Pin, or: why is memory moved around?

Philipp Stanner (Apr 17 2024 at 13:22):

I get the necessity and use cases for Pin – but why does the Rust language / compiler even move memory around implicitly in the first place? What's the advantage?

bjorn3 (Apr 17 2024 at 13:26):

Without Pin a user can explicitly move a value to a different location without any unsafe code.

Philipp Stanner (Apr 17 2024 at 13:28):

So it is 100% about preventing a _programmer_ from moving memory _explicitly_ and Rust does not implicitly move, say, a Box to a different position on the heap?

Alice Ryhl (Apr 17 2024 at 13:28):

Moves of values are closely tied to ownership, which is pretty central to how Rust's ownership and borrow checker works. Having something like C++ move constructors would seriously complicate many things.

But I think it's also worth noting that the decision to make everything movable is from pre-2015 and it wasn't until 2020-ish that they found a use-case that necessitated that Rust add support for Pin.

Alice Ryhl (Apr 17 2024 at 13:30):

It's worth noting that Pin is not really a language feature. The compiler itself doesn't just move values around without you doing an actual move operation in the source code. Pin is just a library where we essentially use private fields and other mechanisms of that sort to prevent you from expressing any move that results in a change of address.

bjorn3 (Apr 17 2024 at 13:31):

Values on the heap are never implicitly moved by rustc. Nor are stack values while a reference is outstanding. That is why Pin was implementable as pure library construct without compiler integration. (Technically there is some compiler integration to disable the noalias optimization flag in some cases to allow self-referential values.)

Alice Ryhl (Apr 17 2024 at 13:33):

It's closely tied with the idea that if some piece of code triggers memory safety, then there must be an incorrect unsafe block somewhere. The problem is that if you make a struct that has internal unsafe code that relies on the struct's address not changing, then, well, it's not unsafe for the owner of the struct to move it. So since the only unsafe blocks in the program are in the implementation of the struct, then it must be the case that incorrect unsafe block is in the implementation of the struct.

Alice Ryhl (Apr 17 2024 at 13:34):

Pinning something fundamentally requires you to call the unsafe method Pin::new_unchecked, so by having the self-referential struct take a pinned reference, then you force the call to use unsafe to call Pin::new_unchecked. Now you also have an unsafe block in the caller, so now it is possible to say "the incorrect unsafe block is the one in the caller".

Alice Ryhl (Apr 17 2024 at 13:34):

(Now, we also have some library methods that allow you to pin stuff safely. These work by using private fields/shadowing/etc to prevent the user from moving the value safely. But they all boil down to Pin::new_unchecked somewhere.)
