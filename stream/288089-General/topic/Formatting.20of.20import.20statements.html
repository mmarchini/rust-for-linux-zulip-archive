Topic: Formatting of import statements

Remo Senekowitsch (Oct 05 2025 at 11:02):

Linus' email complaining about rustfmt's handling of import statements caught some attention online.

I also experienced this issue (on many different projects) and tried to find out what could be done about it. As Miguel mentioned, the relevant rustfmt configuration is sadly unstable. However, rustfmt actually just preserves the style that already exists. Which means, if new import statements are added in a merge-friendly style, rustfmt will be happy to keep that.

I always add import statements via rust-analyzer, which defaults to merging import statements at the "crate" granularity. I imagine most Rust developers do the same. rust-analyzer can be configured to put every item on a separate line, which would be much more merge-friendly. I was just informed that this configuration can be done in a editor independent way, which is obviously important for big projects like the Linux kernel. This is not (yet) documented in rust-analyzer's manual, so not many people know about it.

Now, that approach doesn't completely enforce this style yet. But, if there are people not using rust-analyzer and adding import statements manually, they would probably copy the style that already exists in a file. Also, one item per line is actually easier to type manually than managing the nested curly braces.

So, I think this could be an interesting option for the kernel. The configuration is made by adding a rust-analyzer.toml file in the root of the project, with the following content:

imports.granularity.group = "item"

This style is more verbose, but it leads to much more straight-forward merge conflicts. I have enabled it unconditionally in my own user config.

Andreas Hindborg (Oct 06 2025 at 10:26):

So the problem is Linus cannot merge the imports if they are on the same line, but it is easy if every word is on a separate line?

Remo Senekowitsch (Oct 06 2025 at 13:21):

Kinda. If items are in a group, e.g. use foo::{bar, qux};then it's hard to guess if rustfmt will put it on one line or multiple. That's the immediate problem he's complaining about. But also the fact that rustfmt putting a few items on one line leads to more merge conflicts. Take this example:

use foo::{b, c}; // one line
use foo::{
    b,
    c,
}; // same import, multiple lines

Now, if someone imports foo::a and someone else imports foo::d, the one-line version will generate an annoying conflict while the multi-line version will not. Additionally, if Linus manually fixes the conflict with use foo::{a, b, c, d};, rustfmt might then decide that now is the time to break it onto four lines and annoy him again.

Remo Senekowitsch (Oct 06 2025 at 13:26):

Miguel said in his reply to Linus:

So if the conflicts are already too frequent and/or painful to deal
with, then I would say we should already start writing the imports in
a way that reduces the conflict potential, yet making it pass
rustfmt. It may be uglier than what some may like (depending on who
one asks), but it keeps the rustfmt-clean property and eventually we
may be able to get rustfmt to give us the formatting we want and
migrate to that.

I think imports granularity "item" is exactly that. It's not the prettiest, but we can do it right now. With a one-time manual migration (cargo +nightly fmt) and a rust-analyzer.toml file committed to the repo. Once they are stable, the config options granularity "module" and layout "vertical" might be preferred, as it's prettier and equally merge-friendly.
