Topic: ✔ Arc containing an atomic

Philipp Stanner (Sep 11 2025 at 11:52):

Ahoi.
I've got an Arc that contains an atomic.

#[pin_data]
pub struct DmaFenceCtx {
    #[pin]
    lock: Opaque<bindings::spinlock_t>,
    nr: u64,
    seqno: AtomicU64,
}

impl DmaFenceCtx {
    pub fn new() -> Result<Pin<KBox<Self>>> {
        let ctx = pin_init!(Self {
            // Feed in a non-Rust spinlock for now, since the Rust side never needs the lock.
            lock <- Opaque::ffi_init(|slot: *mut bindings::spinlock| {
                // SAFETY: `slot` is a valid pointer to an uninitialized `struct spinlock_t`.
                unsafe { bindings::spin_lock_init(slot) };
            }),
            // SAFETY: `dma_fence_context_alloc()` merely works on a global atomic. Parameter `1`
            // is the number of contexts we want to allocate.
            nr: unsafe { bindings::dma_fence_context_alloc(1) },
            seqno: AtomicU64::new(0),
        });

        KBox::pin_init(ctx, GFP_KERNEL)
    }

    fn get_new_fence_seqno(&mut self) -> u64 {
        self.seqno.fetch_add(1, Ordering::Relaxed)
    }
}

Rust won't allow me to borrow the arc as mutable. I.e. I can't do my_arc.get_new_fence_seqno().
Presumably because it wants to enforce that locking be used, since Arcs are usually shared.

error[E0596]: cannot borrow data in dereference of `Arc<DmaFenceCtx>` as mutable
   --> rust/kernel/sync/dma_fence.rs:107:26
    |
107 |         let seqno: u64 = fctx.get_new_fence_seqno();
    |                          ^^^^ cannot borrow as mutable
    |
    = help: trait `DerefMut` is required to modify through a dereference, but it is not implemented for `Arc<DmaFenceCtx>`

I don't need locking in this case, however. How do I solve that idiomatically. Do I have to implement DerefMut, as the compiler is suggesting?

Alice Ryhl (Sep 11 2025 at 12:01):

The method should be &self to take a shared reference instead of an exclusive reference. You can't get an exclusive reference to something in an Arc because an Arc is always considered shared.

Philipp Stanner (Sep 11 2025 at 12:21):

I need a mutable (exclusive?) reference for modifying the atomic.

Philipp Stanner (Sep 11 2025 at 12:23):

Hmm, or maybe I don't?
How is it allowed / possible to modify an atomic through a non-mutable reference?

Alice Ryhl (Sep 11 2025 at 12:26):

It's because mutable is a misnomer. &mut really means exclusive and & really means shared. Modifying atomics through shared access is ok, therefore you can do it with an &.

Philipp Stanner (Sep 11 2025 at 12:28):

Yeah, I know it's OK – but how does the compiler know / understand?

Alice Ryhl (Sep 11 2025 at 12:28):

Because the AtomicU64::fetch_add method is declared as &self.
