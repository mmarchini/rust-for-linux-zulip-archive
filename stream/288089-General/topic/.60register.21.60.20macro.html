I have a proof-of-concept of a different register! API as disccused in the thread here: https://lore.kernel.org/rust-for-linux/20260128-register-v4-0-aee3a33d9649@nvidia.com/T/#mbdf1acf98fc691df50845553cea46d7c53725ef6.

The code is here https://github.com/nbdd0121/linux/commit/d5264f51a48340fc108a75b4c470aab44c298caa (ignore the code quality as it's a POC; I created a helper type per register as it's easy, but in practice I think we don't need one per register, just 4 helper types would be sufficient).

The usage of this API would be:

io.read_reg(REGISTER_NAME) // current: REGISTER_NAME::read(io)
io.read_reg(REGISTER_NAME(base)) // current: REGISTER_NAME::read(io, base)
io.read_reg(REGISTER_ARRAY(index)) // current: REGISTER_ARRAY::read(io, index)
io.read_reg(REGISTER_ARRAY(base, index)) // current: REGISTER_ARRAY::read(io, base, index)

the benefit is that it does not have the Deref issue, and IMO this is more consistent with plain of IO access.
