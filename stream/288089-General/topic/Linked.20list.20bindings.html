Topic: Linked list bindings

Alice Ryhl (Jun 08 2023 at 13:24):

I've been thinking about how to write linked list bindings. The current ones need updating because there are some remaining soundness issues and it doesn't use pin-init. Here are my thoughts:

The links field of a struct has these components:

  1. Prev/next pointers.
  2. Some way to synchronize concurrent insertions.
  3. Some way to identify the type of element and perform a container_of.

In Binder, I have these use-cases:

  1. Boxed values that can be pushed to just one linked list. We need mutable access.
  2. Reference counted values that are pushed to a linked list where the concrete type is known. The type also needs to be enqueued to a workqueue at the same time.
  3. A trait implemented by many types where the linked list can hold any implementer of the type.
  4. A list like the one in use-case 3, except that it holds only one specific concrete type. Ideally, we want to use the same link pointers as use-case 3.

Based on the above use-cases and inspired by the workqueue, I designed this API that supports all of them:

struct LinkedList<T: LLItemPtr<ID>, const ID: u64>;

#[repr(C)]
struct LinkPtrs {
    next: *mut LinkPtrs,
    prev: *mut LinkPtrs,
}

trait LLItemPtr<const ID: u64> {
    type PushResult;
    // If the first argument is none, then inserting failed. In either case,
    // pushing returns the push result.
    fn try_into_raw(self) -> (Option<NonNull<LinkPtrs>>, Self::PushResult);
    fn from_raw(ptr: *mut LinkPtrs) -> Self;

    type InnerRef<'a>;
    fn access_raw<'a>(ptr: *mut LinkPtrs) -> Self::InnerRef<'a>;

    // Mutable cursors always make sense. Even if we can't mutate the item, we
    // can still remove it from the list.
    type CursorMutRef<'a>;
    fn access_with_cursor<'a>(ptr: *mut LinkPtrs) -> Self::CursorMutRef<'a>;
}

/// Implemented for types where a mutable iterator makes sense.
trait LLIterMutRef<const ID: u64>: LLItemPtr<ID> {
    type IterMutRef<'a>;
    fn access_with_iter<'a>(ptr: *mut LinkPtrs) -> Self::IterMutRef<'a>;
}

For the first use-case, we can have this helper:

struct LLBox<T> {
    inner: Box<LLBoxInner<T>>,
}

struct LLBoxInner<T> {
     links: LinkPtrs,
     value: T,
}

impl<const ID: u64, T> LLItemPtr<ID> for LLBox<T> {
    type PushResult = ();
    type InnerRef<'a> = &'a T;
    type CursorMutRef<'a> = &'a mut T;
    type IterMutRef<'a> = &'a mut T;
}

impl<const ID: u64, T> LLItemPtr<ID> for Pin<LLBox<T>> {
    type PushResult = ();
    type InnerRef<'a> = Pin<&'a T>;
    type CursorMutRef<'a> = Pin<&'a mut T>;
    type IterMutRef<'a> = Pin<&'a mut T>;
}

For the reference counted use-cases, we can have the following API:

trait LLItem<Ptr, const ID: u64> {
    type ContainerOfStrategy: ContainerOfStrategy<Self, ID>;
}

struct Links<S, D, const ID: u64> {
    links: LinkPtrs,
    /// Used to synchronize insertions to several lists in parallel.
    sync: S,
    /// Extra data used for the container_of operation.
    data: D,
}

trait HasLinks<const ID: u64> {
    const OFFSET: usize;
    type Data;
    type Sync;
}

// Or use named wrappers around AtomicBool and ().
type ArcLinks<D = (), const ID: u64 = 0> = Links<AtomicBool, D, ID>;
type BoxLinks<D = (), const ID: u64 = 0> = Links<(), D, ID>;

impl<const ID: u64, T: ?Sized> LLItemPtr<ID> for Arc<T>
where
    T: LLItem<Arc<T>, ID>,
    T: HasLinks<Sync = AtomicBool, ID>,
{
    type PushResult = Result<(), Arc<T>>;
    type InnerRef<'a> = ArcBorrow<'a, T>;
    type CursorMutRef<'a> = ArcBorrow<'a, T>;
}

// Giving out mutable access to the contents while it is in the list is
// problematic. You could overwrite the links pointers.
//
// Even if we give the links pointers a destructor, its still problematic.
// Sure, we can remove the item from the list, but this leaks the allocation.
impl<const ID: u64, T: ?Sized> LLItemPtr<ID> for Box<T>
where
    T: LLItem<Box<T>, ID>,
    T: HasLinks<ID>,
{
    type PushResult = ();
    type InnerRef<'a> = &'a T;
    type CursorMutRef<'a> = &'a T;
}

impl<const ID: u64, T: ?Sized> LLItemPtr<ID> for Pin<Box<T>>
where
    T: LLItem<Pin<Box<T>>, ID>,
    T: HasLinks<ID>,
{
    type PushResult = ();
    type InnerRef<'a> = Pin<&'a T>;
    type CursorMutRef<'a> = Pin<&'a T>;
}

trait ContainerOfStrategy<T, const ID: u64>{
    fn container_of(ptr: *mut LinkPtrs) -> *mut T;
}

enum SizedTypeCOS {}
impl<T, const: ID: u64> ContainerOfStrategy<T, ID> for SizedTypeCOS
where
    T: HasLinks<ID> + Sized,
{ ... }

enum MetadataCOS {}
impl<T, const: ID: u64> ContainerOfStrategy<T, ID> for MetadataCOS
where
    T: HasLinks<ID, Data = <T as Pointee>::Metadata> + ?Sized,
{ ... }

// or without ptr_metadata feature
enum SelfPtrCOS {}
impl<T, const: ID: u64> ContainerOfStrategy<T, ID> for SelfPtrCOS
where
    T: HasLinks<ID, Data = NonNull<T>> + ?Sized,
{ ... }

This solve each use-case:

Usecase 1. Using LLBox lets us do this. (We can't use a normal Box because it doesn't give mutable access.)

Usecase 2. We can do this like this:

struct MyStruct {
    #[pin]
    links: ArcLinks,
}

impl_has_links! {
    impl HasLinks for MyStruct {
        type Sync = AtomicBool;
        self.links
    }
}

impl LLItem<Arc<Self>> for MyStruct {
    type ContainerOfStrategy = SizedTypeCOS;
}

Usecase 3. We can do this like this:

struct Transaction {
    #[pin]
    links: Links<AtomicBool, Meta>,
}

impl_has_links! {
    impl HasLinks for Transaction {
        type Sync = AtomicBool;
        type Data = Meta;
        self.links
    }
}

type Meta = <dyn MyTrait as Pointee>::Metadata;
// or we could use NonNull<Transaction> as the extra data.

trait MyTrait: HasLinks<Sync = AtomicBool, Data = Meta> { ... }

impl LLItem<Arc<dyn MyTrait>> for dyn MyTrait {
    type ContainerOfStrategy = MetadataCOS;
}

Usecase 4.

We also add this impl:

impl LLItem<Arc<Self>> for Transaction {
    // This doesn't care about the type of the extra data, so it's okay that it
    // is `Meta`.
    type ContainerOfStrategy = SizedTypeCOS;
}

Simpler alternatives:

We can also move further towards the existing linked list solution. With that solution, we always perform the atomic CAS when inserting into the list even when the items are unique. It also doesn't support use-case 4.

Another alternative is to provide two completely separate linked list implementations. One of them only works with LLBox and the other only works with Arc. There are some advantages to this, e.g. we could let the user insert a UniqueArc into a linked list of Arcs to avoid the CAS on insert. The only disadvantages are code duplication and that we can't easily support new smart pointers. (Note that this is similar to the rbtree, where we currently only have what is analogous to the LLBox-only implementation.)
