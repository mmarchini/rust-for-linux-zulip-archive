Topic: Verifying Binder's parsing of userspace input in Lean 4

Natasha Klaus (Jul 06 2026 at 10:57):

We're a formal verification group (Runtime Verification) looking into whether kernel Rust parsing code can be verified post hoc in Lean 4. The toolchain is Charon/Aeneas, which extracts Rust into a form a proof assistant can reason about. Short intro to the pipeline: https://runtimeverification.com/blog/from-rust-source-code-to-mathematical-proof

As a feasibility test we took the BinderObject deserializer from drivers/android/binder/allocation.rs (v7.2-rc2): read_from, as_ref, type_to_size, and the size_check bounds logic. Charon extracted it on the first attempt with zero changes to the parsing logic. Only the environment (kernel crate, bindgen'd UAPI, UserSliceReader) needed thin stubs. The union type-punning, MaybeUninit and the raw pointer cast all survive extraction.

Report and reproduction steps: https://github.com/runtimeverification/kernel-rust-verification-spike

Next step is Aeneas into Lean 4, with first proofs that invalid type tags are rejected and the bounds checks are correct. Longer term we want the offset-validation loop in copy_transaction_data.

A question for people who know this code: if parts of Binder's input parsing carried machine-checked proofs, which properties would actually be worth having?

Natasha Klaus (Jul 10 2026 at 10:56):

An update, and @Miguel Ojeda, you called it: Aeneas rejects the union type-punning at the core of the Binder deserializer, so the parsing functions get no Lean at all. Filed upstream: https://github.com/AeneasVerif/aeneas/issues/1199 the Aeneas author already responded and we're discussing how to model the pun.

Some no-panic progress though. The union-free validators translate, and three theorems are proved, machine-checked and sorry-free: size_check (the overflow+bounds validator) never panics, unconditionally; is_aligned never panics given a nonzero divisor, with a companion theorem showing it genuinely panics on zero; type_to_size never panics modulo one axiom about size_of. Theorems: https://github.com/runtimeverification/kernel-rust-verification-spike/blob/main/lean/NoPanic.lean

Next: re-model BinderObject as [u8; 40] with typed accessors and prove the pun invariants as explicit Lean propositions instead of SAFETY comments, the path to no-panic on the full deserializer.

Natasha Klaus (Jul 15 2026 at 09:09):

Updates:

The no-panic property is proved. For all byte inputs from userspace, the Binder deserializer (parse_one) returns cleanly, a valid object or EINVAL, never a panic. Machine-checked end-to-end: Rust, Charon, Aeneas, Lean 4, no sorry, one opaque axiom about size_of. @Alice Ryhl this is property 1 from your list, on the deserializer fragment. Property 2 (cleanup/translate symmetry) needs the heavier half of allocation.rs, that's next.

The union blocker didn't go away, we went around it: the deserializer re-modelled as [u8; 40] with typed accessors, per the design the Aeneas authors suggested in the issue. Compile-time assertions pin every accessor offset to the real #[repr(C)] layout, so the proof is about the kernel's actual byte format. Theorem and write-up: https://github.com/runtimeverification/kernel-rust-verification-spike (lean/NoPanicRemodel.lean, REMODEL-REPORT.md)

And the coverage map of all kernel Rust: https://github.com/runtimeverification/kernel-rust-coverage-map. 92k LOC, 117 units. Findings: zero language-feature failures, everything that compiles extracts through Charon. Whole crates fail for one reason, the unstubbed kernel crate surface, 27 of 28 failures. So reaching the rest of the tree is one shared stub cost, not a per-driver problem. @Miguel Ojeda relevant to your maintenance point: the map argues for growing one shared kernel-surface model instead of per-target hand-copies. The four biggest untrusted-input targets (nova-core, binder, drm ioctl, iov) all carry the same union pattern, so the re-model approach transfers.

On the toolchain gaps themselves: we contribute to Aeneas upstream, and as verification work surfaces things Aeneas doesn't support, our default is to fix them upstream rather than accumulate local workarounds.

Short version: kernel Rust parsing code can be verified today. The toolchain doesn't take everything, but the gaps have workarounds that don't weaken the result, and the map shows where the effort goes.

Natasha Klaus (Jul 15 2026 at 13:22):

Gary Guo said:

One difficulty I encounter in trying to implement try_lock support for klint is reasoning about is_some(). When looking at a single function MIR level it is very difficulty to know that it is equivalent to matches!(.., Some(_)) while this is rather straightforward in lean.

That's a precise example of the boundary. In Lean this is definitional: is_some unfolds, and the equivalence with matches!(.., Some(_)) is a simp/rfl-level fact, so return-value-dependent atomicity flows through the proof without special handling. So klint runs over the whole tree cheaply, and we pay per-function proof effort. It is a division of labor: klint for breadth, proofs for the call sites where its analysis has to give up (try_lock and friends). If you can point at the patterns klint bails on most, that's a ready-made worklist for us.

Union support update.

We posted a MWE to the Aeneas tracker, distilled from the bindgen'd Binder UAPI shapes:

https://github.com/AeneasVerif/aeneas/issues/1199.
https://github.com/runtimeverification/mwe-unions

The design discussion with the maintainers on how to represent unions is now open, and we've offered to implement whichever model they pick.

Why it matters here: once native union support lands, the [u8; 40] re-model workaround becomes unnecessary. Proofs would be about the kernel's union code exactly as written, no hand-maintained equivalent in between. That's one less artifact to keep in sync, which was the maintenance concern raised earlier in this thread.
