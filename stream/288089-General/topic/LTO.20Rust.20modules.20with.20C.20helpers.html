LTO Rust modules with C helpers

I finally got some time to investigate about possible ways that we can LTO helpers together with Rust kernel modules at a crate level.

Here are 3 possibilities:

  1. Have the LLVM IR of the helpers somehow available to rustc and then it gets used in compilation step.
  2. For each crate, invoke rustc, and then link it with helpers, and then use the object file for the final linking.
  3. Use linker plugin LTO and do a global LTO.

We previously explored option 3 only. @Andreas Hindborg reported that kernel doesn't boot on real machine with this, and @Alice Ryhl reported global LTO is not preferred because it's slow and offers little gain.

I looked into option 1 today and I don't think there're any hacks that we can use to ask Rust to embed arbitrary LLVM IR/BC for compilation. It seems that if rustc is doing the linker, we can trick it into using non-Rust bitcode by make the file name end with .rcgu.o, but since we use --emit, rustc wouldn't run the linker stage at all (plus, this is a horrible hack).

So option 2 looks like the only remaining solution. However since we need link helper multiple times, it will violate one-definition rule (ODR) and cause linker error on duplicate symbol. Using weak linkage avoids the ODR issue, but it also inhibits the inlining (because the symbol can be overriden)

There is one way to workaround the issue, which is to use linkonce_odr. This is entirely LLVM internal and is not exposed to C code, but we can use some hacks:

  1. For helpers, define them using extern inline. This allows an inline hint to be generated, but also keep the method body in compilation result.
  2. Compile the helpers to LLVM IR: clang -S helper.c -emit-llvm -o helper.ll -O2
  3. Replace define dso_local with define linkonce_odr dso_local: sed -i 's/^define dso_local/define linkonce_odr dso_local/g' helper.ll
  4. For performance, convert the processed helper.ll to bitcode: llvm-as helper.ll -o helper.bc

Now for each Rust crate, we:

  1. Compile to LLVM bc: rustc --crate-type=lib --emit=llvm-bc --emit=metadata b.rs -C opt-level=2 -L .
  2. Link with the helper with llvm-link: llvm-link b.bc helper.bc -o b.m.bc (for some reason helper.bc needs to go after b.bc, but not before)
  3. Perform the optimisation and generate an object file: clang -c b.m.bc -o b.o -O2

This way, if LLVM decided to inline the function, it will disappear from the object file entirely. If LLVM decided to not inline the function, it will generate a symbol with weak linkage so it's fine to have multiple copies.

Matthew Maurer (Mar 07 2024 at 17:49):

I do want to point out here that you will likely want to be a little careful here - clang and rustc can only LTO x-lang safely if they share an LLVM. This may be a bit awkward to autodetect in the Linux kernel environment, as:

  1. rustc-as-distributed-by-a-distro will frequently link against a different LLVM than upstream (whatever the distro is currently shipping)
  2. People using clang from host and rustc from rustup is probably a common configuration. This will frequently be a mismatch.
  3. Some major kernel build environments, (Android at a minimum) use LLVMs at head. This means you cannot even depend on the version number in the bitcode being a sufficient check.

I would suggest that while this is a good feature, you should make this a default off config that people can switch on if they know they have paired compilers.

Matthew Maurer (Mar 07 2024 at 18:32):

Yeah, the thing to be careful of here is that LLVM has something that looks like a backwards compatibility gaurantee - newer tools can always load older released bitcode outputs (so if you generate bitcode with LLVM 5.3, then a linker or optimizer running LLVM 8.2 can load it. If you generate bitcode with a git commit between 5.3 and 6.0, it may or may not load.).

However, the semantics are not gauranteed to be the same. This means that if you do any nontrivial transformations (for example, inlining then optimizing) after loading older bitcode, something may go wrong.

bjorn3 (Apr 26 2024 at 09:29):

In rust doing the equivalent of `struct bar* bar = &foo->bar; if (!READ_ONCE(foo)) do_sth(); could spuriously read the value behind *bar even if !foo. Creating a reference is a promise that the reference is dereferenceable. In both stacked borrows and tree borrows creating a shared reference will insert a fake read of the referenced memory. A fake read acts like a real read in that it causes UB when a real read would cause UB, but it just so happens to be lowered to a nop by the compiler unless the compiler decides that speculatively reading it would be advantageous.
