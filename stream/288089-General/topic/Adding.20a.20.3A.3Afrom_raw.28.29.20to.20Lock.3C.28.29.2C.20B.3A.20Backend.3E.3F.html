Lyude (Jul 29 2024 at 18:57):

Hey! This was just a small thought of mine and I'm not too attached to it, but while I was writing some of the vblank bindings for KMS I noticed we actually have a few functions that rely on the caller holding a particular lock. At first I started acquiring the locks with normal unsafe calls to the C bindings but I realized it might make more sense if we added a ::from_raw() to Lock<(), B: Backend> (so, just lock types without directly associated data) so that rust bindings for C interfaces can get a rust-like reference to locks they may have that were initialized on the C side of things. Does this seem like something people would be interested in if I sent patches for it in the near future?
We would need to add a clause to the safety contract for Backend that the implementor use #[repr(C)] or something so that when T == () the layout of Lock<(), B> is equivalent to B::State but that seems easy

Are the functions that you need to call while holding the lock C functions? I.e. do you have this pattern?:

pub struct Foo(Opaque<bindings::foo>);

impl Foo {
    pub fn bar(&self) -> u32 {
        let ptr = self.0.get();
        let bar_lock = unsafe { addr_of!((*ptr).bar_lock) };
        let bar_lock = unsafe { Lock::from_raw(bar_lock) };
        let _guard = bar_lock.lock();
        // SAFETY: we hold the bar_lock.
        unsafe { (*ptr).bar }
    }
}

I don't think that Backend has anything to do with that, just putting #[repr(C)] on Lock should be enough.
