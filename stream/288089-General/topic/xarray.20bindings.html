Matt Gilbride (Jun 27 2023 at 00:35):

Starting a new thread to discuss upstreaming an abstraction for xarray - original patch here .

This is semi-related to Read/black tree deprecation, since we plan use xarray instead of rbtree for a use case in binder.

Obligatory thanks to @Asahi Lina who authored the original patch :clap:

@Miguel Ojeda, @Wedson Almeida Filho and I met today and came up with the following (tentative) plan:

  1. Start a new patch set with a minimal implementation. So the first commit will expose only new, replace, set, get, and remove.
  2. Add get_mut along with GuardMut (mutable version of Guard).

  3. Add find and find_mut. Find is similar to get, but returns the next larger index if there is no value at the provided index. We need this for binder.

    3a. Here we may try to encode the invariant that a lock must be acquired via xa_lock, for certain operations (get/find so far), into the type system.

  4. Add the more interesting stuff that @Asahi Lina needs: alloc, alloc_limits, reserve, and reserve_limits.

Perhaps 3a should just go with 1, even though there will only be 1 function get that uses it at that point in the series.

I'll link a GH branch in the coming days. Any other thoughts? (cc: @Alice Ryhl )

Matt Gilbride (Jun 27 2023 at 00:36):

p.s. for reference, quote from: https://docs.kernel.org/core-api/xarray.html#locking:
"If you want to take advantage of the lock to protect the data structures that you are storing in the XArray, you can call xa_lock() before calling xa_load(), then take a reference count on the object you have found before calling xa_unlock(). This will prevent stores from removing the object from the array between looking up the object and incrementing the refcount. You can also use RCU to avoid dereferencing freed memory, but an explanation of that is beyond the scope of this document."
