Redhawk said:

Hi everyone I was redirected here by Trevor and had some questions about how you guys did things.

  1. is the Rust for Linux Github a mirror? Or is code only sent through mailing lists?

The RFL Github has the branch rust-next, which is staging for the next version of the kernel. It also has PRs which are more or less a tracker for what everyone is working on, and a place to collect feedback. Actual changes need to go through the mailing list (where, if accepted, they will be applied to rust-next or the relevant subsystem's branch). Some more info here: https://rust-for-linux.com/contributing

  1. What exactly is the mission? I assume something like Mircosoft's windows or windows-sys crates. Also safe vs raw bindings

The separation is a bit different on Windows. I think that windows-sys is mostly about writing userspace programs. RFL is about being able to write kernel things in Rust, like drivers, file systems, graphics, network traffic control, etc. There is no Rust API exposed for kernel syscalls, at least not at this time :)

Windows is starting to add rust to their kernel https://www.thurrott.com/windows/282471/microsoft-is-rewriting-parts-of-the-windows-kernel-in-rust but it's not public. The most fair public comparison would probably be what you can do with https://github.com/microsoft/windows-drivers-rs is roughly what you could do with a Linux out of tree module https://github.com/Rust-for-Linux/rust-out-of-tree-module.

  1. Where do I start, a lot of the "good first issues" on Github are already covered and have pr's ready for them.

I wouldn't trust the good first issue label, I think it just hasn't been a lot of places. If you're scanning through the issues and see something of interest, just ask to get a better idea of how complex it is.

Another good way to jump in is to think of a project that interests you. The most supported thing in-tree at this point is phy drivers, and they are really pretty simple - maybe you would like to reimplement the one for your computer's phys as a getting started project. Or look through the different kernel subsystems and see if anything interests you.

If you are just interested in helping with RFL but not contributing to the kernel specifically, there are a lot of things across rustc and bindgen that are needed too and can be a bit easier to get started with: https://github.com/Rust-for-Linux/linux/issues/2

If I were to tell you what helped me out getting familiarized with the kernel, it was studying and working with userspace components. You can learn a lot of things that will apply in kernelspace, for example if you want to use the kernel crypto API, getting to know cryptography in general and libraries like OpenSSL or RustTLS would be a prior requirement. Another example would be USB, you learn about some things of the USB spec plus the libusb API and you'll be ready for using it in the kernel.
