Topic: Documentation error in `Opaque<T>`

Daniel Hast (Apr 02 2025 at 17:00):

Hi, I noticed a documentation error with soundness implications in Opaque<T> here:

No uniqueness for mutable references: it is fine to have multiple &mut Opaque<T> point to the same value.

This is incorrect; it's always Undefined Behavior to have multiple live &mut references pointing to the same value, regardless of the type of the value. The other points in the same documentation correctly describe the assumptions that are removed by Opaque's use of UnsafeCell and MaybeUninit, so I think the only direct documentation change needed is to remove that one bullet point and changing "all" to "most of" on line 257 above it.

However, if there's other Rust code that has UB from following the incorrect documentation, that'll need to be fixed too, which is why I'm mentioning this here instead of just submitting a patch for the documentation. Also, I haven't contributed to the kernel before and I'm not sure on the process for this subsystem—where would be the right place to submit a patch for this? Thanks.

Daniel Hast (Apr 02 2025 at 17:11):

The documentation for UnsafeCell seems to say otherwise:

Note that only the immutability guarantee for shared references is affected by UnsafeCell. The uniqueness guarantee for mutable references is unaffected. There is no legal way to obtain aliasing &mut, not even with UnsafeCell<T>.

If I understand correctly, with the usage of UnsafePinned in that RFC, the mutable access is still ultimately done through *mut raw pointers (which can alias), not through &mut references. So, what this means for Rust code using Opaque<T> is that memory access to the stored data needs to be via the provided Opaque::get API (which gives a *mut T raw pointer), not through &mut references.
