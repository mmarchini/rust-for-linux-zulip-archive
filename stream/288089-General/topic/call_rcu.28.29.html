Philipp Stanner (Apr 17 2026 at 07:00):

Well yes, but actually no :sweat_smile:

The issue is that sooner or later someone will want to add support for the more exotic dma_fence callbacks. We (I) currently only support the mandatory ones, get_timeline_name() and get_driver_name(), whose life time issues we can annihilate completely with owned strings.

But more exotic callbacks are sooner or later going to be implemented in Rust, and there C just returns the embedded C struct.

But the Rust driver will most likely need its generic data T in the Rust callback. So we have to pass it over.

And now what might happen that a different code path of the Rust driver drops that data, and the C backend_ops mechanism has no awareness of it, and then you get a UAF in Rust

Philipp Stanner (Apr 17 2026 at 07:02):

Thus, dropping data needs to be defered (RCU) until the C callbacks definitely cannot see data anymore, i.e. until after the fence has signaled
