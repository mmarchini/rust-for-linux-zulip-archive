Miguel Ojeda (Feb 08 2024 at 18:10):

I met with @Emilio Cobos Álvarez a few days ago, and we discussed this for a bit, including the bindgen-side solution/workaround of emitting extra code so that we can fetch the value.

The subvariations we discussed were "one pass per macro" (slow, but could work without user input), "one extra pass for all macros" (this would be fast, but would require the user to pass e.g. a regex of matching macros to evaluate that are "known to be constants" already) and "append the macros later on" (i.e. add them via a context or similar, which may be possible or not with the current API; we should also check e.g. what Cling does here).

So, in summary, bindgen would be happy with a workaround. But solving it in LLVM is, of course, the best long-term, so that would be great too.

For the LLVM side, Aaron Ballman (the Clang maintainer) told me he would be happy to review our changes if we sent them, in order to help bindgen, so that is great if someone wants to work on it.
