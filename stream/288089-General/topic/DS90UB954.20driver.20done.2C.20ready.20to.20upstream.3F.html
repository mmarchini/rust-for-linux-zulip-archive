  • I2C and regmap from the ncv6336 branch (Fabien Parent)
  • GPIO consumer (Fabien Parent)
  • device properties (Rob Herring)

I also had to add some abstractions myself:

  • GPIO consumer cansleep variant
    (boring, to avoid a warning)

  • msleep
    (boring. might conflict with ongoing work on delay abstractions, I haven't checked)

  • regmap changes
    The existing regmap abstractions required the usage of regmap fields, which the C driver doesn't do. In order to stay close to the C version, I removed that requirement and added simple read & write methods.

  • of: iterator over device tree nodes
    The device tree overlay for my hardware can have multiple child nodes dependening on how many serializers (DS90UB953) are connected. There is some inconsistency here, which might be avoidable. (advice welcome) I use the device property API for reading toplevel values, but the OF API for reading the nested values. Maybe I should only use one or the other?   • of: parse_phandle_with_args (extension of the above)

  • ArrayVec
    A stack-allocated growable array with a statically fixed maximum capacity. Basically a low-quality ripoff of the crate arrayvec. I believe _something like this_ could be generally useful in the kernel.
    My use case is this: The C function of_parse_phandle_with_args takes a pointer to a of_phandle_args as an out-parameter. This contains an array of args and an args_count. It would be unsafe to do this in Rust, since it allows reading from invalid memory. ArrayVec solves this problem and avoids dynamic memory allocation.

  • i2c: new_client_device
    This is easily the sloppiest abstraction I made. Returning a raw pointer to the driver code is surely no good. I just needed to be able to call i2c_new_client_device to set up an i2c client for each of the serializers connected to the main hardware. I have no idea what a "proper" abstraction could look like for this, so I didn't even try. But I'm happy to do that with some advice about designing the API in a way that might be approved by the relevant maintainers.

This is my first time contributing to the kernel, I would appreciate any form of review, including pointing out newbie mistakes, commit messages / structure etc.
Thanks!
