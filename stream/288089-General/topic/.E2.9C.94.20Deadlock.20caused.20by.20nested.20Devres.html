Topic: ✔ Deadlock caused by nested Devres

Boris Brezillon (Jan 30 2026 at 13:46):

We've recently been trying to solve an issue causing a deadlock on ::unbind() in Tyr, and after further investigation, the root cause seems to be nested Devres (an object wrapped into a Devres that also have Devres fields).

My understanding of the situation is that devres_release_all() moves all resources to a temporary list that's iterated to release each resource in reverse creation order. If a root devres has its T::drop() function called, the nested devres will be dropped too, and we're going to hit that path because the resource list has been emptied earlier in devres_release_all(). So we're going to wait on the devm completion, which will never happen because this item is still in the temporary and will only be destroyed later on when release_nodes reaches this item.

Dunno if it's a bug, or if nested Devres are prohibited, but if it's the latter, then we should probably document it and find a way to avoid people shooting themselves in the foot.

Danilo Krummrich (Jan 30 2026 at 16:20):

The answer is a bit longer.

The big hammer is an inner reference count, which I had in the past before getting rid of it for optimization purposes. (There is nothing wrong with the reference count, probe() and remove() are not hot paths at all and a separate allocation for a device resource does not hurt.)

However, with the rework I'm working on the specific issue with registrations having a reference count of something else that owns a Devres goes away. The only issue being left would be if you do Devres<Devres<T>>etc. which makes no sense. In fact, Devres is not a mechanism for drivers to model lifetimes. The only valid pattern is Devres<DEVICE_RESOURCE>, which effectively makes direct nesting invalid.

Therefore, once the rework is done, we would not need an inner reference count, but pay the price that Devres::new becomes unsafe, which I think is OK, since it should - with maybe a few minor exceptions - only be constructed by core code anyways.
