Topic: ✔ Embedded objects with shared refcount

David Rheinsberg (Sep 04 2024 at 10:43):

The kernel has lots of places where a ref-counted object has a unique owner, and provides exclusive features to that owner. For instance, the userdata pointers allow the owner to store back-pointers into the object, while other code accesses the ref-counted object in a generic way without ever looking at userdata. This is fairly easy to model in Rust. However, what do we do if the userdata is not a void-pointer, but a trailing byte buffer where the size is specified dynamically at object-creation, and used to directly store data of the owner?

I can easily mark the kernel object as DST with a trailing buffer, but then everything becomes a wide-pointer, which seems excessive. I'd rather be able to reduce a Box or Arc in size (stripping the trailing buffer) to turn a specificly typed reference to a generic one (stripping this trailing byte buffer from the type, or reducing it to size 0). This works fine in C, since kfree and friends don't take the total size as argument. But is this ok to do with Box? Or does anyone have an alternative?

And when designing new code in Rust, do we have a strategy how to solve this? The entire thing can be avoided by using a backpointer with userdata, but then you cannot share cachelines and allocations.
