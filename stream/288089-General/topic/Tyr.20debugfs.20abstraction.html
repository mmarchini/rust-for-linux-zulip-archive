Topic: Tyr debugfs abstraction

Alvin Sun (May 27 2026 at 06:14):

Per Danilo's v1 review [1], Tyr should attach to DRM's per-device debugfs root rather than creating its own directory. Reworked series design [2]:

  1. SeqShow trait — seq_file output via &SeqFile directly. Backed by single_open/seq_read/single_release. Needed for kernel APIs like drm_debugfs_gpuva_info that take struct seq_file *.

  2. ScopedDir::from_dentry() + Entry::from_raw() — Wrap an externally-managed dentry (e.g. DRM's per-device debugfs root). Files created under it are not auto-removed on drop — lifetime bound to the dentry owner.

  3. drm::Driver::debugfs_init callback — Optional trait method. C callback wraps minor's debugfs_root as ScopedDir and calls T::debugfs_init(data, &dir). Runs during drm_dev_register.

impl drm::Driver for TyrDrmDriver {
    fn debugfs_init<'a>(data: &'a TyrDrmDeviceData, dir: &ScopedDir<'a, 'static>) {
        dir.seq_file(c"gpuvas", &data.debugfs_data.gpuvas);
    }
}

Is the overall design sound? Specifically, are ScopedDir::from_dentry() and Entry::from_raw() the right abstractions for externally-managed debugfs directories?
