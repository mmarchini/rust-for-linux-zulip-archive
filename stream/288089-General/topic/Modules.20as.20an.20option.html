bjorn3 (Nov 01 2023 at 19:51):

Between different rustc versions we emit unique symbol names. Within a rustc version we are only guaranteed to emit unique symbol names if the -Cmetadata argument passed when compiling a crate differs. Otherwise the symbol name could be identical while still being ABI incompatible. In any case the rmeta file contains MIR for all generic and #[inline] functions to be codegened in the crate that uses it as well as on recent nighlies MIR for any function which rustc deems to be useful to inline cross-crate.

bjorn3 (Nov 23 2023 at 21:59):

https://lore.kernel.org/rust-for-linux/2023112312-certified-substance-007c@gregkh/T/#t has been posted for MODVERSIONS support with rust. Since my last reply in this topic, a major change in rustc has happened: Even without #[inline], rustc can now inline functions cross-crate if it thinks they are small enough to be worthwhile to inline. I think this effectively means that the function signature CRCs for rust functions will have to be derived from a hash of the entire crate rather than the signature of the individual function as basically any function may have been inlined and thus can't safely be swapped out between kernel versions. It should still be fine if the respective crate and it's dependencies hasn't been modified though, so hashing just the .rmeta file of the crate that defined the function should be fine I think. The .rmeta file contains the crate hashes for all dependencies already.
