Boqun Feng (Apr 30 2024 at 14:50):

https://github.com/rust-lang/unsafe-code-guidelines/issues/348#issuecomment-1221376388

How LKMM atomics are recommended to implement from Ralf:

First, keep in mind that you could simply transliterate the C versions of READ_ONCE/WRITE_ONCE, barriers, etc. directly to Rust, using ptr::read_volatile/ptr::write_volatile in place of C volatile loads and stores, and asm! in place of C asm blocks. If you do, you'll end up with the same LLVM IR instructions (or GCC equivalent with rustc_codegen_gcc), which will get passed to the same optimizer, and which ultimately will work or not work to the same extent as the C versions.

Indeed I think that is probably the best approach. It avoids accidentally limiting how we can evolve the Rust memory model (that underlies our official Atomic types) in the future, and while it does technically lead to UB in Rust, this is already the exact same situation that the Linux kernel finds itself in for C.

Boqun Feng (Nov 05 2024 at 18:55):

Please see the following example, I update it a bit:

let x: &AtomicI32 = AtomicI32::new(0);
let y: &AtomicI32 = AtomicI32::new(0);

<using Rust native atomics>         <using LKMM atomics>
x.store(1, Relaxed);
y.store(1, Release);
                                    r1 = atomic_read_acquire(y);
                                    r2 = atomic_read(x);

// expect: r1 == 1 && r2 == 0 outcome doens't exist

There are at least two things could be broken:

  • First, IIUC, two memory operations both need to be a C++ atomic primitive to establish the "synchronize-with" relationship, and without the "synchronize-with" relationship the ordering cannot establish to guarantee that "r1 == 1 && r2 == 0" never happens.
    • For example, if compilers see there is only one atomic store on y, they can be "smart" and replace the Release store with a Relaxed store, because there is no Acquire to synchronize with. Though, in real world, compilers are very unlikely to do this optimization.
  • Second, even if we somehow tell the compilers that the store of y using native atomics synchronizes with the load of y using LKMM atomics, from the Rust memory model PoV, the acquire ordering provided by LKMM is still somewhat undefined, so it's unclear whether we have the enough ordering guarantee to prevent the unexpected outcome.

Overall, Rust memory ordering model is pretty much based on an abstract machine and the code only uses Rust native atomics to synchronize, therefore it doesn't really work with another memory ordering model. Because in that abstract machine, there's no other primitives can provide ordering.

Someone suggested that Rust memory ordering model (or C++ memory ordering model) should have ABI, that means each primitive has mapping to asm code, for example riscv, because one library compiled with a compiler should work with another library compiled with a different compiler. If we have that, then maybe we can work out some reasoning at asm level. But I haven't seen any official confirmation on this for all the archs, and it's also unclear what LTO would bring to the table. So this is probably a dead end.
