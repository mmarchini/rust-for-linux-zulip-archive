Benno Lossin (Sep 24 2022 at 12:33):

Alex Gaynor said:

Yeah -- my current uses of ouroboros are all in user-space where that sort of thing is more ok :-) My understanding is that Box shouldn't be required, but they couldn't figure out how to get something right without it.

There are multiple problems associated with not introducing an indirection through Box:

  1. pointers to types with self references cannot be marked noalias (otherwise we would get incorrect optimizations).
  2. self referential types must have a stable address, or need to correct the references to them upon getting moved.

There are fixes, but they result in worse ergonomics.

  1. pointers pointing to types that are !Unpin do not receive the noalias tag.
  2. because rust has no move operator overload, we cannot really correct a moved self referential structure and are forced to use Pin. As you probably know, Pin has rather poor ergonomics.

In userspace it is generally ok to add additional indirection/allocation and ouroboros chooses it, because then the self referential types can still be moved. So users do not need to rewrite their APIs.

Benno Lossin (Sep 24 2022 at 13:45):

So essentially it would look like this:

// inner type generated by some proc macro
// this is only used by Cert
struct CertInner {
    data: Arc<[u8]>,
    raw: RawCert<'static>, // this lifetime is not actually static,, as it is only valid for the lifetime of Cert
}

impl CertInner {
    fn new(data: Arc<[u8]>, parse: impl FnOnce(data: &[u8]) -> RawCert<'_>) -> Self {
        // here we create raw with parse and a cheated 'static lifetime from data
        Self {
            data,
            raw,
        }
    }
}

struct Cert {
    inner: CertInner,
}

impl Cert {
    fn new(data: Arc<[u8]>, parse: impl FnOnce(data: &[u8]) -> RawCert<'_>) -> Self {
        Self { inner: CertInner::new(data, parse) }
    }
}

So all in all very similar to ouroboros, but without the extra allocation.

Benno Lossin (Sep 25 2022 at 07:25):

@Gary Guo do you know any situation where we would need the address of the initialized type to decide whether the initialization fails? If not, we could just move the evaluation of the fields out of the closure (this could lead to better optimizations, though I am not sure):

#![feature(allocator_api, new_uninit)]

use std::alloc::AllocError;

pub unsafe trait Init<T> {
    unsafe fn __init(self, place: *mut T);
}

struct InitClosure<F>(F);

unsafe impl<F: FnOnce(*mut T), T> Init<T> for InitClosure<F> {
    unsafe fn __init(self, place: *mut T) {
        (self.0)(place)
    }
}

impl<F> InitClosure<F> {
    unsafe fn from_closure<T>(f: F) -> Self
    where
        F: FnOnce(*mut T),
    {
        Self(f)
    }
}

macro_rules! init {
    ($t:ident {
        $($field:ident: $val:expr),*
        $(,)?
    }) => {{
        $(let $field = $val;)*
        let init = move |place: *mut $t| {
            unsafe {
                $(::core::ptr::write(::core::ptr::addr_of_mut!((*place).$field), $field);)*
            }
        };
        #[allow(unreachable_code)]
        if false {
            let _: $t = $t {
                $($field: ::core::todo!()),*
            };
        }
        let init = unsafe { $crate::InitClosure::from_closure(init) };
        init
    }};
}

pub trait InitPtr<T>: Sized {
    fn init(init: impl Init<T>) -> Result<Self, AllocError>;
}

impl<T> InitPtr<T> for Box<T> {
    fn init(init: impl Init<T>) -> Result<Self, AllocError> {
        let mut this = Box::try_new_uninit()?;
        unsafe {
            init.__init(this.as_mut_ptr());
        }
        Ok(unsafe { this.assume_init() })
    }
}

#[derive(Debug)]
struct Demo {
    a: Option<Box<Demo>>,
    b: u8,
}

impl Demo {
    fn new() -> Result<impl Init<Self>, AllocError> {
        Ok(init!(Self {
            a: Some(Box::init(init!(Self { a: None, b: 4 }))?),
            b: 3
        }))
    }
}

fn main() -> Result<(), AllocError> {
    let demo = Box::init(Demo::new()?)?;
    println!("{demo:?}");
    Ok(())
}
