Topic: Bindgen on the C API of Rust Modules

David Rheinsberg (Apr 01 2026 at 11:18):

Hey!

I have a Rust module that exposes a C API for the wider kernel to use. I guess a similar scenario will happen when DRM utilities are written in Rust but are exposed to C drivers. While it is rather straightforward to expose the C entry-points in Rust, I would love to get an automatic verification that it matches the prototypes in the C header.

What I did in the bus1 patches is write a C header with the expected C API exposed by the Rust driver, and then run bindgen on it to be able to use those exact symbols in Rust. This avoids duplication of API constants, and even allows verifying function prototypes via static assertions (though I did not include that in the PR).

I wonder whether anyone did something similar, and whether this is how we want this to be done?

Some issues along the way:

  • I copied the bindgen invocation almost verbatim, since the makefile variables are not exported. I don't want the private module symbols to end up in kernel::bindings, as it seems kinda unrelated to it. And this allows recompiling a module without changing the kernel objects (very convenient for hot-reloading during development). Is there some strict requirement to run bindgen only once? Or is it suitable to export the bindgen command so other modules can run it as well?
  • I could not figure out a good way to expose Rust objects other than in a box. Some objects would preferably live on the stack to avoid the overhead of the allocation. But I could not figure out a decent option to get a suitable object size to use in the C header. I could certainly just pick a large size and alignment, but for some objects the selected kernel configurations and architecture make a big difference.

Ideas welcome!
David

Alice Ryhl (Apr 01 2026 at 11:20):

For constants, the Rust code should use the value defined in the header via bindgen. For functions, you should use the #[export] macro that I already added for checking that the signature is correct.

Alice Ryhl (Apr 01 2026 at 11:22):

We don't have a good solution for struct sizes, as headers can't really depend on Rust code. You have to guess/overestimate with an assertion on Rust side.
