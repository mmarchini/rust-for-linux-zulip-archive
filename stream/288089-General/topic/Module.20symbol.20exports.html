Alex Gaynor (Dec 31 2021 at 13:59):

So far all the APIs we support are places there are existing kernel APIs for, and we expose C-ABI functions from Rust, and function-pointer structs (e.g. file_operations).

Which is to say the kernel C code never calls any Vec<T> methods, those are called exclusively from Rust code.

Miguel Ojeda (Dec 31 2021 at 14:35):

So if a module wants to call something on Vec<>, it ends up going via the core kernel, right?

Yes, core, alloc, etc. go into the "shared" Rust support.

Does that mean you end up with a larger kernel because you can't prune out parts of the core library that aren't called (because you don't know what a module will end up using)?

Yeah, but you can also see it the other way around: it allows us to have a smaller kernel, since the idea is that eventually many modules will reuse the same code.

In order to take care of the "rest" that we do not expect anybody using (or that we do not want anybody to use), we are also trimming alloc in our fork (no collections, no sync, no rc...), and we also requested upstream a few "modularization" options for core (some already in, e.g. no_fp_fmt_parse).

Alex Gaynor (Dec 31 2021 at 15:23):

When compiling core (or any crate for that matter) rustc emits the IR for generics methods that can be read by rustc when compiling something downstream and the method will be code-gen'd then if it wasn't already codegen'd inside core.
