Asahi Lina (Oct 02 2022 at 04:06):

I think the crucial point from my thread is that safety/soundness is a property of interfaces. So it is something that can be more clearly defined for the kernel crate, since that is a bunch of interfaces for modules to use. But within any given piece of code, as soon as you introduce one unsafe block, that can impose requirements on other safe code. That's already the case in normal Rust. For example, I was reading the Pin docs earlier today here:

The struct must only be Unpin if all the structural fields are Unpin. This is the default, but Unpin is a safe trait, so as the author of the struct it is your responsibility not to add something like impl<T> Unpin for Struct<T>. (Notice that adding a projection operation requires unsafe code, so the fact that Unpin is a safe trait does not break the principle that you only have to worry about any of this if you use unsafe.)

Having unsafe in one place means doing something safe in another place can turn the design unsound, and make UB possible.

So in the kernel, it's reasonable to say that interfacing with hardware is unsafe, because we can't know what hardware will do. That's already the case (IoMem is unsafe to instantiate). But we don't have to make every little operation unsafe, since once the programmer uses unsafe once to get at a hardware interface, it is their responsibility that the rest of the code is using it safely. There's no need to plaster unsafe everywhere, you only need to make sure the user has to use unsafe at least once.

That doesn't mean we should fully minimize the number of APIs that are unsafe. For example, even if the design of an abstraction guarantees that the user already used unsafe to instantiate it, there can be some operations that are "extra unsafe" for some reason. Like, something that's more likely to be a footgun for users, or which imposes extra safety requirements above those imposed by the initial use of unsafe. And in those cases, it makes sense to have that API be unsafe too.

Once you get into driver design, the challenge is to compartmentalize things as much as possible and make safety something that you can evaluate for small pieces of code. But for something like a GPU driver, the interactions are complex enough that there are safety aspects that span the whole driver. And in that case, I think the goal should be to make those safety aspects a property of the overall driver design, in a way that it's possible to evaluate at a high level and doesn't mean you need to audit all the code in detail and guarantee it's correct. So, safety is tiered: some things (especially those tied to individual unsafe blocks) enforce safety locally and are easy to audit at a low level, and some things are high-level concepts where you can audit safety by just understanding the design, without digging into the details of the implementation too much.

In the end, I think we can agree that the goal should be to minimize bugs. You have to use unsafe to write drivers, so there is no way to make any hard guarantees like there is with userspace Rust. But Rust's tools let us greatly reduce the surface where bugs appear, if used properly, and that's what makes it an awesome language for drivers too. It's all about the balance between ergonomics and "scary markers" (like unsafe), so that us developers can comfortably write good code!

Put another way: if your code has zero unsafe blocks, then you can make arbitrary changes and it will remain safe. If your code has one unsafe code, that guarantee is instantly lost. The unsafe code might be sound, it might invoke UB, or it might be sound only if the rest of the code does certain things a certain way. This is, and has always been, how unsafe Rust works. My understand of the discussion (or what I learn most from the discussion) is the "style" of unsafe, i.e. pre-condition unsafe vs post-condition unsafe.

Most of the unsafe marking is pre-condition style, that is: unsafe marks the code that may directly invoke UB under some conditions, for example, dereferencing a raw pointer. The advantage of this is you can always check these conditions at unsafe invocations. I guess people all agree that pre-condition unsafe is preferred

Unlike pre-condition, post-condition unsafe is that if you have an invocation of the unsafe API/function, you will need to take care of all the following related (even) safe code as well to guarantee soundness.
