Topic: Builtin Rust components fail to build with small code model

Ard Biesheuvel (Dec 26 2025 at 08:54):

As part of my effort to make the x86 boot path more generic and more idiomatic, ELF based etc, I am trying to switch to the 'small' code model for the core kernel when CONFIG_RELOCATABLE=y. This allows the kernel to be linked as a PIE executable, like other sane architectures do, rather than rely on dodgy hacks to perform runtime relocation on a position dependent binary. The ultimate goal here is to allow things like fgkaslr (fine grained kaslr) to be designed/implemented only once, in the ELF domain, and be deployed across architectures.

Currently, when building with -Ccode-model=small, I get an error like

Ard Biesheuvel (Dec 28 2025 at 19:09):

Gary Guo said:

I feel like this should be a new relocation-model rather than an ad-hoc flag.

Agreed, and this has come up in the past for the C side of the kernel too, but we never got anywhere. Something like -fstatic-pie would be useful to convey that the executable needs to be relocatable at load time, but does not use shared objects. Unfortunately, in the ELF world, things like symbol preemption, dynamic loading and minimizing the CoW footprint of an executable are all piled together.

So yes, something like -Crelocation-model=static-pie which results in PIC codegen but no GOT indirections would be useful too. Only potential problem are weak references, but does Rust even support those?
