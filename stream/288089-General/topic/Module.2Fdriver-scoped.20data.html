Topic: Module/driver-scoped data

Gary Guo (Dec 17 2025 at 17:49):

This summarizes my recent thoughts on the topic

What's needed

The ability to define data shared among a driver (or module), which are accessible inside probe callbacks. Or (even better), accessible to most part of driver code.

An example would be creating a debugfs directory on driver load, and each individual probe can add a subdirectory.

Another example is the existing global_lock! today, but this is trying to find a more generic approach to all data (that potentially need Drop, too).

Approaches

Unsafe

One can define static mut or create a static that is uninitialized unsafely, and unsafely initialize it.

The issue for this is that unsafe initialization is needed, and manual drop is needed in the impl Drop for Module, as statics don't have their destructor executed.

Put it inside struct that impl Module

One immediate thought is to put the data inside the struct that implements module, e.g.

struct MyModule {
    dir: kernel::debugfs::Dir,
}

and possible expose a way to get data associated with the current module (while performing a check to see if data is actually implemented).

However, this is problematic as you want to access data before module init is completed. probe can be called before module initialization is done, and at that point the MyModule is not yet fully initialized.

Partially accessing the module fields probably need major rethink and re-work for pin-init.

OnceLock-like approach

Having a boolean flag (static key, if needed) and use OnceLock-like approach is a plausible way. This ensures that data can be safely accessed and initialized.

However this does not handle the destruction part. This can be solved in various ways:

  1. Add dtor support to module system

    * Currently module system support ctor, but only supports a single exit function

  2. Keep a linked list of dtors to run within the module, and have the generated glue for module walk through the destrutor list and execute them.

Discourage globals -- passing pointers around always?

This is problematic for a couple of reasons:

  1. Driver's probe callbacks today do not receive data pointers

    * In C code, people just use globals, so there isn't any case where the registration would need to hand some data across to the probe fn.

  2. Performance

    * You'd need to use an Arc to wrap the data
    * The access becomes indirect access instead of global loads

Gary Guo (Dec 17 2025 at 18:14):

Actually, I have an interesting idea.

It contains a few parts (skip name bikeshedding for now):

  1. A macro that defines a global and unique type for each global (somewhat similar to lazy-static), let's call them FOO
  2. A token type, GlobalOwned<G> which represents that a global has been initialized. Dropping it runs the destructor of global. This will be stored in the module data. So OwnedToken will be stored.
  3. A token type, GlobalRef<'a, G> which will deref to the global's storage. This can be derived from GlobalOwned<G> and associated with the lifetime of GlobalOwned<G>. Borrowck will ensure that while we're holding the ref, the global has been initialized and not yet dropped.
  4. The module will pass GlobalRef to driver when registering. This is a ZST, so it does not actually have to exist in memory. Hence, probe callback and obtain it from a thin-air (minus some caveat similar to Matthew Maurer 's trick discussed in Kangrejos)
  5. Module can then access the global using this token type inside probe callback. It can also further pass it to other data if it wishes.

Drawback: GlobalRef has a lifetime param, so would all structs that reference it.

I am not sure whether I want this approach or the OnceLock-like approach

Re: discouraging globals, could we put references to statics into the match table, and pass the "global" in through the IdInfo parameter?

Since we use macros to declare our match tables already, it would also be feasible to have them generate initialization/destruction code for table-attached statics. We could then make it so that the driver traits being implemented expect an init function to be called before using the table and a drop function to be called when the driver is being released.

The main downside I see for this is that OF vs ACPI-based stuff wouldn't share declarations of the statics.

This doesn't solve global data initialization/dropping for non-devices, but re-using the table scope seems a good way to avoid globals.

Danilo Krummrich (Jan 15 2026 at 20:18):

The point is, there are multiple things that we call drivers, i.e. in C we call the structs to register a driver e.g. struct pci_driver, but we also call the implementation of the callbacks of e.g. a struct pci_driver a driver.

For instance, nouveau is a driver (implementation), nova is driver (implementation), etc.

So, let's look at:

struct NovaCore {
    fw: Firmware,
    ...,
}

impl pci::Driver for NovaCore {
    fn probe(pdev: &pci::Device<Core>, _info: &Self::IdInfo) -> impl PinInit<Self, Error> { ... }
}

HereNovaCore is an implementation (or a specific type) of a pci::Driver. When a new pci::Device is now detected in the system then this PCI device is matched with the driver and probe() is called. So, probe() really is the constructor of a driver type. I.e. we create an instance of this driver type, now that we have to store information for this specific device that we have been matched with.

So, probe() then returns an instance of the NovaCoredriver type. This instance represents the binding of the NovaCore driver type and a PCI device, which we also store in this PCI device.

Alternatively - and I think this is what you propose - we could have something like this:

struct NovaCore;

struct NovaCoreData {
    fw: Firmware,
    ...,
}

impl pci::Driver for NovaCore {
    type Data = NovaCoreData;

    fn probe(pdev: &pci::Device<Core>, _info: &Self::IdInfo) -> impl PinInit<Self::Data, Error> { ... }
}

But what does this improve? We now have a useless empty type NovaCore that we use for nothing but implement pci::Driver for. And we represent the instance of the NovaCore driver as a separate NovaCoreData type. Seems pretty useless and eventually people will most likely just end up just doing this:

struct NovaCore {
    fw: Firmware,
    ...,
}

impl pci::Driver for NovaCore {
    type Data = Self;

    fn probe(pdev: &pci::Device<Core>, _info: &Self::IdInfo) -> impl PinInit<Self, Error> { ... }
}

Danilo Krummrich (Jan 16 2026 at 10:55):

Gary Guo said:

This makes creation of the driver singleton explicit -- you create it when you register your driver. Just like when C code creates a pci_driver. It also makes sure that we don't overload the name "driver".

The misunderstanding relies in the fact that a struct pci_driver does not define or represent the driver as a whole.

A driver conceptually consists out of much more: the implementation of the bus device callbacks, implementation of the class devices callbacks, IRQ registrations, I/O mappings and other physical device resources. All those are instantiated on a per-device basis.

The struct pci_driver is only the structure a (PCI) driver type is registered through in the driver core. I.e. it does not represent the driver as a whole, but only one single instance of a registration of a driver. So, yes an instance of struct pci_driver is kind of a singleton (at least from the perspective of a certain driver, e.g. nova-core). The only thing you ever do with this structure is to make a specific driver type present in the system by registering it. This is why we have the driver::Registration type in Rust.

(And yes, you could probably argue that struct pci_driver should rather be named struct pci_driver_registration instead.)

But the driver as a whole (e.g. nova-core, nouveau, etc.) is - through its per-device state - instanciated per-device and is not a singleton.

Think about it, when we say "the nouveau driver", we do not only mean the struct pci_driver which is used for registration in the bus and in the driver core. We mean the whole thing, with all of its code, device resources and a per-device state, i.e. a per-device instance.

In ancient times, drivers in the kernel did not have a per-device state and a lot of drivers kept their data in globals, i.e. not even per driver registration, but per module.

One of the big goals of the introduction of the driver-model back then was to fix this by introducing instances of drivers that are created per-device (i.e. a per-device state), such that it became transparent to drivers how many times they are instanciated by the driver core when multiple matching devices were found in the system.

Now, back to your proposal:

What you are proposing to add is per-driver-registration data. This sits quite in between per-module and per-device data. I.e. a module can have multiple driver registrations.

For instance, the nouveau module does have (and the nova-core module will have) a platform driver registration and a PCI driver registration, since the driver can support hardware on both busses.

This shows why the solution of per-driver-registration data is not the solution of the problem. The problem is that we really want to support accessing per-module data. Introducing per-driver-registration data instead is just a workaround.

Surely, you could continue the game and reference count the data and pass it to multiple driver registrations, but what are you doing for modules that do not even connect with the driver core at all, i.e. modules that do not have driver registrations in the first place? One example for this is binder. How can binder access per-module data?

To use hazard pointer for revocable:

On the accessor side:

  1. put pointer into a hazard pointer slot
  2. check that the data is not revoked

On the revoker side:

  1. update revocation flag
  2. perform hazptr_synchronize to wait all users of the pointer to go away
  3. free the revoked data
