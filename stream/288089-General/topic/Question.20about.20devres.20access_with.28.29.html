Danilo Krummrich (May 12 2025 at 15:37):

Hi @Joel Fernandes,

What do you mean with "to address #1"? What does it relate to?

In general: Yes, Devres::access() can only be used from a scope where you have a &Device<Bound>.

Note that this is sufficient, since it is guaranteed that during the lifetime of such a reference the device can't be unbound. Obviously probe() is a good example for providing a &Device<Bound>, since the device can't be unbound as long as probe() runs.

It is not possible to have an Arc<Device<Bound>> (note that the compiler won't let you by design), since it would be impossible to guarantee that for the duration of the lifetime of an Arc<Device<Bound>> the device is not unbound, since one could extend the lifetime of an Arc<Device<Bound>>infinitely.

However, the plan is to provide a &Device<Bound> reference whenever possible. For instance, IRQ abstractions can do so from an IRQ handler, since we can (and have to) make it an invariant of the IRQ abstraction that the IRQ handler is removed before the device is unbound, hence for the duration of an IRQ handler we can provide a &Device<Bound>. The same would be true for SYSFS abstractions or, depending on the particular subsystem, subsystem callbacks and / or IOCTLs.

We can even implement "fenced" variants of workqueues, etc. which guarantee that they're teared down before the corresponding Device is unbound and hence can provide a &Device<Bound> reference.

Danilo Krummrich (May 12 2025 at 15:51):

Is it possible to hold a (Arc) reference to Device<Bound> and then we wait for reference count to drop to 0, before allowing device unbinding or does that throw a wrench in device driver unbinding?

No, the lifetime of a struct device is entirely independent from its state of being bound to a driver. It'd also be the wrong solution to forcefully keep a device bound to a driver. The bound state is something the driver has to be capable of to react to, it's nothing a driver should be allowed (or be capable of) controlling -- that's up to core code.

Danilo Krummrich (May 13 2025 at 21:01):

try_access_with() itself is never the problem, it just catches bugs that we also catch in DRM C drivers with drm_dev_enter() and drm_dev_exit().

More general, if you can't prove that the device can't be unbound while using I/O memory, i.e. you're in a call path where the device can be unbound concurrently, we ultimately need some kind of protection.

Think of it in C. Let's say in probe() you call pcim_iomap() and store the base address in some reference counted structure, e.g. your driver specific DRM device.

Think of what happens if in a call path where the (PCI) device can be unbound concurrently (e.g. DRM ioctl()) you just access the I/O memory -- it potentially faults.

DRM ioctls() are prone to that, hence they protect things with drm_dev_enter() and drm_dev_exit(), which in the end is a specialized version of Devres and try_access_with().

Besides that, I doubt that an RCU read side critical section for MMIO accesses becomes a relevant cost. And as mentioned, I'm pretty sure that we can almost always avoid that in nova-core anyways.
