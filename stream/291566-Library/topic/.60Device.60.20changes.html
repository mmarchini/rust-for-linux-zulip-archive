Wedson Almeida Filho (Apr 07 2023 at 18:13):

Gary Guo said:

The commit that changes Device from a pointer wrapper to a struct wrapper is here. Miguel Ojeda Wedson Almeida Filho Asahi Lina

With this change we can get rid of RawDevice trait. This change, though, would depend on AlwaysRefCounted which isn't upstream yet, so I can't send the patch to the mailing list right now.

We discussed this before in a meeting and I really liked the idea of using AsRef. I started making some changes on top of this but now I don't think AsRef is enough for our needs: we need the returned &Device to actually be related to the implementer of AsRef<Device>. But with AsRef, it would ok for an implementer to a returned a pointer to say some unrelated static Device.

This precludes somethings like:

unsafe impl<T: AsRef<Device>> AlwaysRefCounted for T {
    fn inc_ref(&self) {
        unsafe { bindings::get_device(self.as_ref().0.get()) };
    }

    unsafe fn dec_ref(obj: core::ptr::NonNull<Self>) {
        unsafe { bindings::put_device(obj.as_ref().as_ref().0.get()) };
    }
}

So I'm thinking we need a trait like AsRef<T> but that is unsafe and has the safety requirement that the returned &T be part of the implementer. We could all it IsA<T>, Contains<T>, as AsContainedRef<T> or anything to that effect.

The safety requirement would be similar to what we have in RawDevice now, that is:

/// Implementers must ensure that the `*mut device` returned by [`RawDevice::raw_device`] is
/// related to `self`, that is, actions on it will affect `self`. For example, if one calls
/// `get_device`, then the refcount on the device represented by `self` will be incremented.

WDYT?

Boqun Feng (Apr 07 2023 at 18:44):

Do we want to go further: making Contains<T> provide the offset of the field? We can have a container_of function if we do so.

/// Implementers must ensure that the struct contains a `T` in it, and `offset_of` is the internal offset
/// from the beginning of the type to the field whose type is `T`
pub unsafe trait Contains<T>: Sized {
  fn offset_of() -> isize;
  fn container_of(inner_ptr: *const T) -> *mut Self {
      (inner_ptr as *const u8).wrapping_offset(-Self::offset_of()) as *mut Self
  }
  fn as_ref(&self) -> &T {
   // SAFETY: Per safety requirement of `Contains<T>`,  `self` + `Self::offset_of()` points to a valid `T`.
    unsafe { &*((self as *const Self as *const u8).wrapping_offset(Self::offset_of()) as *const T) }
  }
}

bjorn3 (Apr 07 2023 at 23:24):

It doesn't because the implementer is not required to return a contained object. The implementer may return some unrelated &Device.

It would be unsound as it will cause another object to be affected by reference counting than the one ARef points to. For example if you store a device with a single reference in a static and then return this device in AsRef for your own type when dropping an ARef, the device would be deallocated despite there still being a reference stored in the static.

Wedson Almeida Filho (Apr 08 2023 at 01:32):

Gary Guo said:

Wedson Almeida Filho said:

Gary Guo said:

It's very likely that we don't want a blanket impl for AlwaysRefCounted.

We do for devices.

Why?

Because then when we wrap some C struct that contains a struct device (e.g., struct pci_dev, which we'd call pci::Device), then we only need to implement Contains<Device> (or whatever we decide to call it), and we'd then automatically be able to use ARef<pci::Device> without having to explicitly implement AlwaysRefCounted for pci::Device. This, in fact, is how things work on the C side: structs that embed a struct device have their lifetimes dictated by the refcount stored in the struct device field.
