John Baublitz (Jul 08 2022 at 19:03):

Wedson Almeida Filho said:

I don't know much about neli, but given that it's for user space, I assume it's more focused on the client slide of netlink. In the kernel we care more about the server side; do you have a sense of how much overlap there is between the two cases?

in terms of shared code, there is some. for example, a lot of the code is defining interfaces for handling data structure serialization and deserialization. the socket code will be virtually useless in the kernel version but the data structures may be useful. I specifically opted against bindgen for my library which has some benefits and drawbacks. the major benefits are around things like:

  • constants: they're all defined as enums so it makes the usability a little bit better in terms of knowing what constants to use for what arguments, etc.
  • data structure deserialization: I've caught bugs in the kernel before with my library because the parsing ensures that packets are valid according to the outlined protocol. because I'm using higher level data structures, it basically does verification on much of the packets during deserialization to ensure they're valid.

the major drawback is pretty obvious, having to update data structures in two places and maintain two separate codebases.
