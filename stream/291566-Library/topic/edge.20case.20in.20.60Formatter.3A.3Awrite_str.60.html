Topic: edge case in `Formatter::write_str`

Valentin Obst (Jan 12 2024 at 07:56):

While reading the str module I maybe came across some potentially unexpected behavior in the interplay of Formatter::write_strand RawFormatter::write_str. The latter nevers fails a write but sets the internal pos past the end such that the former can detect the lacking space. However, this does not work then end is usize::MAX and the write overflows. I must admit its pretty contrived thus I'd like some fb if inserting a check would make sense (and if the observation even applies or if there is smth I overlooked). (playground)

use core::fmt;

struct RawFormatter {
    _start: usize,
    pos: usize,
    end: usize,
}

impl RawFormatter {
    fn write_str(&mut self, s: &str) -> fmt::Result {
        let pos_new = self.pos.saturating_add(s.len());
        let len_to_copy = core::cmp::min(pos_new, self.end).saturating_sub(self.pos);
        if len_to_copy > 0 {
            println!("Would copy: {}", len_to_copy)
        }
        self.pos = pos_new;
        Ok(())
    }
}

struct Formatter(RawFormatter);

impl Formatter {
    fn write_str(&mut self, s: &str) -> fmt::Result {
        self.0.write_str(s)?;

        if self.0.pos > self.0.end {
            Err(fmt::Error)
        } else {
            Ok(())
        }
    }
}

fn main() {
    let mut f = Formatter(RawFormatter {
        _start: usize::MAX - 1,
        pos: usize::MAX - 1,
        end: usize::MAX,
    });
    assert_eq!(f.write_str("Hello, world"), Ok(()));
    assert_eq!(f.write_str("\n"), Ok(()));
}

Benno Lossin (Jan 12 2024 at 15:46):

Note that the usizes in RawFormatter are actually pointers, so they generally will not be near usize::MAX. I thought about rewriting it to use pointers, but haven't had the time.

Yea, as long as no weird combination of machine and configuration decides to allocate memory at the very top of the address available address space everything is fine (not sure if there are any guarantees that this cannot happen though). Thats why I called the example contreived (not so sure if thats the correct word), but maybe a rewrite could still handle this case consistently, i.e., return an error instead of claiming that the operation worked.
