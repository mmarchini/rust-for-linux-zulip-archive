Miguel Ojeda (Jun 16 2021 at 08:19):

Yeah, it is the middle ground indeed :)

Miguel Ojeda (Jun 16 2021 at 08:22):

We need to be in-tree because 1) we need anyway to use some features like alloc does for e.g. a custom Arc, 2) we want to have freedom (at least in the beginning) to be able to customize as needed and 3) those customizations should be fairly easy to iterate on, without having to wait for the Rust side to implement them.

Miguel Ojeda (Jun 16 2021 at 08:24):

On the other hand, we do not want to diverge too much because 1) we would need to maintain it ourselves, 2) we want to reuse it (ideally as-is, eventually), 3) alloc currently uses quite a few compiler features that are not going to become stable soon or ever (but ideally we would want it to be a "normal" crate).

Miguel Ojeda (Jun 16 2021 at 08:25):

So the idea is having a middle-ground where we do customize alloc, but trying hard to keep it close enough to upstream so that eventually alloc can gain the features/customization points that we need; plus we can more easily synchronize bugfixes etc. from upstream.

Miguel Ojeda (Jun 16 2021 at 08:34):

Santosh Sivaraj said:

This might be a dumb question, how can we tell vec! or something else to use vmalloc allocator (/once it is available/)? How the same driver can use both vmalloc and kmalloc APIs?

So we will have several allocators to use as needed, which you can mix (i.e. one different allocator per instance of each data structure -- that is feature(allocator_api), similar as the C++ STL approach, i.e. you parametrize the type).

For macros like vec!, we can either 1) have custom versions, 2) or we modify the macros to allow to pass the allocator, 3) or we do not use them (I'd say in most cases they are not needed; and where needed, one can workaround it).

Wedson Almeida Filho (Sep 30 2022 at 20:48):

It's great to see that that things are converging, and this is definitely something we do want.

Part of the reason we have the fork is to allow us (rust-for-linux) to experiment without having to wait for upstream Rust. We are not done with the experimentation: in particular, we need to settle on how we're going to handle the different types of allocations needed in the kernel (e.g., the gfp_t flags to kmalloc).

One option suggested by a kernel developer was to infer the allocation type from context (which would allow us to keep most Rust interfaces as they are), but that was shot down by Torvalds, so we may need some interface changes. I believe Miguel was referring to uncertainties of this kind when he said we don't know how we'll use liballoc in the future.

IOW, I'm afraid we may still have a little bit of extra divergence before we fully converge. But once we get there, we'd be more than happy to do away with this fork.
