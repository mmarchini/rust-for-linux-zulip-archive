Topic: Extending vtable macro to provide unique address

Gary Guo (Jun 27 2024 at 10:23):

I am thinking of doing this:

#[vtable]
trait Op {
    #[unique]
    const VTABLE: &'static OpVTable = &OPVTable { ... };
}

#[vtable]
impl Op for Foo {
    #[unique]
    const VTABLE: &'static OpVTable;
}

expanding to

trait Op {
    const VTABLE_IMPL: OPVTable = OPVTable { ... };
    const VTABLE: &'static OpVTable;
    const VTABLE_USE_UNIQUE_ATTR: ();
}

impl Op for Foo {
    const VTABLE: &'static OpVTable = {
        static VTABLE_IMPL: OpVTable = <Foo as Op>::VTABLE_IMPL;
        &VTABLE_IMPL
    };
    const VTABLE_USE_UNIQUE_ATTR: () = ();
}

This only needs const_refs_to_static feature to work (covered by project goal).
I think specifying the type name again is needed if we don't also want to use associated_type_defaults.

In userspace at least non-mutable statics that contain relocations get placed in .data as .rodata is made read-only before relocations are processed. Rustc by default enables RELRO though, which places them in .data.rel.ro which is part of the data segment, but has an instruction for the dynamic linker to make it read-only after relocations are processed. I don't know how it works in the kernel though.
