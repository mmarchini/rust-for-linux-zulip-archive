Topic: Inconsistency of `from_raw`.

Boqun Feng (Jul 10 2024 at 01:56):

We have ARef::from_raw(), which doesn't increase the refcount itself, i.e. it takes away the reference count of the input pointer. However, Device::from_raw() will increase the refcount of device, i.e. it uses its own reference count.

I find this may be a bit confusing in the future, hence bring this up. Thoughts? @Danilo Krummrich

Alice Ryhl (Jul 10 2024 at 12:27):

I would say that for from_raw, it shouldn't increment the refcount for consistency with te standard library Arc::from_raw.

Danilo Krummrich (Jul 10 2024 at 15:52):

I agree with Alice, but for a different reason. I'd argue that ARef::from_raw takes ownership of the reference that is passed in. In Device we need an extra increment of the refcount, because we don't own the reference of the raw device pointer that is passed to us, but instead are only guaranteed that it's valid for the duration of the function to increase the reference count to take our own one.

Boqun Feng (Jul 10 2024 at 16:30):

Yes, so a -> ARef<_> function may have at least two semantics: 1) using the reference count provided by the caller, or 2) using an own reference count. I feel we better have some general naming rule to separate these two cases.

Boqun Feng (Jul 10 2024 at 16:32):

For example, case #2 could have a get_ prefix indicating the refcount increased.

impl Device {
    /// Gets a refcounted reference to [`Self`]
    ///
    /// # Safety
    /// `ptr` must be a valid pointer with at least one reference count until the function return.
    pub unsafe fn get_from_raw(ptr: *mut device) -> ARef<Self> {
        // SAFETY: Per function safety requirement, `ptr` is a valid pointer and has at least one
        // reference count until the function return, which is larger then the scope of the `&Device`
        // created by `Device::as_ref(ptr)`.
        unsafe { Device::as_ref(ptr) }.into()
    }
}
