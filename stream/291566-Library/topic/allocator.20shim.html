Topic: allocator shim

bjorn3 (Jul 03 2021 at 16:56):

Rust-for-Linux currently defines the __rust_* methods that are normally defined by the allocator shim in rust. As this is an unstable implementation detail, I tried to remove the need for this: https://github.com/rust-lang/rust/pull/86844 I have no idea if it will be accepted as it is technically a breaking change when using libstd as dylib and #[global_allocator] in the main executable or another dylib. Previously it was ignored, now it may be used depending on the way the dynamic linker resolves the __rust_* methods in case of duplicates.

bjorn3 (Jul 04 2021 at 10:28):

Looks like the GlobalAlloc impl doesn't implement alloc_zeroed or realloc. Instead the default implementations are used for them which manually zero the data cq alloc+copy+dealloc the data. This is currently not an issue as only the __rust_* functions are called, which do have them directly implemented using the kernel allocator, but when the allocator shim is gone, this will become a perf issue.
