Lyude (Feb 15 2024 at 19:54):

Hi! I've been writing up a rust interface for KMS drivers written in rust, and little while ago I had asked for some advice regarding https://rust-for-linux.zulipchat.com/#narrow/stream/415254-DRM/topic/Early.20drm_plane_state.20bindings.20feedback
I was skimming through the wiki again this morning and I think I answered my question, mainly it looks like what I probably want for this is some sort of LockedBy equivalent that can handle ww_mutex so that I can make structs like Plane<T> opaque until the caller can actually prove they hold the right locks to access it's contents.
I'm now looking at implementing wrappers for ww_mutex, but since this lock acquisition scheme is not infallible (see https://lwn.net/Articles/548909/ ) it doesn't seem like sync::lock::Backend would be suitable for this. Which brings me to ask: if I'm writing such wrappers, should I start by introducing a new variant of sync::lock::Backend designed for locks with fallible acquisition schemes? Or should I just go straight to writing a wrapper for ww_mutex, and worry about moving this stuff out to a trait later if we end up finding more users?

Boqun Feng (Feb 15 2024 at 21:10):

I don't think that the current lock design can cover ww_mutex. One important in ww_mutex is the ww_acquire_ctx, and Rust API will need to design the interface where ctx interact with mutex. Also what ww_mutexAPI implies is the user somehow tracks all the acquired locks (so that they can be unlocked when "wound"), for a safe API, I think you will need to maintain that in an vector-like data structure in the ctx.
