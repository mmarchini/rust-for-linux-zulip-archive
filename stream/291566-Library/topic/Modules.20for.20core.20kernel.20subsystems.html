Topic: Modules for core kernel subsystems

Asahi Lina (Sep 28 2022 at 04:55):

What do we do for kernel subsystems which have components which can be built as modules? Right now I have kernel.drm which is gated on CONFIG_DRM which is binary, but then I have kernel.drm.gem.shmem which depends on CONFIG_DRM_GEM_SHMEM_HELPER which can be built as a module. What happens to the Rust abstraction in this case, where should it live? We also need a separate helpers.c for this part, since I need some helpers for this header and they won't link against the core kernel if shmem support is built as a module...

Miguel Ojeda (Sep 28 2022 at 11:38):

The plan is to split the kernel crate and introduce a dependencies DAG, having independent bindings header, C helpers, etc. for each crate/subsystem/module, as well as being able to put the sources anywhere in the tree too.

Asahi Lina (Oct 03 2022 at 17:13):

It'll be a long while until the driver is mainlined, and for the time being we can just make it depend on CONFIG_DRM_GEM_SHMEM_HELPER=y, but having that built-in dependency still there when it is submitted will probably raise some eyebrows, so it'd be nice to have this solved by then ^^
