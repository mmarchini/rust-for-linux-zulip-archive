Topic: Address types

Asahi Lina (Sep 27 2022 at 10:20):

Hi! I'm dealing with a lot of memory-management code for the GPU driver, and one thing that keeps coming up is what types to use for addresses of different kinds (physical, CPU virtual, GPU virtual, and possibly others). Right now I have a mix of u64 and usize that doesn't make a whole lot of sense. I think the Linux kernel isn't very good at typing addresses on the C side, and I wonder if we should just define our own standard for Rust code and hide the messiness behind that.

  • io_pgtable.h uses unsigned long for virtual addresses, phys_addr_t for physical addresses, and size_t for page sizes and page counts.
  • drm_mm.h uses u64 for all VAs, sizes, and everything
  • dma-map-ops.h uses dma_addr_t for device addresses, phys_addr_t for physical ones, and size_t for sizes
  • rtkit.h uses dma_addr_t for device VAs
  • io.h uses resource_size_t for both physical addresses and sizes, which is a typedef for phys_addr_t, except a few functions which use unsigned long?

I think the root of this is mostly the phys_addr_t and dma_addr_t types (which can be separately 32-bit/64-bit depending on the architecture). Perhaps we should just wrap those publicly in rust/kernel, and maybe add some impls to make operating on them more ergonomic? I also think we should have separate variants for addresses and sizes, to avoid type confusion (you can add an addr and a size, but not two addrs, and sizes are the same width as addresses of a given context). I think those two types are probably 64 bits on all 64-bit-pointer platforms, so it probably also makes sense to allow adding usize to them directly, even if the width does not match. And then there's the question of how to handle wrapping... panicking probably doesn't make sense, but forcing callers to Result-check every single op is really weird with operator overloading... maybe just turn it into a warning and keep going with the wrapped result?
