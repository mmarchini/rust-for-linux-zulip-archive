Topic: Should we consider a Weak<T> type?

Daniel Almeida (Oct 21 2025 at 14:28):

@Boris Brezillon is trying to change how the assignment of address spaces to VMs work, and in doing so, he's found an issue with the way things are currently designed. The plan is to have a new type, AddressSpaceManager in charge of AS allocation.

We currently have a more rudimentary slot allocator (in slot_allocator.rs) that merely tracks which AS are free, but that doesn't reclaim slots when the VM is destroyed, nor does it keep a distinction between idle slots (which should remain in place as long as we have unassigned slots) and unassigned slots themselves. So this type is an overall improvement over slot_allocator.rs.

Ideally, we would be able to do this:

impl Drop for Vm {
   fn drop(&mut self) {
      mmu.as_manager.lock().remove_vm(self); //vm_dropped
   }
}

So the Vm must somehow store a reference to the AddressSpaceManager.

We would also need to access the VM on the fault handler, i.e.: from the Mmu threaded IRQ handler. At the moment, this means storing an Arc<Mutex<Vm>> in AddressSpaceSlotData, i.e.:

struct AddressSpaceSlotData<'a> {
    /// VM currently bound to this AS
    vm: &'a Vm, // This would be an Arc<Mutex<Vm>> instead.

    /// Last time this slot was used
    last_used_seqno: u64,
}

enum AddressSpaceSlot<'a> {
    Free,
    Active(AddressSpaceSlotData<'a>),
    Idle(AddressSpaceSlotData<'a>),
}

pub(crate) struct AddressSpaceManager<'a> {
    /// IO mem region
    iomem: Arc<Devres<IoMem>>,

    /// Number of AS slots
    slot_count: usize,

    /// AS slots
    slots: [AddressSpaceSlot<'a>; 32],

    /// Sequence number incremented each time vm_active() is called
    use_seqno: u64,
}

This creates a circular reference, because now the AddressSpaceManager holds an Arc<Mutex<Vm>>. I see two ways around this

a) rewrite it so the circular reference is gone
b) add a Weak<T> type to the kernel

I am convinced that a) is possible, but I haven't been able to come up with something so far, perhaps anyone has a better idea?

Here's a branch with an initial implementation by Boris. He would have to switch from &Vm to Arc<Mutex<Vm>>, but otherwise the basic idea is there.

Danilo Krummrich (Oct 21 2025 at 14:51):

I wonder what impl Drop for TyrDriver does, I see it's hooked up to drm_device::release on the C side

Cc Danilo Krummrich

TyrDriver is the driver's device private data type you return from probe(). It is dropped when the driver is unbound.

When the DRM device is dropped, it drops TyrDriver::Data which lives within the drm::Device.

The fact that you implement both traits for the same type should not have any practical relevance, other than that it can indeed be a bit confusing at a first glance. :)

I hope that helps!
