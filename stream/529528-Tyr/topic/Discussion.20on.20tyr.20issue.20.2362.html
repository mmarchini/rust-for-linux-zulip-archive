The reason glmark2 fails with "tyr: group_submit: invalid address space" when multiple groups share the same VM is:

  • When one of those groups is unbound, unbind_vm is called and vm.address_space is set to None.
  • When another group that shares the same VM later calls group_submit, sched.bind sees that the group is already bound and skips calling bind_group (and thus bind_vm). Later, when sched.submit runs, it reports "invalid address space" because the VM no longer has an address space assigned.

Alvin Sun (Feb 02 2026 at 03:24):

I added logic in sched.bind: when the group is already bound but the VM’s address space is None, I call bind_vm again. With this change, glmark2 runs for a while (I can see the spinning horse), then exits abnormally. If I run glmark2 again, ioctl(group_submit) returns EBUSY.

From debugging, it looks like the address space (AS) is being exhausted.

Boris Brezillon (Feb 04 2026 at 08:56):

One more thing: which branch should I work on—tyr-mini-demo or tyr-dev (the latter seems to be the latest)?

I would say, tyr-for-upstream even. Long story short, we've moved our focus towards upstreaming since last December. This means tyr-mini-demo stays what the branch says it is: a demo to show that having a rust kernel driver for Mali GPUs is possible, but it also means it's a dead end in the long run. Our upstreaming effort is not just about picking bits and pieces from this branch, it also involves

  • using generic abstractions provided by others (JobQueue to replace drm_sched, new version of GpuVm, ...)
  • redesigning the components when it makes sense to overcome the limitations/issues seen while working on the demo
  • revisiting the ownership model
  • revisiting the locking (fine grained locking as opposed to big locks protecting entire components)

We had a quick chat with @Daniel Almeida and @Deborah Brouwer yesterday, and we think it'd be unfair to have you and @Andrew Ballance work on things that will either be dropped entirely or massively revisited during the upstreaming process. Of course, if you're okay with that, we don't object to you working on this downstream branch. But if you're interested in participating in the upstreaming effort, we can probably find tasks for you to pick. This comes with the downside that you want be able to test things immediately, as you would with the demo branch, because there's quite a few missing pieces still. It's up to you, really. I just thought I'd mention the ongoing process so things are clear from the start and there's no disappointment down the road.
