Topic: [RFC] Moving register definitions into the module using them

Alexandre Courbot (May 02 2026 at 06:13):

Hi everyone,

I have been pondering this idea for a few days and would like to quickly check whether there is any reason not to do it.

Currently all our registers are defined into a single regs.rs file. There are textual separations to isolate groups, but otherwise they are all laid out flatly, which is a bit messy. Another unclean area is all the types we import from basically every module of the driver into regs.rs because register types need them. And the worst point: all registers are pub(crate), meaning any module can access any register, including those it has no business dealing with.

So how about splitting that file, and putting each register group under the module that directly uses them, with pub(super) visibility? For instance, all NV_PFALCON_* registers would be defined under falcon/regs.rs. No other module can touch falcon registers - if some functionality export is somehow needed, it would have to go through an interface of the falcon module.

I think this design would set clear ownership for every register and result in better interfaces overall (and less risk of poking at the wrong bits). Any argument against it?

Alexandre Courbot (May 16 2026 at 13:14):

Yes, having a regs.rs within the corresponding submodule is the base idea. Ideally you shouldn't need to introduce new modules like mc, you can use the module that makes use of the register if there is only one (which should apply to most registers).

For registers used in several modules, the idea would be to add the register to the module it naturally belongs to, and then export the functionality needed by other modules through new methods of the module that owns the register (so each regs.rs can remain private to its module).

That second step requires a bit more design, so for now I'd suggest picking some registers that are used in only one module, and move them there as a first exercise - do the low-hanging fruits to get familiar before addressing the more tricky ones.
