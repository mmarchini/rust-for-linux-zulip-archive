And decided if I can try to fix it. Apparently, we just need to tell the kernel this device can handle DMA segments larger than the default 64 KiB per-segment limit. This is how other drivers also fix that:

drivers/gpu/drm/i915/i915_driver.c:     dma_set_max_seg_size(i915->drm.dev, UINT_MAX);
drivers/gpu/drm/amd/amdgpu/amdgpu_ttm.c:        dma_set_max_seg_size(adev->dev, UINT_MAX);

Subject: [PATCH] gpu/nova-core: declare unlimited DMA max segment size

The GSP firmware image (~60 MiB) is copied into a vmalloc'd buffer and
mapped for DMA as one scatter-gather table via SGTable::new(). The Rust
scatterlist abstraction caps each segment's length at dma_max_mapping_size(),
which is effectively unbounded for a plain direct/IOMMU device. Physically
contiguous pages therefore coalesce into segments whose length is bounded
only by memory layout -- up to the entire image when the backing pages
happen to be contiguous -- and readily exceed the PCI-core default
max_seg_size of 64 KiB.

nova-core sets a DMA mask in Gpu::new() but never raises the device's
declared maximum segment size, so with CONFIG_DMA_API_DEBUG=y booting
against a real GSP (RTX 5080 / GB203 via VFIO passthrough) splats. Here the
firmware pages were fully contiguous, so the whole 60 MiB coalesced into a
single segment:

DMA-API: nova-core 0000:00:03.0: mapping sg segment longer than
device claims to support [len=62914560] [max=65536]
WARNING: kernel/dma/debug.c:1232 at debug_dma_map_sg+0x1de/0x4d0

The mapping itself is correct -- Nova re-decomposes every segment into 4 KiB
GSP page-table entries in map_into_lvl(), so it imposes no upper bound on
segment length -- but the device advertises a capability it does not have,
and whether the warning fires at all depends on memory fragmentation.
Declare the true capability by calling dma_set_max_seg_size(u32::MAX)
alongside the existing DMA-mask setup, matching the established DRM-driver
idiom.

With this change the warning is gone and GSP boot / RPC still succeed with
CONFIG_DMA_API_DEBUG=y.
