Topic: Storing driver-bound references into sub-devices

Alexandre Courbot (Jun 01 2026 at 11:55):

With the HRT series being merged, we now have the option of getting rid of a pattern we used in Nova since the beginning, namely the constant passing of a &Bar0 to every method that needs to perform I/O and, to a lesser extend, the passing of a bound Device to do things like allocating memory.

Instead, we now can store references to these elements into each type that needs them, and access them from self, pretty much like what Nouveau does.

The most obvious candidate is Falcon. If we do this, method invocations like

falcon.dma_load(dev, bar, fw)

can be simplified into

falcon.dma_load(fw)

This makes a difference in the code: converting falcon to use this removes more than 50 LoCs in the codebase. On the other hand, Falcon gains a lifetime parameter, meaning that Falcon<Gsp> is now referred to as Falcon<'_, Gsp>.

To summarize:

Pros:

  • Less distracting arguments to methods,
  • Overall reduction of code,
  • That's probably how we would have done it since the beginning, had we had the
    option.

Cons:

  • Every structure gains a lifetime, both in their declaration and when referred
    to.

  • Structures get slightly bigger (although we are talking 16 bytes on average
    for a very limited number of instances, so that's really negligible?).

  • bar becomes self.bar in methods' code.

I think we should only apply these to top types like Falcon and Gsp - HALs which are only called locally can work efficiently with arguments. But IMHO the global readability improvement is a win.

Opinions? This sounds like a "good first issue", but since it is rather consequential for the direction of the project I'd like to see it done quickly if possible (but after merging Blackwell :slight_smile: ) if we decide this is the right way to proceed.

I can post a diff for Falcon somewhere if folks would like to see what it looks like in practice.

Danilo Krummrich (Jun 01 2026 at 12:03):

It is expected that lifetimes propagate through and I don't think it is a bad thing. I considered whether I want to send a patch series that does those changes, but also figured it would be a good first task for a new contributor.

In terms of where I would draw the line:

  • If a type only needs it in the constructor or a single method that is always called from a context where the device resource is already available, don't store the device resource in the type.
  • If a Device<Bound> is not needed, ARef<Device> can still be used.
