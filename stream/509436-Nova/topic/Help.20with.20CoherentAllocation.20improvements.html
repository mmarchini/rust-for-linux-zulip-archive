Hey! I'm giving a try to CoherentAllocation improvements [COHA] (or what's left of it). The as_slice and write methods already exist in CoherentAllocation, but I was trying to solve some mentions of it left in the code, e.g. here, but I've run into some problems.

I've tried to replace the implementation of the function by T::from_bytes(unsafe { fw.as_slice(offset, size_of::<T>())? }).ok_or(EINVAL), but I get problems with the lifetimes, because the compiler correctly infers that the lifetime of the resulting reference is the same as the lifetime of the DmaObject, which isn't the case now.
I see several options:

  • Modify the as_slice method to return a ref with a different lifetime. It works, but that's outside Nova and even if the function is unsafe I feel like that makes it a bit more dangerous, even if that's okay for the use case here.
  • Not use as_slice or do some tricks, so the compiler allows me to return the ref with a different lifetime. I've tried it and it works, but is uglier.
  • Modify the code that uses transmute, so it drops the immutable refs before getting mutable ones. Doable, but also ugly, specially since we know the code is sound and the refs refer to different parts of the memory.

I also have another problem, because I can't use from_bytes_mut if the type doesn't implement AsBytes. I checked the types we are using and it should be fine to implement it for all of them except for FrtsCmd which has padding (AsBytes doesn't allow padding). One solution would be to read its parts read_vbios and frts_region separately.

Sorry for the message being this long :sweat_smile:. I would greatly appreciate any tips on which path to take! I plan on submitting a patch for this soon.

Alexandre Courbot (Oct 14 2025 at 22:10):

I don't think CoherentAllocation's as_slice and write can ever be safe, for the reason you mentioned. We do know that the device is not accessing that memory though, because we are in the constructor of the object and can guarantee that we are the sole user of it at that time.
