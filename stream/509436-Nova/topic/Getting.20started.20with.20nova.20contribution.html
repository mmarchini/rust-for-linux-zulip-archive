Alexandre Courbot (May 02 2026 at 06:02):

Hi,

Usually people start contributing because they have a particular itch they want/need to scratch, so if you are wondering where to start it might be a good idea to broaden your area of exploration. ;)

Nova is not particularly great for beginners IMHO. It requires a recent NVIDIA GPU to test the code, there are currently lots of moving pieces forcing you to rebase your code all the time, and GPU internals are complex. For a first contribution, I think the broader Rust-for-Linux project is more likely to provide you with good first issues. It will always be time to come back to Nova once you have landed a few patches and become more familiar with the project.

Also one thing that is often overlooked by new contributors is that reviewing patches is a great way to contribute and get familiar with both the code and the people. So if you are already fluent with Rust, it's a great starting point: read patches, test them, give feedback.

Alexandre Courbot (May 16 2026 at 13:17):

Technically yes, but there are other ways to receive the emails than subscribing using your email address to the mailing list.

lei is a fantastic tool that lets you download emails from the public inbox and store them in a maildir directory on your local system. It's as if you had received the email, minus the subscription.

You can use it to receive the new emails from a mailing-list since the last time you ran it, but it is also capable of fetching any email thread in case you need to review/apply an older patch.

This article is a good way to get started: https://people.kernel.org/monsieuricon/lore-lei-part-1-getting-started

Alexandre Courbot (May 17 2026 at 13:55):

As far as code reviews are concerned, the only visible code is the one you send to the mailing-list. You will receive feedback on it, which typically results in a v2, v3, etc. being sent until the code is merged. If you are familiar with Gerrit, the workflow is similar, but done by email.

But what is merged in the end is the patches you sent by email (using git am or something close), so there is no need to share your branch online - you can push it to a public or private git repo if you want, but that's mostly for your own convenience (and backup) and isn't related to code review. There are pull requests in Linux, but they are typically used by maintainers to send their work upwards, until it lands into Linus' tree.
