Here:

pub(crate) fn unload(&self, bar: &Bar0) -> Result {
    let mut guard = self.cmdq.lock();
    let cmdq_pin = guard.as_mut();
    let mut cmdq = unsafe { cmdq_pin.get_unchecked_mut() };
    commands::unload_guest_driver(&mut cmdq, bar)
}

..."unsafe" is required because Cmdq is pinned, and by design you must not be allowed to get a plain &mut to it, because that would allow moving a pinned item.

So it's unsafe because you're promising not to move it. The compiler cannot verify this promise.

...but I have a strong feeling that I'm saying things you already know. But you did say "for reasons I'm not 100% sure of", and that's the part I was trying to respond to first.

@John Hubbard @Alexandre Courbot at least from nouveau the unload sequence on hopper+ is just send the command and wait in a loop for the riscv to halt

John Hubbard (Feb 06 2026 at 03:03):

That matches what I'm seeing in Open RM, which is:

  1. Send UNLOCK_WPR command to GSP-RM (non-blocking)
  2. Wait for GSP RISC-V core to become inactive
  3. Call gspStateUnload_HAL() to cleanup GSP state
  4. kfspStateUnload releases memory descriptors (FRTS, GSP-FMC, boot args)
  5. FSP itself: NO explicit shutdown - FSP continues running
