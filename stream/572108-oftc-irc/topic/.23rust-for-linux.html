Gary Guo (Mar 19 2026 at 12:41):

for the preallocated percpu slots, make the worker thread busy wait (those are short lived anyway)

I'm not sure if this is will be true in general. If we implement sleepable revocable using hazptr, then the reader side might go into sleep which won't be short-lived?

My prior experience with hazard ptr is via the Rust arc-swap crate, and the use cases there is definitely not always short-lived. You might be holding on to a slot for very long time.

However, arc-swap is very difficult way of using hazard ptr compared to the current hazard ptr series. It combines reference counting with hazard ptr, so if synchronize side finds out the pointer is being used, it just increases the reference count on behalf of the reader side, and when the reader side release the slot it will figure this out and it would need to drop the reference count.
